# Starchart map mechanics — the normative deep layer

Moved VERBATIM from SKILL.md (v0.6.3) in the 0.7.0 distillation. SKILL.md keeps the executable core; where its sketch and this file differ in detail, THIS file governs. Campaign's `references/state-writes.md` remains the normative home of the state-write guard this file mirrors.

**Read this file ENTIRELY on any of its triggers:** a map merge conflict or pull/push failure, two Resolutions of one ticket, duplicate ticket numbers or slugs, a missing/orphaned ticket file or index line, a torn Resolution, a `.gitignore` swallow, an ephemeral (repo-less) map question, the map→campaign conversion.

## 1. Artifacts: commit/push discipline, arbitration, NNN/slug repairs, write guard

`<repo>/.catalyst/map/<effort-kebab>/MAP.md` — one directory per effort, so maps never share a path or ticket numbering; tickets — `<repo>/.catalyst/map/<effort-kebab>/tickets/NNN-<slug>.md`. Starting starchart while a DIFFERENT effort's map dir exists → leave it untouched, create your own effort's directory. A map converted into a campaign stays in place (committed history; the campaign's Decision log points at it). `<repo>` resolves as `git rev-parse --show-toplevel` — every `.catalyst/` path anchors to the repo root, never a repo subdirectory's cwd (a session opened in a subdirectory must not fork a second map); no git repo at all → the map cannot be committed: surface it to the user (init a repo, or explicitly accept an ephemeral map — the acceptance covers MAP state only: the Done exit cannot convert an ephemeral map into a campaign without re-raising campaign's own repo question there, and any route into arcane execution re-raises arcane's repo gate, which has no repo-less mode), never proceed silently. The map directory is **committed to the repo** (multi-session memory — it must survive `git clean` and clones, unlike the ephemeral self-ignored `.catalyst/` workspaces); if the project's `.gitignore` swallows it (a wholesale `.catalyst/` rule, or `.catalyst/*` without this dir's negation), fix the rule at the first map commit (named to the user, never silent): ensure `.catalyst/*` plus BOTH `!.catalyst/map/` and `!.catalyst/campaign/` (unconditionally — a map often converts into a campaign, and the half-remedy would swallow it) — a bare negation under an ignored directory doesn't work, and `git add -f` is banned (the next clone re-hits the same swallow); a rejected push at session end → fetch + merge under the arbitration below, retry once; still failing or offline → tell the user the map is local-only (the unpushed-map red flag below is about SILENT unpushed state, not a named failure); escalations one map load discovers (conflicting Resolutions, Out-of-scope conflicts, index lines without ticket files) are presented as ONE batched question, never a drip; the session that writes a Resolution — or any map edit — commits it immediately (to the effort's base branch, never a scratch branch), session end is a safety net, not the commit point; every map write carries campaign's state-write guard — the normative text lives in campaign's `references/state-writes.md`, mirrored here compressed: re-read the referent from disk immediately before writing (a referent changed since your derivation was built = a stale ask — adopt the landed record, re-derive, never overwrite), edit MAP.md and tickets surgically in place (never regenerate from the session's in-memory model — a regeneration silently reverts a sibling's landed write), immediately before staging re-check the file's HEAD blob against what the re-read saw (campaign's absent-vs-alias discipline applies verbatim), and after committing verify the new commit's PARENT blob still matches the pre-staging hash — a mismatch means a sibling landed inside the window: re-derive from the parent's actual content and RESTORE the vanished record in an immediate follow-up commit; the guard exists because a LIVE SIBLING session on the same checkout produces the merge problem without a merge — no pull, no conflict, no arbitration cell fires; and map commits verify the checkout is ON the map's `base:` branch first (`git branch --show-current` against MAP.md's `base:` line — a detached HEAD or a wrong branch is surfaced, never committed onto: a detached-HEAD commit lands on no branch and dies silently in the reflog); when a remote exists — pull before reading the map (pull failure shapes routed, never improvised: divergence → fetch and merge under this arbitration; remote unreachable → proceed on local state and tell the user the map may be stale), push at session end, and a merge conflict is arbitrated by what's on disk in the tickets: an on-disk Resolution wins over an index line; TWO DIFFERENT Resolutions of the same ticket → the user decides (a user decision is never auto-picked or silently discarded) — the tickets' dates tell the user which is newer; a Destination conflict → the user (the destination is a crucible-hardened user decision — never auto-merged); conflicts in Not-yet-specified/Notes merge as a union; Out-of-scope conflicts → the user; TWO tickets minting the same NNN for DIFFERENT questions (concurrent machines — the file merge is conflict-free, so nothing else fires): the later-created ticket renumbers to the next free NNN — a repair with a FIXED shape: every map reference to it (`## Blocked by` edges, index lines — each reference is a link carrying the slug, and matches by SLUG, never by bare number) is rewritten FIRST, the ticket file renames LAST, all in ONE commit — so the duplicate-NNN signature survives any crash mid-repair and re-fires it; the repair closes with a completeness grep for `<old NNN>-<renamed slug>` across the map dir (the old number legitimately stays with the SURVIVING ticket — the grep hunts references still binding it to the RENAMED slug; any hit = the repair is unfinished, not done). Creation dates tied (same day) → the later ticket is the one whose slug sorts later in BYTE order, C locale — locale collation differs per machine (then Question text, same byte order — a numbering choice is contentless, deterministic, never a user question); dates absent → the user — same evidence-first discipline as campaign state. Aftershocks of the repair resolve by slug, not number: a reference minted against the OLD number on a machine that missed the repair re-binds to its slug's ticket at merge (never silently to the number's new owner); two machines repairing independently can land the same ticket under two numbers — the next merge detects the duplicate by slug/Question content (no duplicate-NNN signature exists for it) and COLLAPSES it to the lower number with the same reference-first discipline: references to the losing number rewritten first (slug-matched), the two files' content merged as a UNION (blocked-by edges and notes union; two different Resolutions → the user, per the arbitration above), the losing file deleted LAST, one commit, closed by the same grep for `<losing NNN>-<slug>`. A merge surfacing BOTH signatures (a duplicate NNN and a duplicate-slug ticket) repairs in a FIXED order — collapses first, then renumbers — so every machine emits the same final map from any merge order. If the project has an issue tracker and the user prefers it — the same structure lives there (map issue + child issues), same format.

## 2. Ticket anatomy, the Frontier index, orphan repair

Ticket: a `created: <ISO date>` line + `## Question` (one decision, sized to one session) + `## Blocked by` (links) + after closing `## Resolution` (decision + why + context pointer, DATED — ISO, like every date here — the record's date is what arbitration and the map→campaign conversion read; the pointer targets a COMMITTED path or the extract is inlined, never a self-ignored workspace like `.catalyst/research/` — a committed decision must not rest on evidence that dangles on every other machine). **Frontier** = open, UNBLOCKED tickets — the unblocked subset of the `## Frontier` index (blocked lines sit there too, carrying their refs), reconciled on load like Decisions so far (an on-disk Resolution or edge beats a stale index line; a map predating the index builds it once from the ticket files — a repair, committed). A ticket is CREATED in ONE commit carrying both the file and its index line (the NNN repair's one-commit lesson); the reverse orphan — a ticket file no index names — can still arrive via a crash or merge, so the build-once repair AND the Done gate close with a completeness check: `ls tickets/` against the union of `## Frontier` + Decisions so far — a file in neither index is an orphan routed by its CONTENT: an on-disk `## Resolution` → its Decisions-so-far line (the ticket is closed — never a Frontier line for a decided question); no Resolution → its Frontier line (a repair, committed); "no open tickets remain" is declared off that check, never off the index alone.

## 3. Working the map — full protocol

**Working the map:** load MAP.md (the map is an index: read only the ticket you take) → take ONE ticket from the frontier (the `## Frontier` index names the candidates — no open-ticket sweep) (an on-disk `## Resolution` means the ticket is CLOSED even if the index lacks its line — reconcile Decisions so far, never re-decide; the mirror case — an index line whose ticket FILE is missing — is first a LOST REFERENT: the map is committed, so recover the file from git history (`git log --diff-filter=D -- <path>` / `git show`; unshallow a shallow clone first) — contentless bookkeeping, a repair, committed; genuinely never committed or unrecoverable → a lost decision: surface it to the user, never re-decide silently and never treat the one-liner as the full decision; a Resolution missing its required fields is a torn write, not a decision: the why / context pointer may be completed from the ticket's own evidence, but a missing DECISION forces re-open with the user — self-completing it is answering a Grilling ticket for the user; reconciliation is bookkeeping and does not consume this session's ONE ticket) → resolve it by its type → write the Resolution and, immediately after, its line in Decisions so far → new questions: into tickets or into the fog; matured fog — into tickets → **stop**. **Never more than one ticket per session** (exception — research tickets: they may be burned down in a batch by parallel scouts). A ticket's resolution spawns the next session, not a continuation of this one: fresh context per decision is the whole point of the map.

## 4. Done — full exit routes and the campaign conversion contract

**Done:** Not yet specified is empty, the frontier is empty, AND no open tickets remain. An empty frontier with open tickets is a blocked-by cycle, not a charted map — break it with the user (merge the tickets into one decision, or close one on an explicitly recorded assumption). When truly charted: Exit routes (the user picks): the remaining work fits one spec → catalyst:crucible for final hardening → spec → catalyst:arcane-mode; the work clusters into several phases toward a milestone → convert the map into a catalyst:campaign under campaign's conversion contract — the COMPLETE mapping, not a subset: Destination → Intent, Decisions so far → Decision log (D-lines keep the Resolutions' ORIGINAL dates), work clusters → phases, `## Out of scope` → Non-goals (the scope fence survives the conversion — nothing graduates back silently), `## Notes` standing agreements → Decision log with their dates (an undated legacy line converts dated by the conversion, annotated as such — campaign's rule), and the campaign's creation D-line records the map's path (matched by content — after date-sorted renumbers it need not sit first). If the effort turned out small — straight to the spec.
