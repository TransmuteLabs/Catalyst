# Campaign — the program layer

## Overview

Arcane-mode executes one plan; a campaign is the umbrella above many of them: a roadmap of phases toward a milestone, alive across weeks and dozens of sessions. The orchestrator surfacing from one finished branch must never navigate from memory — the campaign state answers "where are we and what's next". Core: GSD's roadmap/milestone lifecycle, reduced to two files over the existing family (no tracker, no config layer).

**Campaign decides sequence; each phase is still delivered by the family flow** (crucible → arcane-mode, whose premortem gate precedes Task 1 → verification). The campaign never re-implements those skills' jobs.

## State (two files)

`<repo>/.catalyst/campaign/<name>/`:


Rules: PROGRAM.md and ROADMAP.md are **committed to the repo** — they are the program's memory for weeks and must survive `git clean`, clones, and machine changes. No git repo at all → the state cannot be committed: surface it to the user (init a repo, or an explicitly accepted ephemeral run), never proceed silently — starchart's arm. If the project's `.gitignore` swallows the committed dir — a wholesale `.catalyst/` rule, OR a `.catalyst/*` rule missing this dir's negation (the shape an old half-remedy or a hand edit can leave) — the first state commit fixes the rule (named to the user, never silent): ensure `.catalyst/*` plus `!.catalyst/campaign/` and `!.catalyst/map/` — a bare negation UNDER an ignored directory doesn't work (git never descends into it), and `git add -f` is banned: it leaves the rule in place so every future clone re-hits the same swallow. State commits go to the campaign's BASE branch (the branch phases fork from), never onto a phase work branch — a discarded phase branch must not carry the program's memory away with it. Mechanics: when a remote exists, the router pulls the base checkout FIRST (step 0 below), and session end pushes the state commits AND the phase branch (an unpushed phase branch is unrecoverable from any other machine). Before that pull, a flip edit deferred by the single-checkout workflow (below) is committed FIRST — commit-then-pull, never stash. Pull failure shapes are routed, never improvised: the pull hitting divergence → fetch and merge, conflicts resolved by the arbitration below; remote unreachable → proceed on local state and tell the user the roadmap may be stale; a remote ATTACHED mid-campaign (no upstream at contact) → set the upstream, push the base state, and PUBLISH every live local referent from the machine that has it — kept and executing phase branches, the never-defer history — named to the user; the publish duty keys on ANY live local referent absent from the remote, checked at step 0 whenever a remote exists (a partial publish resumes — never first-contact-only); UNRELATED histories at contact (two machines attached the same fresh remote independently) → fetch and merge with the explicit unrelated-histories allowance — the one sanctioned use — file-level add/adds of STATE files (the campaign dir, flip-carried specs/plans/premortem yamls) arbitrated by these same cells (both trees are legitimate state), while SOURCE-file add/adds are ordinary git conflict work surfaced to the user — these rules never auto-arbitrate code, the phase-divergence principle — never a force-push (until published, other machines legitimately read local-only state through the recovery arms). Push failures are routed the same way: a rejected push of STATE commits (the remote advanced during the session) → fetch + merge under the same arbitration, then retry once; still failing, offline, or auth-refused → tell the user the state is local-only (the red flag below is about SILENT unpushed state, not about a named failure). A rejected push of the PHASE branch because it DIVERGED (two machines worked the same phase — nothing detects this earlier) is a CODE merge, not state arbitration: stop, surface to the user whose work continues where, resolve by ordinary git conflict work on one machine — these rules never auto-arbitrate code. State commits verify the base checkout is actually ON the base branch first — `git branch --show-current` compared against PROGRAM.md's `base:` line (the schema; a legacy file without it backfills the line first), never a guessed default: a detached HEAD or a wrong branch is surfaced, never committed onto — a detached-HEAD commit lands on no branch and dies silently in the reflog. Merge-conflict arbitration for ROADMAP.md/PROGRAM.md: per phase row, a status WITH its evidence stamp beats one without; both stamped → the further status wins (evidence is cumulative) — but EVERY row-arbitration cell consults the Decision log FIRST: an answering D-line covering this row's divergence (or membership) is the standing record — the chosen side, as possibly further advanced, wins, and a removed row stays removed, REGARDLESS of whether an unresolved shape is party to the merge (further-wins and stamped-beats-stampless run only when no answering D-line covers the divergence — a recorded user answer is never mechanically overridden by later unilateral advance of the non-chosen side, which is surfaced instead) — and EXCEPT a stampless side that is a DELIBERATE demotion by the row-history test (the reconciliation guard's test: the row previously carried the further status and a later commit rewrote it backward): that is a user decision → the user, never stamped-beats-stampless (EXCEPT a backward rewrite whose commit carries the amendment's D-log event AND the stamped `reopened` shape — the sanctioned mechanical demotion, routed by the reopened arm, never the hand-demotion shape; the D-line WITHOUT the stamped shape is a malformed write — surface it to the user, route neither arm). The rounds-keyed EXCEPTs below compare LINEAGE TOKENS (schema): a waived lineage compares its waiver dates the same way, and a `done` vs a `verified` from DIFFERENT lineages (one carries rounds, the other waived) → the user — the two sides' verification histories diverged. EXCEPT `done` vs a `verified` whose token is LATER than the token in the `done` stamp: that acceptance was overtaken by a re-opened verification (acceptance must reference the final head — arcane's verification rule), and "cumulative" is false → the user (a `verified` whose token is EARLIER is the stale side — the `done` wins mechanically, genuinely cumulative; unorderable tokens → the user); and EXCEPT `done` vs a `reopened` row whose "was" token EQUALS the `done` stamp's token — equality here is the voiding signature, not cumulative evidence (that `done` IS the acceptance the demotion voided; this is what the "was" token exists for): the `reopened` side wins either way, and the presentation follows the schema's pedigree rule — the pair's dates consistent: assert the voiding flat; inconsistent or missing: PRESUMED voiding, the coincidental-restart possibility named — and the `done` side's later `branch:` append (if any) is surfaced to the user as an integration executed on a (possibly presumed) voided acceptance; a `done` vs a `reopened` whose "was" token DIFFERS from the `done`'s token is NOT that cell — compare DIRECTION (tokens are monotonic within a phase, the schema invariant): the `done` token LATER than the "was" token = a post-re-verification acceptance, cumulative — the `done` side wins mechanically; the `done` token EARLIER = an acceptance even older than the one this demotion voided, superseded generations ago — the `reopened` side wins; tokens of different kinds or otherwise unorderable → the user, the histories diverged (the first EXCEPT reads a `verified` row's live token, never a `reopened` row's "was" token); two `done` rows where one side merely EXTENDS the other's stamp with its `branch:` part → the superset wins mechanically (the append is cumulative evidence, no escalation); a row parked on one side and advanced in ## Phases on the other → the user ("further wins" never auto-un-parks: parking is a user decision, and a user decision is never silently discarded); a row parked on BOTH sides with differing preserved status/stamp values arbitrates the preserved pair by these same per-row rules — the winner stays PARKED (arbitration never un-parks); differing park reasons/dates → the user; conflicting `done` rows compare DIRECTION like the reopened cells: orderable tokens → the later-token side wins mechanically (a post-re-verification acceptance supersedes a stale one) — and the LOSING side's `branch:` append (if any) is surfaced to the user as an integration executed on a superseded acceptance, never dropped silently; unorderable → the user; neither stamped, or two different values for the same stamp → the user; scope differences (added/removed phases) reconcile through the Decision log with the user — never blind ours/theirs. PROGRAM.md's own sections arbitrate too, each by its nature: the `base:` line — two differing values (two machines' backfill repairs, or a hand edit) → the user, never mechanical (a backfilled value is only a repair, and every state transport depends on this line — the conflict leads the batch), durabilized before the merged file commits as `base: unresolved (between: <X> ||| <Y>; merged <date>)`: the branch where the merge ran is the shape's INTERIM home — while the shape stands, routing and evidence flips are suspended and no state commits land on any GUESSED base, with exactly two sanctioned write sites, both on the interim home: the durabilizing merge commit itself and recorded-ANSWER commits (the never-defer classes and batch recordings — a given user answer must not die uncommitted; pushed as usual); before ASKING, FETCH — state-safe, always sanctioned — and inspect the candidates' PROGRAM.md on the remote for an answering rewrite or D-line: a delivered answer is ADOPTED, never re-asked (the durabilizing machine must not re-ask a question whose answer another machine already recorded); the user's answer rewrites the line with its D-log event in ONE commit on the interim home, and when the chosen base is a DIFFERENT branch the SAME session then CARRIES the campaign state onto it (`git checkout <interim> -- .catalyst/campaign/<name>`, or a merge, committed on the chosen base — the named transport step) — the interim copy stays as history and later merges arbitrate by the answering D-line; the Decision log is an append-only EVENT LOG — a merge UNIONS both sides' lines by CONTENT, never drops either (both machines' events happened); D-ids are per-file ordinals, never cross-machine references — consumers match D-lines by content (kind + phase ids + date; every D-line carries its date), so after the union the log renumbers sequentially by date — same-date ties break by kind (the sanctioned event names as written in this file: creation, scope, parking/un-parking, parallel sanction, voided acceptance, amendment, declined backfill, milestone closing, re-open, divergence answer, milestone-status answer, scope answer; plain decisions sort as "decision"), then referent ids, then text, lexicographically (a stated total order: the same final log from any merge order) — and the renumber CLOSES with a completeness check against the POST-COLLAPSE union (the same-event collapse is part of the union step: collapses first, then the renumber, then the check): the same line count and the same multiset of FULL LINE TEXTS minus the ids (a lost or mutated line — including a mutated why-text — is a broken repair: redo it; at volume the renumber rewrites ids only, never retypes lines); all dates in stamps, records, and D-lines are ISO — YYYY-MM-DD, plus HH:MM where a same-day arm applies (normative here because dates are sorting and arbitration keys; in-file records use `HH:MM` — artifact NAMES use the hyphenated `_HH-MM`, a deliberately different shape); two lines recording the SAME event (same kind and referents, different wording — and, for answer-kind events, the SAME CHOSEN SIDE: two answering D-lines for the same divergence choosing DIFFERENT sides never collapse — they are two conflicting recorded user decisions: a strictly later-dated one supersedes (records arbitrating records, the milestone discipline); with no strict order the pair RE-DURABILIZES between the two chosen sides (SCOPE-answer conflicts — keep vs remove — re-durabilize as the row's `unresolved-scope` shape, their native form, never the between shape), noted as conflicting answers → the user) collapse to one, keeping the fuller text (the LONGER; equal lengths tie-break lexicographically — a total order, the same final text on every machine); the Milestones `status:` line arbitrates by the RECORDS' DATES, never by shape precedence — every transition writes a dated record (a closure: the audited/waived shape plus its D-log event; a RE-OPEN: the un-park rule's D-log line — a record too; a record's arbitration date is the date of its EVENT, never of its writing — a backfill carries the historical date): the later-dated record wins (a re-open dated after a closure re-opens it everywhere — a user's recorded re-open is never mechanically overridden by a stale closure; a later re-closure closes it again), and this covers closed-vs-closed of any shapes the same way (a post-re-open waiver beats a stale audit; a later audit beats an older waiver); a same-day second transition appends the time, like the extend-waiver token — the same-date escalation is for genuinely simultaneous records, not every fast day; recordless sides (a bare `closed` — a hand edit or the migration shape; NOT one standing under a declined-backfill D-line — that milestone is closed-with-record for ROUTING, but the decline is a NON-ANSWER record: it never competes with a genuine closing record by date — any closing record, dated or undatable-attested, supersedes it by KIND (the schema's cell); the decline's date serves only the once-only ask and decline-vs-decline collapse), same-date records, or any doubt → the user; the top `status: active|complete` line is DERIVED — never arbitrated, recomputed from the merged Milestones section (the writer that flips it writes the date of ITS flip). Spec files and premortem yamls travel in the same state commits and get their own merge cells — never ours/theirs: a MARKER-line conflict (plain vs pending) resolves from the merged open/deferred ENTRIES — any `pending re-approval` entry with no answer appended keeps the marker PENDING (a restore only ever follows recorded per-entry answers, crucible's shapes, so a pending-clause-less marker — plain or waived — over an unanswered entry is the bad-merge shape every consumer already refuses); premortem-yaml conflicts recompute the verdict from the merged yaml's own records (`accepted_risks`/`mitigated` and rejection reverts are append-only unions); two sides appending DIFFERENT answers to the SAME pending entry (re-approved on one, rejected+reverted on the other) → the user — two recorded user decisions conflict, never auto-picked; the merged entry keeps BOTH appends (the double-answer shape IS the durable signature — crucible's cross-check treats it as pending on every machine, so a death between the merge commit and the ask loses nothing); until the answer, the spec BODY holds the reverted approved text (the safer state — EXCEPT the entangled-revert shape: its body holds the pre-revert text until its disposition, crucible's pending shape) and the yaml recompute waits; spec-body conflicts OUTSIDE recorded answers have no mechanical cells — ordinary git conflict work, surfaced to the user AND recorded: the resolving merge lands a dated spec open/deferred note naming the resolved sections (`;`-separated names — crucible's list grammar) (the resolution record — crucible's hand-edit test reads it: a merge hunk matching neither parent with no record NAMING its section is the smuggle shape — the record binds by section name, an unrelated note exempts nothing). Rows stamped by pre-0.4.10 versions (a bare `done (accepted: <date>)`, no lineage token, no `branch:`) are a MIGRATION shape, not deaths: ask the user once per campaign to backfill the executed choices — never re-execute them blindly; a missing lineage token arbitrates as unknown (→ the user). Run phases in WORKTREES off the base (the default — the base checkout stays available), and **the session that produces the evidence flips the status in the base checkout and commits there immediately**, not at handoff time (the Session-end check is a safety net, not the flip point). Single-checkout branch workflow: edit the flip immediately, commit it at the next moment base is checked out — branch finish at the latest, never onto the phase branch. The NEVER-DEFER CLASS — commits that exist to be seen cross-machine — never defers, each via a momentary base checkout or a base worktree (the phase branch keeps the working state): (1) the `reopened` demotion — visible DURING the fix wave (branch finish, the usual deferral backstop, is exactly what the red suspended): commit it to base NOW; (2) a spec-gate premortem run after the `specced` flip — its yaml and spec edits (premortem's transport rule); (3) an AMENDMENT wave's spec edits + pending marker (arcane's route — produced by crucible's fast path, same transport: base must show the pending shape, or every other machine consumes the stale approved spec); (4) each recorded ANSWER to a pending re-approval entry (approve, reject, or arbitrate — premortem's one-commit-per-answer); (5) each recorded ANSWER to a standing unresolved shape (row, milestone status, or scope) — the rewrite and its D-log event in ONE commit (separated, the answered row reads as independent advancement and the ask re-fires); (6) the hand-demotion suspect ask's answering D-line (the router's accounting answer — the record that stops the ask re-firing on other machines). Every never-defer commit also PUSHES now when a remote exists — a base commit is invisible to every other machine until pushed, and this class exists precisely to be seen before the session ends (the push-failure routing above applies: a named failure, never silent; the pushed BASE commit is what lets the bridge read the yaml from the base tree on another machine before any branch finish — recovery from a pushed phase branch stays the fallback, premortem's lost-referent arm). Parallel phases (user-sanctioned): the sanction is DECISION-LOGGED (a D-line naming the sanctioned phases and each phase's owner as a user-legible label — a machine, a worktree, a person) — the durable trace every machine reads; without it a two-`executing` roadmap is indistinguishable from the violation. ONE session, named by the user, owns all roadmap flips — the others leave the evidence on disk and name it in their handoffs; the owner (or the next router run's reconciliation) flips from it. ROADMAP is an index — details live in the linked specs/ledgers. **Status changes only with evidence, and the flip writes the evidence stamp onto the row** (stamp formats in the schema above) — and when the stamp's referent is a repo file (the spec and its spec-gate premortem yaml for `specced`, the plan and its premortem yaml for `planned`), the flip's state commit CARRIES that file into the base branch — with the transport NAMED: the file is usually produced on the phase branch/worktree, so bring it into the base checkout explicitly (`git checkout <phase-branch> -- <path>`, or copy from the worktree) and commit it together with the flip; a stamp whose repo-file referent is uncommitted on base is half-done — "evidenced on every machine" holds only because the evidence travels with the commit. A `discarded` branch-finish choice never deletes the branch before the row's referents are committed on base — the branch may hold their only copies: an APPROVED spec (crucible's `status: approved` marker, including its skip-order shape `status: approved (waived: <date>)` and the pending shape `status: approved (edits pending re-approval: <date>)` (keyed on the clause — the waived combo carrying it is pending too) — approved for evidence, but arcane's input rule owes its fast re-approval before the pending sections are consumed; a markerless file is a draft, not evidence) → specced; premortem PASS/WARN — BLOCK bars it — plus the written plan → planned; ledger shows tasks closing → executing; the ledger's `verification: converged` line → verified — OR its sanctioned substitute `verification: waived by user <date> — <scope>` (economics' review-floor drop, recorded; the stamp is `verified (waived: <date>)`, an honest mark every reader sees: the rounds did NOT run, the user chose that, and no session re-demands them); user accepted → done (no other acceptance artifact exists — an unstamped `done` is unevidenced by definition). Roadmap edits that change scope (add/remove/reorder phases) are user decisions, logged in the Decision log — never silent. All escalations ONE router run discovers (arbitration cells, reconciliation gaps, the migration backfill ask) are presented to the user as ONE batched question — arbitration cells first, then reconciliation gaps, then routing — never an interrogation drip, and never silently self-resolving the "small" arms to spare the user. A LARGE batch is still one message, walked as a numbered list the user answers item-by-item (the batch bounds the interruptions, not the answers' depth — crucible's one-question doctrine governs each item); a PARTIAL answer records the answered members' outcomes at the moment they are given (never re-asked — the same at-the-moment rule as the disposition trailer), while unanswered members keep their routes blocked and re-surface in the NEXT run's batch: a partial answer never unblocks the unanswered arms and never re-litigates the answered ones. Unanswered ARBITRATION cells (whose only native signature is the transient merge conflict) are made DURABLE before the merged file commits — the row is written as `unresolved (between: <full row X> ||| <full row Y>; merged <ISO date>)` with the sides in LEXICOGRAPHIC order, ` ||| `-separated (machine-independent — two machines durabilizing the same divergence write the same pair; a legacy single-pipe pair from a 0.4.24-era session is unparseable into sides — NORMALIZE it to the triple separator on first read: the split point is the ` | ` whose two halves BOTH parse as well-formed roadmap rows (well-formed = the schema's five cells `id | phase | milestone | status | links`, the status in the enum or a sanctioned stamped shape, the links cell legitimately EMPTY — the row shape is the oracle) — exactly one qualifying split → rewrite it, a repair, never a new divergence; zero or several qualifying splits → the user with every candidate parse shown, never a guess), a sanctioned shape the self-consistency check treats as a standing batch member (routing blocked: when the routed phase IS the unresolved row the milestone STALLS on the batch member — phase order is never skipped past it; an unresolved row also bars the all-done milestone-audit gate; every other row routes normally); the user's later answer rewrites the row to the chosen side, recorded at the moment given — the conflict is never consumed by commit mechanics into a silent ours/theirs. The unresolved shape is N-ARY — sides sorted lexicographically, separated by ` ||| ` (a roadmap row's own pipes never collide with the triple) — and has its own merge cells: unresolved rows over the same divergence merge by UNION OF SIDES (flattening — an unresolved row is never itself a side: same sides collapse, the earlier `merged` date kept; two VARIANTS of one side — the same side advanced in place on one machine — collapse by the per-row arbitration cells (the same rules as any row pair: an answering D-line first, then further-wins/stamped-beats-stampless and the token-direction cells — the mechanical winner's text becomes the side); a variant pair those cells route to the user is itself a divergence: BOTH variants join the ask as separate sides — never resolved by text length, which would discard a recorded stamp value — EXCEPT a pair escalated by a NON-choice surfacing cell (the malformed-write cell's route-neither-arm): that keeps the escalating cell's own framing, surfaced, never pickable); the user's ANSWER replaces the pair with the chosen side AND writes a dated D-log event naming the divergence, QUOTING each side verbatim, and marking the chosen one (the milestone cell's verbatim discipline — this record is what the coverage, match/extend, and non-chosen-side tests below read; a loose paraphrase leaves them undecidable) — rewrite and D-line in ONE commit, never-defer transport (Mechanics class 5) — the record the next merge arbitrates by; an AMENDMENT ordered against a phase whose row stands unresolved consults the ORDER's text: an order that NAMES a side (quotes or otherwise discriminates one side's status/stamp) is that ask's answer — record it first (the same rewrite-plus-D-line commit), then arcane's demotion runs on the chosen row; an order referencing only the phase or its spec discriminates NEITHER side — the standing ask fires FIRST (present the pair, take the user's pick) and the amendment proceeds on the answered row: never mint an answer from a non-discriminating order, never demote an unanswered pair; a NORMAL row meeting the pair is the recorded answer ONLY on that D-line's evidence (it matches — or extends with later cumulative evidence: a `branch:` append, a later same-lineage token, or a further status carrying the same token — the CHOSEN side: it wins, never re-asked — EXCEPT against a LATER-dated sanctioned voiding record of the chosen side: a re-gate red or amendment demotion postdating the answer voids the chosen lineage — the demoted shape stands and its own route governs, the answer stays history; a voiding record and the answer with NO strict date order are unorderable → the user, never auto-picked either way; a divergence between the chosen side and its own demoted shape is NOT covered by the answer D-line); ABSENT the D-line, a normal row matching or extending one side is independent advancement by a machine that never saw the conflict, never an answer: it UPDATES that side in place (the side text is replaced by the advanced row — sides stay current, the divergence and the ask stand); a row REWRITE whose was-clause carries a side's own lineage token is that side DEMOTED, never a new side: it REPLACES that side (the voiding travels — the stale pre-demotion text is not pickable, the ask presents the demoted shape); a normal row matching and extending no side JOINS as a new side (the durable form of the new divergence — the user picks among all sides). The same durability governs NON-row →user cells: an unanswered Milestones `status:` cell persists as `status: unresolved (between: <side X> ||| <side Y>; merged <date>)` — the derived top status treats an unresolved milestone as NOT closed (a campaign never completes over it), the milestone-audit route is BARRED on it — an unresolved `status:` is blocked-on-user, never "not closed, all phases done → audit": a fresh audit's later-dated closing record would mechanically supersede the disputed sides and moot the pending arbitration — the same side-union merge cells apply, and the user's answer keeps the chosen shape's OWN historical date while writing the answering D-log event dated by the answer and NAMING the chosen shape verbatim (the event-plus-named-shape pair is the arbitration record — a historical date is never rewritten; one commit, Mechanics class 5) so the next merge arbitrates records, never re-asks; unanswered scope differences persist the disputed ROW as the WHOLE-LINE shape `unresolved-scope (was: <full row>; merged <date>)` — like the between shape, the line replaces the row entirely, never embedded inside a status cell (a full row inside one cell breaks the table's own grammar and truncates every columnar parse; the `was:` row keeps its own cells verbatim) — a sibling shape with the between-row's full consumer set (enum carve-out, standing batch member re-surfacing until the Decision-logged answer, routing stalled on it, the milestone-audit/all-done gate barred while one exists) and its own merge cells: two `unresolved-scope` markings of the same row collapse (the earlier `merged` date); a machine advancing the row against the other's scope marking UPDATES the `was:` row text in place — never consumes the ask; the answer is the Decision-logged user decision (one commit with the row outcome, Mechanics class 5): keep → the row returns as its current `was:` text; remove → the row deletes and the D-line is the durable record — a later merge meeting a still-standing `unresolved-scope` for the same row consults the D-log FIRST: an answering D-line consumes it, never re-durabilized, and when the consumed marking's `was:` text had been UPDATED past the answered text (advanced work), the discarded advance is SURFACED to the user in the same message — every merge order shows the same visibility — arbitrated as records thereafter. Family assignment is exclusive: row-VALUE conflicts durabilize as the between shape in the row's AGREED section — parked on BOTH sides → the pair stays in ## Parked (arbitration never un-parks; a parked pair keeps Parked's exemptions — it stalls no routing and bars no MILESTONE gate, the ask re-surfaces via the standing batch, and the campaign's terminal route waits on it — the completion guard; an un-park request against a parked PAIR first ANSWERS the ask — the user naming a side to un-park IS the answer, class 5 — and the un-park protocol then runs on the chosen row); parking itself disputed (park-vs-advanced) → the ACTIVE section (parking is the claim under dispute; deterministic on both machines); row-MEMBERSHIP conflicts (present on one side only) as `unresolved-scope` — one conflict, one shape. The batch carries escalations answerable at DISCOVERY time; a question whose premise is established by post-routing work (the branch-finish integrate question behind its re-gate, a premortem-BLOCK acceptance decision) is NAMED in the batch ("after routing, X will be asked") but asked in-route by its own protocol — the batch rule governs discovery-time asks, it never front-runs a gate. The migration backfill ask's outcome is RECORDED: backfilled stamps, or `branch: unknown (user <date>)` on rows the user can't reconstruct — a recorded unknown is terminal (never re-asked; the audit treats it like `kept`, no ancestry claim to check), and a declined/unreconstructable backfill also writes its Decision-log EVENT line (the schema's declined-backfill record — the D-log trace of the once-per-campaign ask); a `branch:` part carrying a bare choice word without its artifact (`merged` with no sha — the 0.4.10 shape) joins the same backfill class: recover the artifact from git history with the user aware, never treat the bare word as fully evidenced.

## Creating a campaign

1. Intent and milestones — through catalyst:crucible (the intent is a decision, not a transcription); starchart's finished map converts naturally AND completely: Destination → Intent, closed decisions → Decision log (each D-line carrying its Resolution's ORIGINAL date), ## Out of scope → Non-goals (the scope fence survives the conversion — nothing graduates back silently), ## Notes' standing agreements → Decision log (each with its own date; an UNDATED legacy Notes line converts dated by the CONVERSION and annotated as such — a historical date is never invented), remaining work clusters → phases; the creation D-line names the map path (the pointer starchart promises).
2. Slice phases tracer-first (each phase leaves the system demonstrably better/shippable), 3-9 phases per milestone (a legitimately smaller MILESTONE — 2 phases, a small program or a trailing milestone — is fine: the floor guards against skipped decomposition, not small scope; ONE phase is the single-feature red flag); a phase should fit one arcane-mode plan (a few tasks) — a bigger chunk splits into more phases; a milestone is a set of phases, never one oversized phase.
3. Write both files — PROGRAM.md's `base:` line records the branch the approval commits on (`git branch --show-current` at creation; the schema's field every later ON-base check reads) — get the user's approval of the roadmap as ONE question, set every milestone `status: open` and every phase `pending` — and Decision-log the approval as the creation line naming the approved phase ids (the provenance every later scope check reads).

## Milestone audit — against intent, not the checklist

When a milestone's phases are all `done` — each row carrying its full `done` stamp (a stampless, `branch:`-less, or bare-choice-word row is a backward-reconciliation case, not an audit input), and the audit spot-checks the `branch:` claims against git first: a `merged <sha7>` whose commit is not an ancestor of the base branch is a claim git refutes — a reconciliation case, never an audit input: goal-backward audit of the MILESTONE against `PROGRAM.md ## Intent` and the milestone's must_haves — fresh-eyes auditor (top tier, clean context — the milestone-intent tiering's normative home), the same discipline as arcane verification but one level up — and the audit FIRST sweeps the milestone's rows' spec markers with the terminal sweep's own arms: a pending, double-answered, or vanished-section (lost-referent) entry — crucible's shapes — is a standing user ask that bars closure (batched), exactly like an unresolved shape; a spec unreachable on this machine is a LOST REFERENT, never a silent pass — recover it from git history first (unshallow a shallow clone first), genuinely absent → the sweep bars and surfaces it; a REJECTION answered on a `done` row is a spec-code divergence — the code implements the rejected edit: surfaced with the amendment route offered, closure never rides over it silently: phases all done ≠ intent achieved (integration seams between phases are exactly what per-phase verification never saw). Findings → fix phases into the roadmap (user approves), not silent patches. Convergence uses the same stop criterion — iterative rounds until 2 consecutive clean; then the milestone closes in PROGRAM.md as `closed (audited: <date>, rounds N..M)` PLUS its D-log event line — the closing record other machines arbitrate on; a `closed` written without its record is exactly the hand-edit shape the arbitration refuses to trust; next milestone or campaign end — the user's call. The user's EXPLICIT order to close WITHOUT the audit bends this gate like any named gate (arcane's Iron rules): cost stated in one line, then write `status: closed (waived: <date>)` — the waiver's recorded home — plus the D-log event; a recorded waiver is never re-demanded (a bare `closed` is never freshly written as a closure — it survives only as the migration/hand-edit shape the backfill ask exists for, or standing under a declined-backfill D-line, the schema's terminal record).

## Session end

Campaign sessions end through catalyst:handoff; the handoff carries the campaign path so the next session's router starts warm. Before the handoff, VERIFY the roadmap is already current — statuses flipped when their evidence appeared, each flip carrying its evidence stamp (a stamp is exactly what another machine routes on — a bare flip found here is half-done), links laid; a missed flip found here is fixed now, but session end is the safety net, not the flip point. The roadmap, not the handoff, is the program's memory.

## Red Flags — STOP

- Navigating the program from memory or the conversation instead of reading ROADMAP.md.
- Starting a new phase while another is `executing` ("it's stuck anyway") — resume or explicitly park it with the user first.
- A status advanced without its evidence (specced with no spec file, verified with neither a converged verification nor its recorded waiver line).
- Roadmap scope edited silently — add/remove/reorder is a user decision with a Decision-log line.
- Milestone declared done from the phase checklist without the intent audit.
- Program details accumulating in ROADMAP.md instead of the linked artifacts — the index rots into a monolith.
- A campaign created for a single-feature effort — that's crucible → arcane-mode, the umbrella adds only weight.
- Campaign state uncommitted at session end, or committed onto a phase work branch instead of the base branch. (The one sanctioned deferral: a single-checkout flip edit awaiting the next base checkout per Mechanics — committed at branch finish at the latest; the never-defer CLASS — the `reopened` demotion, a post-flip spec-gate yaml + its spec edits, an amendment wave's edits + pending marker, each recorded pending-entry answer, each unresolved-shape answer, and the hand-demotion suspect's answering D-line — is NOT in it: those commit to base now — and push now when a remote exists, per Mechanics.)
- Routed with a remote configured but without the step-0 pull; or (when a remote exists) the session ended with unpushed state commits / an unpushed phase branch.
- A status flipped without writing its evidence stamp onto the row.
