# Campaign — the program layer

## Overview

Arcane-mode executes one plan; a campaign is the umbrella above many of them: a roadmap of phases toward a milestone, alive across weeks and dozens of sessions. The orchestrator surfacing from one finished branch must never navigate from memory — the campaign state answers "where are we and what's next". Core: GSD's roadmap/milestone lifecycle, reduced to two files over the existing family (no tracker, no config layer).

**Campaign decides sequence; each phase is still delivered by the family flow** (crucible → arcane-mode, whose premortem gate precedes Task 1 → verification). The campaign never re-implements those skills' jobs.

## State (two files)

`<repo>/.catalyst/campaign/<name>/`:


Rules: PROGRAM.md and ROADMAP.md are **committed to the repo** — they are the program's memory for weeks and must survive `git clean`, clones, and machine changes. If the project's `.gitignore` swallows the committed dir — a wholesale `.catalyst/` rule, OR a `.catalyst/*` rule missing this dir's negation (the shape an old half-remedy or a hand edit can leave) — the first state commit fixes the rule (named to the user, never silent): ensure `.catalyst/*` plus `!.catalyst/campaign/` and `!.catalyst/map/` — a bare negation UNDER an ignored directory doesn't work (git never descends into it), and `git add -f` is banned: it leaves the rule in place so every future clone re-hits the same swallow. State commits go to the campaign's BASE branch (the branch phases fork from), never onto a phase work branch — a discarded phase branch must not carry the program's memory away with it. Mechanics: when a remote exists, the router pulls the base checkout FIRST (step 0 below), and session end pushes the state commits AND the phase branch (an unpushed phase branch is unrecoverable from any other machine). Before that pull, a flip edit deferred by the single-checkout workflow (below) is committed FIRST — commit-then-pull, never stash. Pull failure shapes are routed, never improvised: the pull hitting divergence → fetch and merge, conflicts resolved by the arbitration below; remote unreachable → proceed on local state and tell the user the roadmap may be stale. Push failures are routed the same way: a rejected push of STATE commits (the remote advanced during the session) → fetch + merge under the same arbitration, then retry once; still failing, offline, or auth-refused → tell the user the state is local-only (the red flag below is about SILENT unpushed state, not about a named failure). A rejected push of the PHASE branch because it DIVERGED (two machines worked the same phase — nothing detects this earlier) is a CODE merge, not state arbitration: stop, surface to the user whose work continues where, resolve by ordinary git conflict work on one machine — these rules never auto-arbitrate code. State commits verify the base checkout is actually ON the base branch first (`git branch --show-current`): a detached HEAD or a wrong branch is surfaced, never committed onto — a detached-HEAD commit lands on no branch and dies silently in the reflog. Merge-conflict arbitration for ROADMAP.md/PROGRAM.md: per phase row, a status WITH its evidence stamp beats one without; both stamped → the further status wins (evidence is cumulative). The rounds-keyed EXCEPTs below compare LINEAGE TOKENS (schema): a waived lineage compares its waiver dates the same way, and a `done` vs a `verified` from DIFFERENT lineages (one carries rounds, the other waived) → the user — the two sides' verification histories diverged. EXCEPT `done` vs a `verified` whose token differs from the token in the `done` stamp: that acceptance was overtaken by a re-opened verification (acceptance must reference the final head — arcane's verification rule), and "cumulative" is false → the user; and EXCEPT `done` vs a `reopened` row whose "was" token EQUALS the `done` stamp's token — equality here is the voiding signature, not cumulative evidence (that `done` IS the acceptance the demotion voided; this is what the "was" token exists for): the `reopened` side wins, the `done` side's later `branch:` append (if any) is surfaced to the user as an integration executed on a voided acceptance; a `done` vs a `reopened` whose "was" token DIFFERS from the `done`'s token is NOT that cell — compare DIRECTION (tokens are monotonic within a phase, the schema invariant): the `done` token LATER than the "was" token = a post-re-verification acceptance, cumulative — the `done` side wins mechanically; the `done` token EARLIER = an acceptance even older than the one this demotion voided, superseded generations ago — the `reopened` side wins; tokens of different kinds or otherwise unorderable → the user, the histories diverged (the first EXCEPT reads a `verified` row's live token, never a `reopened` row's "was" token); two `done` rows where one side merely EXTENDS the other's stamp with its `branch:` part → the superset wins mechanically (the append is cumulative evidence, no escalation); a row parked on one side and advanced in ## Phases on the other → the user ("further wins" never auto-un-parks: parking is a user decision, and a user decision is never silently discarded); a row parked on BOTH sides with differing preserved status/stamp values arbitrates the preserved pair by these same per-row rules — the winner stays PARKED (arbitration never un-parks); differing park reasons/dates → the user; conflicting `done` rows compare DIRECTION like the reopened cells: orderable tokens → the later-token side wins mechanically (a post-re-verification acceptance supersedes a stale one) — and the LOSING side's `branch:` append (if any) is surfaced to the user as an integration executed on a superseded acceptance, never dropped silently; unorderable → the user; neither stamped, or two different values for the same stamp → the user; scope differences (added/removed phases, parking) reconcile through the Decision log with the user — never blind ours/theirs. PROGRAM.md's own sections arbitrate too, each by its nature: the Decision log is an append-only EVENT LOG — a merge UNIONS both sides' lines by CONTENT, never drops either (both machines' events happened); D-ids are per-file ordinals, never cross-machine references — consumers match D-lines by content (kind + phase ids + date; every D-line carries its date), so after the union the log renumbers sequentially by date — same-date ties break by kind, then referent ids, then text, lexicographically (a stated total order: the same final log from any merge order); two lines recording the SAME event (same kind and referents, different wording) collapse to one, keeping the fuller text; the Milestones `status:` line arbitrates by the RECORDS' DATES, never by shape precedence — every transition writes a dated record (a closure: the audited/waived shape plus its D-log event; a RE-OPEN: the un-park rule's D-log line — a record too): the later-dated record wins (a re-open dated after a closure re-opens it everywhere — a user's recorded re-open is never mechanically overridden by a stale closure; a later re-closure closes it again), and this covers closed-vs-closed of any shapes the same way (a post-re-open waiver beats a stale audit; a later audit beats an older waiver); recordless sides (a bare `closed` — a hand edit or the migration shape), same-date records, or any doubt → the user; the top `status: active|complete` line is DERIVED — never arbitrated, recomputed from the merged Milestones section (the writer that flips it writes the date of ITS flip). Rows stamped by pre-0.4.10 versions (a bare `done (accepted: <date>)`, no lineage token, no `branch:`) are a MIGRATION shape, not deaths: ask the user once per campaign to backfill the executed choices — never re-execute them blindly; a missing lineage token arbitrates as unknown (→ the user). Run phases in WORKTREES off the base (the default — the base checkout stays available), and **the session that produces the evidence flips the status in the base checkout and commits there immediately**, not at handoff time (the Session-end check is a safety net, not the flip point). Single-checkout branch workflow: edit the flip immediately, commit it at the next moment base is checked out — branch finish at the latest, never onto the phase branch. ONE exception never defers: the `reopened` demotion exists to be visible cross-machine DURING the fix wave (branch finish — the usual deferral backstop — is exactly what the red suspended), so commit it to base NOW via a momentary base checkout or a base worktree; the phase branch keeps the working state. Parallel phases (user-sanctioned): the sanction is DECISION-LOGGED (a D-line naming the sanctioned phases and each phase's owner as a user-legible label — a machine, a worktree, a person) — the durable trace every machine reads; without it a two-`executing` roadmap is indistinguishable from the violation. ONE session, named by the user, owns all roadmap flips — the others leave the evidence on disk and name it in their handoffs; the owner (or the next router run's reconciliation) flips from it. ROADMAP is an index — details live in the linked specs/ledgers. **Status changes only with evidence, and the flip writes the evidence stamp onto the row** (stamp formats in the schema above) — and when the stamp's referent is a repo file (the spec and its spec-gate premortem yaml for `specced`, the plan and its premortem yaml for `planned`), the flip's state commit CARRIES that file into the base branch — with the transport NAMED: the file is usually produced on the phase branch/worktree, so bring it into the base checkout explicitly (`git checkout <phase-branch> -- <path>`, or copy from the worktree) and commit it together with the flip; a stamp whose repo-file referent is uncommitted on base is half-done — "evidenced on every machine" holds only because the evidence travels with the commit. A `discarded` branch-finish choice never deletes the branch before the row's referents are committed on base — the branch may hold their only copies: an APPROVED spec (crucible's `status: approved` marker, including its skip-order shape `status: approved (waived: <date>)` — a markerless file is a draft, not evidence) → specced; premortem PASS/WARN — BLOCK bars it — plus the written plan → planned; ledger shows tasks closing → executing; the ledger's `verification: converged` line → verified — OR its sanctioned substitute `verification: waived by user <date> — <scope>` (economics' review-floor drop, recorded; the stamp is `verified (waived: <date>)`, an honest mark every reader sees: the rounds did NOT run, the user chose that, and no session re-demands them); user accepted → done (no other acceptance artifact exists — an unstamped `done` is unevidenced by definition). Roadmap edits that change scope (add/remove/reorder phases) are user decisions, logged in the Decision log — never silent. All escalations ONE router run discovers (arbitration cells, reconciliation gaps, the migration backfill ask) are presented to the user as ONE batched question — arbitration cells first, then reconciliation gaps, then routing — never an interrogation drip, and never silently self-resolving the "small" arms to spare the user. The batch carries escalations answerable at DISCOVERY time; a question whose premise is established by post-routing work (the branch-finish integrate question behind its re-gate, a premortem-BLOCK acceptance decision) is NAMED in the batch ("after routing, X will be asked") but asked in-route by its own protocol — the batch rule governs discovery-time asks, it never front-runs a gate. The migration backfill ask's outcome is RECORDED: backfilled stamps, or `branch: unknown (user <date>)` on rows the user can't reconstruct — a recorded unknown is terminal (never re-asked; the audit treats it like `kept`, no ancestry claim to check), and a declined/unreconstructable backfill also writes its Decision-log EVENT line (the schema's declined-backfill record — the D-log trace of the once-per-campaign ask); a `branch:` part carrying a bare choice word without its artifact (`merged` with no sha — the 0.4.10 shape) joins the same backfill class: recover the artifact from git history with the user aware, never treat the bare word as fully evidenced.

## Router — every campaign session starts here

**Step 0 — pull first** when a remote exists (Mechanics above; failure shapes routed there). Then read ROADMAP.md and PROGRAM.md's Milestones section (nothing else yet) **from the BASE checkout** — a phase worktree's copy is stale as of its fork; resuming inside a worktree (e.g. via a handoff), locate the base checkout first, and anchor every `.catalyst/` path to the repo root, never the cwd; a roadmap failing self-consistency splits by severity: duplicate phase ids or a status outside the enum make the roadmap structurally unreadable — surfaced alone, routing aborts before any batch; a phase id no Decision-log line accounts for (neither the creation line nor a later scope line — a hand-added row; scope edits are never silent), a row returned from ## Parked with no un-park D-line, a `closed` milestone whose status carries neither the audited nor the waived record (a hand edit, or the pre-0.4.16 migration shape — the backfill ask, dated, never skipped past as closed), and a `closed` milestone one of whose rows is NOT `done` (a `reopened` row, an un-parked phase — arbitration or hand edits can produce it: the closure no longer holds, re-open it with the user per the un-park rule) — these are BATCH members, surfaced with the run's one question, never routed past and never an interrogation drip (a roadmap predating the creation-line rule has its provenance baseline in git instead: ids present in ROADMAP.md's first committed version are accounted for — unshallow a shallow clone first, its truncated history's "first version" is a late snapshot). Every "surface to the user" the rest of this router run produces is a MEMBER of the one batched question (Rules) — only that self-consistency failure aborts routing on its own, before any batch exists. An `executing` phase ANYWHERE in the campaign takes precedence: resume it first — milestone ordering governs which phase STARTS next, it never preempts one mid-flight (so a milestone re-opened by an un-park queues behind a later milestone's executing phase; the one-executing-phase rule wins). TWO `executing` rows are consistent only with the parallel-sanction D-line in the Decision log whose named phase ids MATCH the currently-executing rows (a sanction names specific phases — one naming other ids is spent history, not a standing license): matching → each session resumes the row whose owner label on the D-line is its own; a session that can't tell which label it is, or whose label isn't there, ASKS the user rather than squatting on an owned row (two sessions in one phase break the single-writer workspace); no matching D-line → surface to the user, never auto-park either row. Otherwise take the first milestone whose `status:` is not `closed`: if ALL its phases are `done`, the route is its milestone audit — a later milestone's pending phases never skip an unclosed earlier one past its audit. Otherwise route by that milestone's first phase whose status is not `done`. Before dispatching into the route, reconcile BOTH ways: (a) forward — if the phase's NEXT evidence already exists on disk (an approved spec for `pending`, the ledger's `verification: converged` line — OR its sanctioned substitute `verification: waived by user <date> — <scope>` — as the ledger's final line for `executing` — trailing `session:`/`uat:`/`minor:`/`disposition:` lines are sanctioned (verification.md's currency rule), but TASK lines below that final verification line mean re-opened verification that never refreshed — under a converged line it is STALE, route resume; under a waived line the waiver is state-specific, route the disposition ask (verification.md); don't flip either way; resolve the ROADMAP ledger link, which may point into a phase worktree; an unreachable ledger is NOT evidence: route resume, and arcane re-derives from `git log` only on the machine that has the phase branch — on a fresh clone without it, the phase's progress is unrecoverable: say so to the user, never silently restart the phase), the previous session died in the flip window — flip (with the stamp) first, then route; (b) backward — over EVERY row of the routed milestone, not just the routed phase (this is the only checkpoint the milestone-audit and terminal routes pass through — a hand-edited stampless `done` on the LAST phase must not slip a milestone closed): a status whose row lacks its evidence STAMP and whose evidence isn't reachable locally either (`planned` with neither stamp nor plan file, `done` with no acceptance stamp — a hand edit?) → do NOT route on it: surface the gap to the user with the evidence rule quoted; a `done` stamp missing its `branch:` part — or carrying a bare choice word without its artifact (`merged` with no sha, the 0.4.10 shape: recover the artifact from git history with the user aware, per the backfill class): a BARE pre-0.4.10 stamp (no lineage token either) is the migration shape — the backfill ask per the arbitration rules, never a re-execution; a token-carrying one died in the choice window — verify against git FIRST: base already contains the phase head → the choice DID execute, recover the merge sha from `git log` and append the artifact instead of re-asking (ancestry lies in two known shapes: a squash-merge integration leaves the phase head a NON-ancestor — check the integration history for the squash commit before concluding "never executed"; a shallow clone unshallows first — a false "not integrated" here re-opens an integrated phase and can void its acceptance); otherwise route to arcane's branch-finish question for THAT phase first, never past it ("keep the branch as-is" is itself a recorded choice — `branch: kept`); if the re-gate on that resume goes red, the fix re-opens convergence and VOIDS the acceptance (it no longer references the final head): demote the row IMMEDIATELY at the red — `verified (converged: reopened <date>; was rounds N..M)` (waived lineage: `verified (waived: reopened <date>; was waived <date>)`), the one sanctioned backward flip, committed before the fix wave starts so no other machine integrates on the voided acceptance meanwhile — the voiding is named to the user and Decision-logged (a recorded acceptance never vanishes silently); re-verification rewrites the stamp with its refreshed lineage token (a waived lineage routes the post-fix disposition ask first — the `verified` route below), and acceptance re-runs before any new `done` with the UAT pass re-offered in FULL (the fix changed the behavior the old `uat:` lines recorded — they are void with the acceptance); a stamp git REFUTES on a full history (a `merged <sha7>` that is not an ancestor of the base branch) → surface to the user with both facts, never routed on and never auto-repaired — and on a shallow clone an absent commit proves nothing: unshallow first, refute only on full history; a stampless row whose evidence IS locally reachable → write the missing stamp from that evidence now (a repair, committed), then route — left stampless, the row loses the next stamp-beats-stampless arbitration to any stale stamped counterpart; a stamped row is evidenced on every machine ONCE its repo-file referent is committed: for `specced`/`planned` rows resolve the referent path in the BASE tree — an absent referent is a half-done flip or a pre-0.4.16 row, not a death: recover the file from the phase branch/worktree when reachable and commit it (a repair), otherwise surface it in the batch — NEVER silently re-run crucible or plan-writing over an approved-but-uncommitted artifact; rows whose stamps carry no repo-file referent need no reach — no false alarm. Then warm up: if `.catalyst/handoffs/` (or `thoughts/shared/handoffs/` — the ContinuousClaude tree handoff writes to when the project has one) holds a handoff carrying this campaign's pointer, read the latest one — the roadmap decides WHAT happens next, the handoff's mental model saves re-discovering HOW things work.

| Every milestone `closed` | the campaign is COMPLETE: write `status: complete` (with the date) at the top of PROGRAM.md, commit, tell the user — nothing routes; bootup skips complete campaigns. Both evidenced shapes count — `closed (audited: …)` and `closed (waived: …)`; a bare `closed` is the self-consistency backfill case, never silently complete |

One session advances ONE phase's state as far as it honestly goes; parallel phases only by explicit user decision and only when file-disjoint.

## Creating a campaign

1. Intent and milestones — through catalyst:crucible (the intent is a decision, not a transcription); starchart's finished map converts naturally: destination → Intent, closed decisions → Decision log, remaining work clusters → phases.
2. Slice phases tracer-first (each phase leaves the system demonstrably better/shippable), 3-9 phases per milestone; a phase should fit one arcane-mode plan (a few tasks) — a bigger chunk splits into more phases; a milestone is a set of phases, never one oversized phase.
3. Write both files, get the user's approval of the roadmap as ONE question, set every milestone `status: open` and every phase `pending` — and Decision-log the approval as the creation line naming the approved phase ids (the provenance every later scope check reads).

## Milestone audit — against intent, not the checklist

When a milestone's phases are all `done` — each row carrying its full `done` stamp (a stampless, `branch:`-less, or bare-choice-word row is a backward-reconciliation case, not an audit input), and the audit spot-checks the `branch:` claims against git first: a `merged <sha7>` whose commit is not an ancestor of the base branch is a claim git refutes — a reconciliation case, never an audit input: goal-backward audit of the MILESTONE against `PROGRAM.md ## Intent` and the milestone's must_haves — fresh-eyes auditor (top tier, clean context), the same discipline as arcane verification but one level up: phases all done ≠ intent achieved (integration seams between phases are exactly what per-phase verification never saw). Findings → fix phases into the roadmap (user approves), not silent patches. Convergence uses the same stop criterion — iterative rounds until 2 consecutive clean; then the milestone closes in PROGRAM.md as `closed (audited: <date>, rounds N..M)` PLUS its D-log event line — the closing record other machines arbitrate on; a `closed` written without its record is exactly the hand-edit shape the arbitration refuses to trust; next milestone or campaign end — the user's call. The user's EXPLICIT order to close WITHOUT the audit bends this gate like any named gate (arcane's Iron doctrine): cost stated in one line, then write `status: closed (waived: <date>)` — the waiver's recorded home — plus the D-log event; a recorded waiver is never re-demanded (a bare `closed` is never written by anyone — it is the migration/hand-edit shape the backfill ask exists for).

## Session end

Campaign sessions end through catalyst:handoff; the handoff carries the campaign path so the next session's router starts warm. Before the handoff, VERIFY the roadmap is already current — statuses flipped when their evidence appeared, each flip carrying its evidence stamp (a stamp is exactly what another machine routes on — a bare flip found here is half-done), links laid; a missed flip found here is fixed now, but session end is the safety net, not the flip point. The roadmap, not the handoff, is the program's memory.

## Red Flags — STOP

- Navigating the program from memory or the conversation instead of reading ROADMAP.md.
- Starting a new phase while another is `executing` ("it's stuck anyway") — resume or explicitly park it with the user first.
- A status advanced without its evidence (specced with no spec file, verified with neither a converged verification nor its recorded waiver line).
- Roadmap scope edited silently — add/remove/reorder is a user decision with a Decision-log line.
- Milestone declared done from the phase checklist without the intent audit.
- Program details accumulating in ROADMAP.md instead of the linked artifacts — the index rots into a monolith.
- A campaign created for a single-feature effort — that's crucible → arcane-mode, the umbrella adds only weight.
- Campaign state uncommitted at session end, or committed onto a phase work branch instead of the base branch. (The one sanctioned deferral: a single-checkout flip edit awaiting the next base checkout per Mechanics — committed at branch finish at the latest.)
- Routed with a remote configured but without the step-0 pull; or the session ended with unpushed state commits / an unpushed phase branch.
- A status flipped without writing its evidence stamp onto the row.
