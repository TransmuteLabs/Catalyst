# Campaign — the program layer

## Overview

Arcane-mode executes one plan; a campaign is the umbrella above many of them: a roadmap of phases toward a milestone, alive across weeks and dozens of sessions. The orchestrator surfacing from one finished branch must never navigate from memory — the campaign state answers "where are we and what's next". Core: GSD's roadmap/milestone lifecycle, reduced to two files over the existing family (no tracker, no config layer).

**Campaign decides sequence; each phase is still delivered by the family flow** (crucible → arcane-mode, whose premortem gate precedes Task 1 → verification). The campaign never re-implements those skills' jobs.

## State (two files)

`<repo>/.catalyst/campaign/<name>/`:


Rules: PROGRAM.md and ROADMAP.md are **committed to the repo** — they are the program's memory for weeks and must survive `git clean`, clones, and machine changes. No git repo at all → the state cannot be committed: surface it to the user (init a repo, or an explicitly accepted ephemeral run), never proceed silently — starchart's arm. If the project's `.gitignore` swallows the committed dir — a wholesale `.catalyst/` rule, OR a `.catalyst/*` rule missing this dir's negation (the shape an old half-remedy or a hand edit can leave) — the first state commit fixes the rule (named to the user, never silent): ensure `.catalyst/*` plus `!.catalyst/campaign/` and `!.catalyst/map/` — a bare negation UNDER an ignored directory doesn't work (git never descends into it), and `git add -f` is banned: it leaves the rule in place so every future clone re-hits the same swallow.

State commits go to the campaign's BASE branch (the branch phases fork from), never onto a phase work branch — a discarded phase branch must not carry the program's memory away with it. Mechanics: when a remote exists, the router pulls the base checkout FIRST (step 0 below), and session end pushes the state commits AND the phase branch (an unpushed phase branch is unrecoverable from any other machine). Before that pull, a flip edit deferred by the single-checkout workflow (below) is committed FIRST — commit-then-pull, never stash. Pull failure shapes are routed, never improvised: the pull hitting divergence → fetch and merge, conflicts resolved by the cells in `references/arbitration.md`; remote unreachable → proceed on local state and tell the user the roadmap may be stale; a remote ATTACHED mid-campaign (no upstream at contact) → set the upstream, push the base state, and PUBLISH every live local referent from the machine that has it — kept and executing phase branches, the never-defer history — named to the user; the publish duty keys on ANY live local referent absent from the remote, checked at step 0 whenever a remote exists (a partial publish resumes — never first-contact-only); UNRELATED histories at contact (two machines attached the same fresh remote independently) → fetch and merge with the explicit unrelated-histories allowance — the one sanctioned use — file-level add/adds of STATE files (the campaign dir, flip-carried specs/plans/premortem yamls) arbitrated by `references/arbitration.md`'s cells (both trees are legitimate state), while SOURCE-file add/adds are ordinary git conflict work surfaced to the user — these rules never auto-arbitrate code, the phase-divergence principle — never a force-push (until published, other machines legitimately read local-only state through the recovery arms). Push failures are routed the same way: a rejected push of STATE commits (the remote advanced during the session) → fetch + merge under the same arbitration, then retry once; still failing, offline, or auth-refused → tell the user the state is local-only (the red flag below is about SILENT unpushed state, not about a named failure). A rejected push of the PHASE branch because it DIVERGED (two machines worked the same phase — nothing detects this earlier) is a CODE merge, not state arbitration: stop, surface to the user whose work continues where, resolve by ordinary git conflict work on one machine — these rules never auto-arbitrate code. EVERY state write — an answer recording, a flip, a demotion — RE-READS its referent from disk immediately before writing: a referent changed since the ask/derivation was built (another session's commit — two windows on ONE checkout produce the merge problem WITHOUT a merge, so no cell would ever fire) means the ask is STALE — adopt the landed record like a fetched answer, re-derive, never overwrite (a recorded decision is never last-writer-wins; adoption covers asks not yet answered locally — a landed record that CONFLICTS with an answer the user already gave THIS session is two user decisions → the user per arbitration's conflicting-answers rule, never a silent drop of either). The re-read NARROWS the race window, it does not close it (two writers can both read clean and then both write — TOCTOU): the serialization anchor is the COMMIT — every state write commits IMMEDIATELY (the never-defer discipline), and the commit step carries its OWN guard, because git's index lock serializes only simultaneous commit invocations — a second SEQUENTIAL committer writing from a stale full-file buffer commits cleanly and silently reverts the sibling's landed record, no conflict and no failure: immediately before staging, re-check that the file's blob at HEAD still equals the blob the pre-write re-read saw (`git rev-parse HEAD:<path>` against the hash noted at re-read; changed → a sibling landed since — re-read and re-derive, never proceed); a write lost in the residual check-to-commit window is DETECTED by the D-log/self-consistency sweep (a sibling's landed record vanished against the pre-write head) and RESTORED from git history — then the two records arbitrate normally, nothing is last-writer-wins, and state files are edited surgically in place, never regenerated from a session's in-memory model (a regeneration silently reverts every sibling's landed write). State commits verify the base checkout is actually ON the base branch first — `git branch --show-current` compared against PROGRAM.md's `base:` line (the schema; a legacy file without it backfills the line first), never a guessed default: a detached HEAD or a wrong branch is surfaced, never committed onto — a detached-HEAD commit lands on no branch and dies silently in the reflog. **Arbitration & legacy shapes — the normative home is `references/arbitration.md`.** Any merge conflict in state files (ROADMAP.md/PROGRAM.md/specs/plans/premortem yamls), any standing `unresolved (between: …)` / `unresolved-scope (was: …)` / `base: unresolved (…)` shape, a base:-answer D-line whose transport receipt is missing (the answer commit not an ancestor of the base's head), the user's ORDER to change the base, and any legacy shape WITHOUT its terminal record (a bare `done (accepted: <date>)`, a bare milestone `closed`, a single-pipe pair, a missing `base:` line — but a bare `closed` standing under its declined-backfill D-line is RESOLVED: no read owed, the router's inline terminal rule handles it) → STOP and read that file ENTIRELY before resolving, repairing, or routing past anything — its cells are normative, and resolving a state conflict without them is a defect. Its →user outcomes are members of the one batched question (below); its recorded answers are never-defer commits (the class below).

Run phases in WORKTREES off the base (the default — the base checkout stays available), and **the session that produces the evidence flips the status in the base checkout and commits there immediately**, not at handoff time (the Session-end check is a safety net, not the flip point). Single-checkout branch workflow: edit the flip immediately, commit it at the next moment base is checked out — branch finish at the latest, never onto the phase branch. The NEVER-DEFER CLASS — commits that exist to be seen cross-machine — never defers, each via a momentary base checkout or a base worktree (the phase branch keeps the working state): (1) the `reopened` demotion — visible DURING the fix wave (branch finish, the usual deferral backstop, is exactly what the red suspended): commit it to base NOW; (2) a spec-gate premortem run after the `specced` flip — its yaml and spec edits (premortem's transport rule); (3) an AMENDMENT wave's spec edits + pending marker (arcane's route — produced by crucible's fast path, same transport: base must show the pending shape, or every other machine consumes the stale approved spec); (4) each recorded ANSWER to a pending re-approval entry (approve, reject, or arbitrate — premortem's one-commit-per-answer); (5) each recorded ANSWER to a standing unresolved shape (row, milestone status, or scope) — the rewrite and its D-log event in ONE commit (separated, the answered row reads as independent advancement and the ask re-fires); (6) the hand-demotion suspect ask's answering D-line (the router's accounting answer — the record that stops the ask re-firing on other machines); (7) an acceptance WITHDRAWAL — the `withdrawn:` append plus its recompute in one commit (premortem's rule: no machine may dispatch on a reversed decision). The never-defer class OUTRANKS parallel-sanction flip ownership: a non-owner session with a class member (a demotion at a red, an answer) commits and pushes it NOW — ownership governs ordinary evidence flips only, never this class (the stale-write guard above covers the shared-file race). Every never-defer commit also PUSHES now when a remote exists — a base commit is invisible to every other machine until pushed, and this class exists precisely to be seen before the session ends (the push-failure routing above applies: a named failure, never silent; the pushed BASE commit is what lets the bridge read the yaml from the base tree on another machine before any branch finish — recovery from a pushed phase branch stays the fallback, premortem's lost-referent arm).

Parallel phases (user-sanctioned): file-disjointness is a HARD precondition checked when writing the sanction D-line (overlapping `Files:` across the sanctioned phases' plans → surface it, no sanction; a later plan edit that introduces overlap voids the sanction — surface, never silently run on); the sanction is DECISION-LOGGED (a D-line naming the sanctioned phases and each phase's owner as a user-legible label — a machine, a worktree, a person) — the durable trace every machine reads; without it a two-`executing` roadmap is indistinguishable from the violation. ONE session, named by the user, owns all roadmap flips — the others leave the evidence on disk, name it in their handoffs, AND (when the owner is another MACHINE — ledgers are self-ignored and handoffs machine-local, so on-disk evidence never crosses machines) record a committed evidence-receipt D-line on base (`evidence: P<N> <payload>, head: <sha> (<machine label>, <date>)`, never-defer): the owner flips from the receipt, which is the durable provenance every machine reads; without it a cross-machine owner has no lawful flip source. The receipt covers EVERY owned flip and append, not only verification, and it always names the phase branch head it attests — a receipt is the writer's OWN attestation, so it is tied to concrete fetchable state, never a bare claim; the per-flip payload grammar, the head-OID rule, the receipt's event identity, and the contradicted-receipt arm live in `references/arbitration.md` (the receipt norm home) — an owned transition with no receipt form would strand cross-machine exactly like the verification flip once did.

ROADMAP is an index — details live in the linked specs/ledgers. **Status changes only with evidence, and the flip writes the evidence stamp onto the row** (stamp formats in the schema above) — and when the stamp's referent is a repo file (the spec and its spec-gate premortem yaml for `specced`, the plan and its premortem yaml for `planned`), the flip's state commit CARRIES that file into the base branch — with the transport NAMED: the file is usually produced on the phase branch/worktree, so bring it into the base checkout explicitly (`git checkout <phase-branch> -- <path>`, or copy from the worktree) and commit it together with the flip; a stamp whose repo-file referent is uncommitted on base is half-done — "evidenced on every machine" holds only because the evidence travels with the commit. A `discarded` branch-finish choice never deletes the branch before the row's referents are committed on base — the branch may hold their only copies: an APPROVED spec (crucible's `status: approved` marker, including its skip-order shape `status: approved (waived: <date>)` and the pending shape `status: approved (edits pending re-approval: <date>)` (keyed on the clause — the waived combo carrying it is pending too) — approved for evidence, but arcane's input rule owes its fast re-approval before the pending sections are consumed; a markerless file is a draft, not evidence) → specced; premortem PASS/WARN — BLOCK bars it — plus the written plan → planned; ledger shows tasks closing → executing; the ledger's `verification: converged` line → verified — OR its sanctioned substitute `verification: waived by user <date> — <scope>` (economics' review-floor drop, recorded; the stamp is `verified (waived: <date>)`, an honest mark every reader sees: the rounds did NOT run, the user chose that, and no session re-demands them); user accepted → done (no other acceptance artifact exists — an unstamped `done` is unevidenced by definition).

Roadmap edits that change scope (add/remove/reorder phases) are user decisions, logged in the Decision log — never silent.

All escalations ONE router run discovers (arbitration cells, reconciliation gaps, the migration backfill ask) are presented to the user as ONE batched question — arbitration cells first, then reconciliation gaps, then routing — never an interrogation drip, and never silently self-resolving the "small" arms to spare the user. A LARGE batch is still one message, walked as a numbered list the user answers item-by-item (the batch bounds the interruptions, not the answers' depth — crucible's one-question doctrine governs each item); a PARTIAL answer records the answered members' outcomes at the moment they are given (never re-asked — the same at-the-moment rule as the disposition trailer), while unanswered members keep their routes blocked and re-surface in the NEXT run's batch: a partial answer never unblocks the unanswered arms and never re-litigates the answered ones. The batch carries escalations answerable at DISCOVERY time; a question whose premise is established by post-routing work (the branch-finish integrate question behind its re-gate, a premortem-BLOCK acceptance decision) is NAMED in the batch ("after routing, X will be asked") but asked in-route by its own protocol — the batch rule governs discovery-time asks, it never front-runs a gate.

## Router — every campaign session starts here

**Step 0 — pull first** when a remote exists (Mechanics above; failure shapes routed there) — the branch pulled and read is PROGRAM.md's `base:` line, read from the local file BEFORE the pull (the schema; a legacy file without the line runs its backfill arm first; a `base: unresolved (…)` value is arbitration.md's standing ask, not a branch name — routing suspends on it), never a guessed default; STEP 0 LOOPS: if the pulled state rewrites `base:` (or delivers a base-change / divergence-answer D-line), re-run step 0 against the NEWLY named base before reading anything else — the first pull was consumed by the old branch, and the new base's local checkout may be stale (bounded: a second rewrite on the re-run durabilizes `base: unresolved` → the user, never a chase); and when the local Decision log carries a base:-ANSWER or base-change D-line (both kinds — the order path records `base change`, not `divergence answer`), verify its TRANSPORT RECEIPT per arbitration.md's receipt rule (ancestry, its carry-receipt D-line for squash bases, its shallow caveat) — not receipted → the owed carry runs FIRST, before anything routes on that base (a cheap check that only fires when such a D-line exists). Then read ROADMAP.md and PROGRAM.md's Milestones section and Decision log — the router's own checks below read D-lines — (nothing else yet) **from the BASE checkout** — a phase worktree's copy is stale as of its fork; resuming inside a worktree (e.g. via a handoff), locate the base checkout first, and anchor every `.catalyst/` path to the repo root, never the cwd; the self-consistency sweeps are INCREMENTAL where an anchor exists: a session re-sweeps only the D-lines and rows the pull/merge INTRODUCED (the diff against the pre-pull local head — git shows it); the FULL-log sweep runs on first contact with a campaign on this machine (no prior local head) — never on every run of a long campaign (the spec test's frontier principle; an agent-executed full scan of hundreds of lines gets silently skimmed under pressure, which defeats the detection); a roadmap failing self-consistency splits by severity: duplicate phase ids or a status outside the enum make the roadmap structurally unreadable — surfaced alone, routing aborts before any batch (the sanctioned merge shapes — `unresolved (between: … ||| …; merged <date>)` and `unresolved-scope (was: …; merged <date>)` — are NOT outside the enum (references/arbitration.md): standing batch members, never an abort); a phase id no Decision-log line accounts for (neither the creation line nor a later scope line — a hand-added row; scope edits are never silent), the MIRROR orphan — a D-line ASSERTING A LIVING REFERENT whose named phase id or milestone appears in no roadmap row (kinds whose semantics REMOVE their referent — a scope-remove answer — are expected-absent by design, never orphans (a PARKING event's referent lives in ## Parked and resolves as a ROW — parked rows are rows, never expected-absent), and a creation line whose row a later remove-answer D-line accounts for is paired WITHIN the log: the check is log-internal first, no history archaeology in the happy path; only an unpaired living-referent D-line escalates to row history, and history absence concludes only on FULL history — unshallow a shallow clone first), a record fabricated beside nothing, or a rewrite that never landed: the suspect shape — paired writes land together, so an unpaired half is surfaced, never trusted alone, a row returned from ## Parked with no un-park D-line, a `reopened` row with NO accounting D-line (every sanctioned demotion binds one into its commit — a voiding event or the amendment D-line; EXCEPT a reopened row whose spec carries the pending marker: the marker-only legacy amendment shape, sanctioned; without either, the stamp is the hand-demotion suspect, batched BEFORE any fix-wave resume, never routed silently — its resolution protocol (both facts presented, attest and can't-attest outcomes, the never-defer answer write) is arbitration.md's cell), a `closed` milestone whose status carries neither the audited nor the waived record (a hand edit, or the pre-0.4.16 migration shape — the backfill ask, dated, never skipped past as closed; EXCEPT one whose declined-backfill D-line exists — that record is terminal, not a batch member), and a `closed` milestone one of whose rows is NOT `done` (a `reopened` row, an un-parked phase — arbitration or hand edits can produce it: the closure no longer holds, re-open it with the user per the un-park rule) — these are BATCH members, surfaced with the run's one question, never routed past and never an interrogation drip (a roadmap predating the creation-line rule — or one whose creation line a torn genesis lost, indistinguishable from state — has its provenance baseline in git instead: ids present in ROADMAP.md's first committed version are accounted for — unshallow a shallow clone first, its truncated history's "first version" is a late snapshot).

Every "surface to the user" the rest of this router run produces is a MEMBER of the one batched question (Rules) — only that self-consistency failure aborts routing on its own, before any batch exists.

An `executing` phase ANYWHERE in the campaign takes precedence: resume it first — and this scan reads WRAPPED sides too: an `unresolved (between: …)` row any of whose sides carries `executing`, and an `unresolved-scope` whose was-row does, COUNT as executing rows for this precedence check and for the two-`executing` consistency check below, in EVERY milestone (the wrapper must not hide a live phase; a wrapped executing row is not resumable — it holds its milestone on the ask instead, and no new phase starts anywhere while it stands unanswered, the same bar as a plain executing row) — milestone ordering governs which phase STARTS next, it never preempts one mid-flight (so a milestone re-opened by an un-park queues behind a later milestone's executing phase; the one-executing-phase rule wins).

TWO OR MORE `executing` rows (the sanction defines N ≥ 2 — three sanctioned phases are three executing rows, same rules) are consistent only with the parallel-sanction D-line in the Decision log whose named phase ids COVER the currently-executing rows — the live executing set must be a SUBSET of one open sanction's ids (a phase finishing normally shrinks the set without spending the sanction; a sanction naming only other ids is spent history, not a standing license; when the LAST sanctioned phase closes, the sanction is spent): matching → each session resumes the row whose owner label on the D-line is its own; a session that can't tell which label it is, or whose label isn't there, ASKS the user rather than squatting on an owned row (two sessions in one phase break the single-writer workspace); no matching D-line → surface to the user, never auto-park either row.

Otherwise take the first milestone whose `status:` is not `closed`: if ALL its phases are `done`, the route is its milestone audit (an `unresolved (between: …)` milestone `status:` is blocked-on-user — arbitration.md's bar — never auditable) — a later milestone's pending phases never skip an unclosed earlier one past its audit. Otherwise route by that milestone's first phase whose status is not `done` — and when that phase's row stands in an unresolved shape (a standing ask), OR is a `verified` row under a standing acceptance decline (its `acceptance declined:` D-line recorded and no NEW verification evidence since — the decline freezes the row: the acceptance ask is barred until new evidence or a re-open, and the re-gate is deliberately deferred with it — running it would either change nothing or force a fix wave on a phase the user just declined to move, so the row waits for one of the decline's two exits), route the milestone's NEXT not-done phase instead — skipping any further phase that itself stands in an unresolved shape by the same rule; no candidate left → the milestone stalls — (the stall binds THE ROW and the milestone's closure, not every disjoint phase: phase order within a milestone yields to a standing ask UNLESS a later phase's spec/plan names a dependency on the stalled one, or ANY side of the unresolved pair — or an `unresolved-scope` was-row — carries `executing`: the campaign-wide executing scan above already counts such a row as executing (a phase may be live on another machine) — it holds the whole campaign's new starts on the ask, never relieved; the shape stays a batch member every run, and the milestone cannot close or audit over it — program-wide suspension is reserved for `base: unresolved`).

Before dispatching into the route, reconcile BOTH ways: (a) forward — if the phase's NEXT evidence already exists on disk (an approved spec for `pending`, the ledger's `verification: converged` line — OR its sanctioned substitute `verification: waived by user <date> — <scope>` — as the ledger's final line for `executing` — trailing `session:`/`uat:`/`minor:`/`disposition:`/`Task N: re-opened` lines are sanctioned (verification.md's below-the-line rule; a re-opened trailer is also the amendment/rejection resume signal), but TASK lines below that final verification line mean re-opened verification that never refreshed — under a converged line it is STALE, route resume; under a waived line the waiver is state-specific, route the disposition ask (a `Task N: re-opened` trailer first dispatches the re-opened work — the ask follows the wave, verification.md's placement) (verification.md); don't flip either way; resolve the ROADMAP ledger link, which may point into a phase worktree; an unreachable ledger is NOT evidence: route resume, and arcane re-derives from `git log` only on the machine that has the phase branch — on a fresh clone without it, the phase's progress is unrecoverable — the UNRECOVERABLE-PROGRESS rule the routes below invoke by name: say so to the user, never silently restart the phase), the previous session died in the flip window — flip (with the stamp) first, then route — UNLESS the ROW-HISTORY TEST fires: the roadmap's own history shows the row previously carried the FURTHER status and a later commit rewrote it backward (row-level — `git log -L` on the row or searching the roadmap's history for the row's earlier value; never file-level, which false-positives on any later commit touching the file): that is a deliberate demotion, a user-side decision — surface it in the batch, never silently re-flip forward (a user decision is never silently discarded; an UNCOMMITTED backward edit in the working tree is the same decision — surfaced, not overwritten; the death shape leaves the row NEVER having carried the further status in any commit — the evidence exists but the flip never landed);

(b) backward — over EVERY row of the routed milestone, not just the routed phase (this is the only checkpoint the milestone-audit and terminal routes pass through — a hand-edited stampless `done` on the LAST phase must not slip a milestone closed): a status whose row lacks its evidence STAMP and whose evidence isn't reachable locally either (`planned` with neither stamp nor plan file, `done` with no acceptance stamp — a hand edit?) → do NOT route on it: surface the gap to the user with the evidence rule quoted; a `done` stamp missing its `branch:` part — or carrying a bare choice word without its artifact (`merged` with no sha, the 0.4.10 shape: recover the artifact from git history with the user aware, per the backfill class): a BARE pre-0.4.10 stamp (no lineage token either) is the migration shape — the backfill ask per references/arbitration.md, never a re-execution; a token-carrying one died in the choice window — verify against git FIRST: base already contains the phase head → the choice DID execute, recover the merge sha from `git log` and append the artifact instead of re-asking (ancestry lies in two known shapes: a squash-merge integration leaves the phase head a NON-ancestor — check the integration history for the squash commit before concluding "never executed"; a shallow clone unshallows first — a false "not integrated" here re-opens an integrated phase and can void its acceptance); otherwise route to arcane's branch-finish question for THAT phase first, never past it ("keep the branch as-is" is itself a recorded choice — `branch: kept`);

if the re-gate on that resume goes red, the fix re-opens convergence and VOIDS the acceptance (it no longer references the final head): demote the row IMMEDIATELY at the red — `verified (converged: reopened <date>; was rounds N..M)` (waived lineage: `verified (waived: reopened <date>; was waived <date>)`), the voiding D-log event riding the SAME commit (the amendment path's one-commit rule — no crash window between void and record), the one sanctioned backward flip, committed — and pushed when a remote exists (Mechanics' never-defer push rule) — before the fix wave starts so no other machine integrates on the voided acceptance meanwhile — the voiding is named to the user and Decision-logged (a recorded acceptance never vanishes silently); re-verification rewrites the stamp with its refreshed lineage token (a waived lineage routes the post-fix disposition ask first — the `verified` route below), and acceptance re-runs before any new `done` with the UAT pass re-offered in FULL (the fix changed the behavior the old `uat:` lines recorded — they are void with the acceptance); a stamp git REFUTES on a full history (a `merged <sha>` that is not an ancestor of the base branch) → surface to the user with both facts, never routed on and never auto-repaired — and on a shallow clone an absent commit proves nothing: unshallow first, refute only on full history — and the ask's answer is RECORDED so it never re-fires (the correction repair or the terminal `branch: unknown (user <date>)` — arbitration.md's migration class); a stampless row whose evidence IS locally reachable → write the missing stamp from that evidence now (a repair, committed), then route — left stampless, the row loses the next stamp-beats-stampless arbitration to any stale stamped counterpart;

a stamped row is evidenced on every machine ONCE its repo-file referent is committed: for `specced`/`planned` rows resolve the referent path in the BASE tree — and a spec whose open/deferred records a spec-gate verdict requires `<spec>.premortem.yaml` beside it (premortem's detectability rule); an absent referent is a half-done flip or a row predating the referent-commit rules, not a death: recover the file from the phase branch/worktree — or from base git history when it was committed and later deleted (unshallow a shallow clone first) — when reachable, and commit it (a repair), otherwise surface it in the batch — NEVER silently re-run crucible or plan-writing over an approved-but-uncommitted artifact; rows whose stamps carry no repo-file referent need no reach — no false alarm;

and a `verified` stamp's LINEAGE TOKEN is a cache of the ledger's verification line: on the machine where the phase ledger is reachable, cross-check it before anything consumes it — on mismatch the ledger wins (the deeper source; the stamp rewrites, a repair, committed with the state), with two guards: a reachable ledger with NO verification line at all is an INCOMPLETE source, never a mismatch (verification lines live only in the ledger and a rebuild from `git log` cannot recreate them — the stamp, possibly the ONLY record of a user's waiver, stands); and a `done` stamp's token is NOT a cache — it RECORDS which verification the acceptance referenced (the schema): a ledger whose last verification line carries a LATER token than a `done` stamp is the overtaken-acceptance signature → the user (arbitration.md's done-vs-verified overtaken-acceptance cell), never a rewrite — rewriting it would fabricate an acceptance of rounds the user never saw; an EARLIER last verification line under a `done` stamp is the incomplete-source shape (a partially recovered ledger — the stamp's referent line is missing, not contradicted): the stamp stands.

An unreachable ledger leaves any stamp standing (the evidenced-on-every-machine doctrine), never re-derives.

Then warm up: if `.catalyst/handoffs/` (or `thoughts/shared/handoffs/` — the ContinuousClaude tree handoff writes to when the project has one) holds a handoff carrying this campaign's pointer, read the latest one — the roadmap decides WHAT happens next, the handoff's mental model saves re-discovering HOW things work.

## Creating a campaign

1. Intent and milestones — through catalyst:crucible (the intent is a decision, not a transcription); starchart's finished map converts naturally AND completely: Destination → Intent, closed decisions → Decision log (each D-line carrying its Resolution's ORIGINAL date), ## Out of scope → Non-goals (the scope fence survives the conversion — nothing graduates back silently), ## Notes' standing agreements → Decision log (each with its own date; an UNDATED legacy Notes line converts dated by the CONVERSION and annotated as such — a historical date is never invented), remaining work clusters → phases; the creation D-line names the map path (the pointer starchart promises).
2. Slice phases tracer-first (each phase leaves the system demonstrably better/shippable), 3-9 phases per milestone (a legitimately smaller MILESTONE — 2 phases, a small program or a trailing milestone — is fine: the floor guards against skipped decomposition, not small scope; ONE phase is the single-feature red flag); a phase should fit one arcane-mode plan (a few tasks) — a bigger chunk splits into more phases; a milestone is a set of phases, never one oversized phase.
3. Write both files — PROGRAM.md's `base:` line records the branch the approval commits on (`git branch --show-current` at creation; the schema's field every later ON-base check reads) — get the user's approval of the roadmap as ONE question, set every milestone `status: open` and every phase `pending` — and Decision-log the approval as the creation line naming the approved phase ids — ROADMAP.md, PROGRAM.md, and the creation line land in ONE genesis commit (paired writes land together: a torn genesis leaves every row a hand-added suspect on first resume) (the provenance every later scope check reads).

## Milestone audit — against intent, not the checklist

When a milestone's phases are all `done` — each row carrying its full `done` stamp (a stampless, `branch:`-less, or bare-choice-word row is a backward-reconciliation case, not an audit input), and the audit spot-checks the `branch:` claims against git first: a `merged <sha>` whose commit is not an ancestor of the base branch is a claim git refutes — a reconciliation case, never an audit input: goal-backward audit of the MILESTONE against `PROGRAM.md ## Intent` and the milestone's must_haves — fresh-eyes auditor (top tier, clean context — the milestone-intent tiering's normative home), the same discipline as arcane verification but one level up — and the audit FIRST sweeps the milestone's rows' spec markers AND premortem yamls with the terminal sweep's own arms: a pending, double-answered, or vanished-section (lost-referent) entry, or a `withdrawn:` acceptance with no landed re-acceptance (the shipped-risk divergence — surface it and offer the amendment/fix-phase route, the same arm as a rejection answered on a `done` row) — crucible's shapes — is a standing user ask that bars closure (batched), exactly like an unresolved shape; a spec unreachable on this machine is a LOST REFERENT, never a silent pass — recover it from git history first (unshallow a shallow clone first), genuinely absent → the sweep bars and surfaces it; a REJECTION answered on a `done` row is a spec-code divergence — the code implements the rejected edit: surfaced with the amendment route offered, closure never rides over it silently: phases all done ≠ intent achieved (integration seams between phases are exactly what per-phase verification never saw). Findings → fix phases into the roadmap (user approves), not silent patches. Convergence uses the same stop criterion — iterative rounds until 2 consecutive clean; then the milestone closes in PROGRAM.md as `closed (audited: <date>, rounds N..M)` PLUS its D-log event line — the closing record other machines arbitrate on; a `closed` written without its record is exactly the hand-edit shape the arbitration refuses to trust; next milestone or campaign end — the user's call. The user's EXPLICIT order to close WITHOUT the audit bends this gate like any named gate (arcane's Iron rules): cost stated in one line, then write `status: closed (waived: <date>)` — the waiver's recorded home — plus the D-log event; a recorded waiver is never re-demanded (a bare `closed` is never freshly written as a closure — it survives only as the migration/hand-edit shape the backfill ask exists for, or standing under a declined-backfill D-line, the schema's terminal record).

## Session end

Campaign sessions end through catalyst:handoff; the handoff carries the campaign path so the next session's router starts warm. Before the handoff, VERIFY the roadmap is already current — statuses flipped when their evidence appeared, each flip carrying its evidence stamp (a stamp is exactly what another machine routes on — a bare flip found here is half-done), links laid; a missed flip found here is fixed now, but session end is the safety net, not the flip point. The roadmap, not the handoff, is the program's memory.

## Red Flags — STOP

- Navigating the program from memory or the conversation instead of reading ROADMAP.md.
- Starting a new phase while another is `executing` ("it's stuck anyway") — resume or explicitly park it with the user first.
- A status advanced without its evidence (specced with no spec file, verified with neither a converged verification nor its recorded waiver line).
- Roadmap scope edited silently — add/remove/reorder is a user decision with a Decision-log line.
- Milestone declared done from the phase checklist without the intent audit.
- Program details accumulating in ROADMAP.md instead of the linked artifacts — the index rots into a monolith.
- A campaign created for a single-feature effort — that's crucible → arcane-mode, the umbrella adds only weight.
- Campaign state uncommitted at session end, or committed onto a phase work branch instead of the base branch. (The one sanctioned deferral: a single-checkout flip edit awaiting the next base checkout per Mechanics — committed at branch finish at the latest; the never-defer CLASS — the `reopened` demotion, a post-flip spec-gate yaml + its spec edits, an amendment wave's edits + pending marker, each recorded pending-entry answer, each unresolved-shape answer, the hand-demotion suspect's answering D-line, and an acceptance withdrawal — is NOT in it: those commit to base now — and push now when a remote exists, per Mechanics.)
- Routed with a remote configured but without the step-0 pull; or (when a remote exists) the session ended with unpushed state commits / an unpushed phase branch.
- A status flipped without writing its evidence stamp onto the row.
