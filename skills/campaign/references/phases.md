# Campaign phases (normative)

Verbatim campaign doctrine (0.7.0 distillation; split from mechanics.md by read-trigger in 0.7.x). Where SKILL.md's sketch differs in detail, THIS file governs. Internal references like "Rules", "Mechanics above", "(below)" refer to the pre-split SKILL.md layout; their targets live across the `references/` files — `mechanics.md` is the router. Merge cells and legacy shapes: `arbitration.md`.

## 4. Worktrees and who flips a status

Run phases in WORKTREES off the base (the default — the base checkout stays available), and **the session that produces the evidence flips the status in the base checkout and commits there immediately**, not at handoff time (the Session-end check is a safety net, not the flip point). Single-checkout branch workflow: edit the flip immediately, commit it at the next moment base is checked out — branch finish at the latest, never onto the phase branch. The NEVER-DEFER CLASS — commits that exist to be seen cross-machine — never defers, each via a momentary base checkout or a base worktree (the phase branch keeps the working state): (1) the `reopened` demotion — visible DURING the fix wave (branch finish, the usual deferral backstop, is exactly what the red suspended): commit it to base NOW; (2) a spec-gate premortem run after the `specced` flip — its yaml and spec edits (premortem's transport rule); (3) an AMENDMENT wave's spec edits + pending marker (arcane's route — produced by crucible's fast path, same transport: base must show the pending shape, or every other machine consumes the stale approved spec); (4) each recorded ANSWER to a pending re-approval entry (approve, reject, or arbitrate — premortem's one-commit-per-answer); (5) each recorded ANSWER to a standing unresolved shape (row, milestone status, or scope) — the rewrite and its D-log event in ONE commit (separated, the answered row reads as independent advancement and the ask re-fires); (6) the hand-demotion suspect ask's answering D-line (the router's accounting answer — the record that stops the ask re-firing on other machines); (7) an acceptance WITHDRAWAL — the `withdrawn:` append plus its recompute in one commit (premortem's rule: no machine may dispatch on a reversed decision). The never-defer class OUTRANKS parallel-sanction flip ownership: a non-owner session with a class member (a demotion at a red, an answer) commits and pushes it NOW — ownership governs ordinary evidence flips only, never this class (the stale-write guard above covers the shared-file race). Every never-defer commit also PUSHES now when a remote exists — a base commit is invisible to every other machine until pushed, and this class exists precisely to be seen before the session ends (the push-failure routing above applies: a named failure, never silent; the pushed BASE commit is what lets the bridge read the yaml from the base tree on another machine before any branch finish — recovery from a pushed phase branch stays the fallback, premortem's lost-referent arm).

## 5. Parallel phases (user-sanctioned)

Parallel phases (user-sanctioned): file-disjointness is a HARD precondition checked when writing the sanction D-line (overlapping `Files:` across the sanctioned phases' plans → surface it, no sanction; a later plan edit that introduces overlap voids the sanction — surface, never silently run on); the sanction is DECISION-LOGGED (a D-line naming the sanctioned phases and each phase's owner as a user-legible label — a machine, a worktree, a person) — the durable trace every machine reads; without it a two-`executing` roadmap is indistinguishable from the violation. ONE session, named by the user, owns all roadmap flips — the others leave the evidence on disk, name it in their handoffs, AND (when the owner is another MACHINE — ledgers are self-ignored and handoffs machine-local, so on-disk evidence never crosses machines) record a committed evidence-receipt D-line on base (`evidence: P<N> <payload>, head: <sha> (<machine label>, <date>)`, never-defer): the owner flips from the receipt, which is the durable provenance every machine reads; without it a cross-machine owner has no lawful flip source. The receipt covers EVERY owned flip and append, not only verification, and it always names the phase branch head it attests — a receipt is the writer's OWN attestation, so it is tied to concrete fetchable state, never a bare claim; the per-flip payload grammar, the head-OID rule, the receipt's event identity, and the contradicted-receipt arm live in `references/arbitration.md` (the receipt norm home) — an owned transition with no receipt form would strand cross-machine exactly like the verification flip once did.

## 6. Evidence flips and stamp discipline

ROADMAP is an index — details live in the linked specs/ledgers. **Status changes only with evidence, and the flip writes the evidence stamp onto the row** (stamp formats in the schema above) — and when the stamp's referent is a repo file (the spec and its spec-gate premortem yaml for `specced`, the plan and its premortem yaml for `planned`), the flip's state commit CARRIES that file into the base branch — with the transport NAMED: the file is usually produced on the phase branch/worktree, so bring it into the base checkout explicitly (`git checkout <phase-branch> -- <path>`, or copy from the worktree) and commit it together with the flip; a stamp whose repo-file referent is uncommitted on base is half-done — "evidenced on every machine" holds only because the evidence travels with the commit. A `discarded` branch-finish choice never deletes the branch before the row's referents are committed on base — the branch may hold their only copies: an APPROVED spec (crucible's `status: approved` marker at the TOP of the file — body-embedded status shapes are content, never the marker, including its skip-order shape `status: approved (waived: <date>)` and the pending shape `status: approved (edits pending re-approval: <date>)` (keyed on the clause — the waived combo carrying it is pending too) — approved for evidence, but arcane's input rule owes its fast re-approval before the pending sections are consumed; a markerless file is a draft, not evidence) → specced; premortem PASS/WARN — BLOCK bars it — plus the written plan → planned; ledger shows tasks closing → executing; the ledger's `verification: converged` line → verified — OR its sanctioned substitute `verification: waived by user <date> — <scope>` (economics' review-floor drop, recorded; the stamp is `verified (waived: <date>)`, an honest mark every reader sees: the rounds did NOT run, the user chose that, and no session re-demands them); user accepted → done (no other acceptance artifact exists — an unstamped `done` is unevidenced by definition).

## 7. The one batched question

All escalations ONE router run discovers (arbitration cells, reconciliation gaps, the migration backfill ask) are presented to the user as ONE batched question — arbitration cells first, then reconciliation gaps, then routing — never an interrogation drip, and never silently self-resolving the "small" arms to spare the user. A LARGE batch is still one message, walked as a numbered list the user answers item-by-item (the batch bounds the interruptions, not the answers' depth — crucible's one-question doctrine governs each item); a PARTIAL answer records the answered members' outcomes at the moment they are given (never re-asked — the same at-the-moment rule as the disposition trailer), while unanswered members keep their routes blocked and re-surface in the NEXT run's batch: a partial answer never unblocks the unanswered arms and never re-litigates the answered ones. The batch carries escalations answerable at DISCOVERY time; a question whose premise is established by post-routing work (the branch-finish integrate question behind its re-gate, a premortem-BLOCK acceptance decision) is NAMED in the batch ("after routing, X will be asked") but asked in-route by its own protocol — the batch rule governs discovery-time asks, it never front-runs a gate.
