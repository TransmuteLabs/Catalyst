# Campaign state arbitration & legacy shapes (normative)

This file is the normative home of every state-file merge cell, every durable unresolved shape, and every legacy/migration shape. SKILL.md's Rules OWN the trigger list — merge conflicts in state files, standing `unresolved` / `unresolved-scope` / `base: unresolved` shapes, legacy shapes, a `base:` answer missing its transport receipt, the user's ORDER to change the base (and the resolved carve: a bare `closed` standing under its declined-backfill D-line owes no read) — **→ read this file ENTIRELY before resolving or routing past anything.** The body's list is authoritative; this sentence is a summary, never a second home. Every "→ the user" outcome here is a MEMBER of the router run's one batched question (SKILL.md Rules); dates, stamps, and lineage tokens are defined in SKILL.md's schema.

## Per-row merge cells (ROADMAP.md)

Merge-conflict arbitration for ROADMAP.md/PROGRAM.md: per phase row, FIRST: a side whose status is outside the enum and is not a sanctioned stamped shape is UNORDERABLE — stamp rules never rank it (in either direction); the pair durabilizes → the user (the ask names mapping the alien status to a family one). Then: a status WITH its evidence stamp beats one without; both stamped → the further status wins (evidence is cumulative) — but EVERY row-arbitration cell consults the Decision log FIRST: an answering D-line covering this row's divergence (or membership) is the standing record — the chosen side, as possibly further advanced, wins, and a removed row stays removed, REGARDLESS of whether an unresolved shape is party to the merge (further-wins and stamped-beats-stampless run only when no answering D-line covers the divergence — a recorded user answer is never mechanically overridden by later unilateral advance of the non-chosen side, which is surfaced instead) — and EXCEPT a stampless side that is a DELIBERATE demotion by the row-history test (the reconciliation guard's test: the row previously carried the further status and a later commit rewrote it backward): that is a user decision → the user, never stamped-beats-stampless (EXCEPT a backward rewrite whose commit carries the amendment's D-log event AND the stamped `reopened` shape — the sanctioned mechanical demotion, routed by the reopened arm, never the hand-demotion shape; the D-line WITHOUT the stamped shape is a malformed write — surface it to the user, route neither arm). The rounds-keyed EXCEPTs below compare LINEAGE TOKENS (schema): a waived lineage compares its waiver dates the same way, and a `done` vs a `verified` from DIFFERENT lineages (one carries rounds, the other waived) → the user — the two sides' verification histories diverged. EXCEPT `done` vs a `verified` whose token is LATER than the token in the `done` stamp: that acceptance was overtaken by a re-opened verification (acceptance must reference the final head — arcane's verification rule), and "cumulative" is false → the user (a `verified` whose token is EARLIER is the stale side — the `done` wins mechanically, genuinely cumulative; unorderable tokens → the user); and EXCEPT `done` vs a `reopened` row whose "was" token EQUALS the `done` stamp's token — equality here is the voiding signature, not cumulative evidence (that `done` IS the acceptance the demotion voided; this is what the "was" token exists for): the `reopened` side wins either way, and the presentation follows the schema's pedigree rule — the pair's dates consistent: assert the voiding flat; inconsistent or missing: PRESUMED voiding, the coincidental-restart possibility named — and the `done` side's later `branch:` append (if any) is surfaced to the user as an integration executed on a (possibly presumed) voided acceptance; a `done` vs a `reopened` whose "was" token DIFFERS from the `done`'s token is NOT that cell — compare DIRECTION (tokens are monotonic within a phase, the schema invariant): the `done` token LATER than the "was" token = a post-re-verification acceptance, cumulative — the `done` side wins mechanically; the `done` token EARLIER = an acceptance even older than the one this demotion voided, superseded generations ago — the `reopened` side wins; tokens of different kinds or otherwise unorderable → the user, the histories diverged (the first EXCEPT reads a `verified` row's live token, never a `reopened` row's "was" token); two `done` rows where one side merely EXTENDS the other's stamp with its `branch:` part → the superset wins mechanically (the append is cumulative evidence, no escalation); a row parked on one side and advanced in ## Phases on the other → the user ("further wins" never auto-un-parks: parking is a user decision, and a user decision is never silently discarded); a row parked on BOTH sides with differing preserved status/stamp values arbitrates the preserved pair by these same per-row rules — the winner stays PARKED (arbitration never un-parks); differing park reasons/dates → the user; conflicting `done` rows compare DIRECTION like the reopened cells: orderable tokens → the later-token side wins mechanically (a post-re-verification acceptance supersedes a stale one) — and the LOSING side's `branch:` append (if any) is surfaced to the user as an integration executed on a superseded acceptance, never dropped silently; unorderable → the user; neither stamped, or two different values for the same stamp → the user; scope differences (added/removed phases) reconcile through the Decision log with the user — never blind ours/theirs.

## PROGRAM.md section cells

PROGRAM.md's own sections arbitrate too, each by its nature: the `base:` line — the user's explicit ORDER to change it is the one sanctioned rewrite outside creation/backfill: the new value plus its dated D-log event (a base-change decision) in ONE commit, and the transport-receipt rule applies — the campaign state is CARRIED onto the new base (the merge) before anything routes there; two differing values (two machines' backfill repairs, or a hand edit) → the user, never mechanical (a backfilled value is only a repair, and every state transport depends on this line — the conflict leads the batch), durabilized before the merged file commits as `base: unresolved (between: <X> ||| <Y>; merged <date>)`: the branch where the merge ran is the shape's INTERIM home — while the shape stands, routing and evidence flips are suspended and no state commits land on any GUESSED base, with exactly two sanctioned write sites, both on the interim home: the durabilizing merge commit itself and recorded-ANSWER commits (the never-defer classes and batch recordings — a given user answer must not die uncommitted; pushed as usual); before ASKING, FETCH — state-safe, always sanctioned — and inspect the candidates' PROGRAM.md on the remote for an answering rewrite or D-line: a delivered answer is ADOPTED, never re-asked (the durabilizing machine must not re-ask a question whose answer another machine already recorded) — but adoption VERIFIES the answer's sanctioned shape first: the rewrite and its dated D-line land in ONE commit; a bare rewrite with no D-line, or an orphan D-line whose paired rewrite is absent, is the SUSPECT shape (a defective writer, the hand-demotion class), never an answer — batched to the user, never adopted; the user's answer rewrites the line with its D-log event in ONE commit on the interim home, and when the chosen base is a DIFFERENT branch the SAME session then CARRIES the campaign state onto it — a MERGE of the interim home into the chosen base, committed there (the named transport step: the merge carries the campaign dir AND every flip-carried referent the durabilizing merge arbitrated — specs, plans, premortem yamls, entry answers — which live OUTSIDE `.catalyst/campaign/`; a single-directory `git checkout <interim> -- .catalyst/campaign/<name>` is NOT the transport — it strands every referent outside that dir, and their arbitration (kept double-answer appends, pending markers, reverted bodies), on the interim home forever) — and the transport is OWED until the ANSWER COMMIT ITSELF is an ancestor of the chosen base's head — the merge brings it into ancestry; a copy-commit that merely reproduces the D-line's text (the banned single-dir checkout) leaves the answer commit outside the base's ancestry and satisfies nothing — the receipt is checked by ANCESTRY, never by bare text match (the paired-effects principle) — with the family's two ancestry caveats: a base that only integrates by SQUASH legitimately destroys ancestry — there the carry itself lands the receipt as CONTENT the carrier controls — a CARRY-RECEIPT D-line appended to PROGRAM.md's Decision log in the carry (`carry receipt: base answer <date> carried from <interim ref>; referents: <path list, or none>` — a forge-squash rewrites commit MESSAGES, so the receipt lives in the FILE, which the squash preserves; `referents:` is the carry's own manifest of the flip-carried referents the durabilizing merge arbitrated (`none` when it arbitrated nothing outside the campaign dir — the honest minimal carry is then campaign-dir-only BY DESIGN, not stranding)) — and a carry-receipt D-line in the base's PROGRAM.md TERMINATES the check ONLY after content verification. LOCATE protocol (each step is load-bearing — the naive invocation fails on honest histories): search the ID-LESS receipt content (`git log -m -S'carry receipt: base answer …'` — never the full line with its `D-NN:` prefix, which a later renumber rewrites, making `-S` return the renumber commit; `-m` is required or a receipt minted inside a merge commit is invisible to the pickaxe); multiple hits → verify the EARLIEST (a byte-matching re-add is the recovery classification, exempt); a renumber/rewrite commit is never the minting commit; a shallow clone's truncated history verifies NOTHING — the graft boundary commit shows a whole-tree diff that would falsely pass anything — unshallow first, the same arm as the ancestry check. VERIFY: every path in the receipt's `referents:` manifest must be byte-identical at the base head to its interim source (blob compare — byte-identical referents legitimately produce NO diff, so diff-span is not the test); `referents: none` self-verifies with a campaign-dir-only diff. A receipt with no locatable minting commit, or whose named referents are absent/differ at base, is the SUSPECT shape (the stranding carry delivers its own receipt — bare presence proves nothing): the owed carry re-runs for the stranded referents and the suspect line joins the batch; the user's answer lands as a dated D-line — `carry attested: <date> (user)` terminates the ask, or the re-carry's own new receipt does. A legacy receipt without a `referents:` manifest verifies by the 0.5.5 diff-span rule under this LOCATE protocol; undecidable → suspect. A verified receipt is never re-run; and a shallow clone concludes non-ancestor only after unshallowing (absence in truncated history proves nothing): the answer D-line doubles as the TRANSPORT RECEIPT — any session reading a base:-answer D-line (locally, or on any fetched candidate branch) whose answer commit is NOT an ancestor of the chosen base's head performs the owed carry FIRST (the merge, committed and pushed) before routing anything on that base — a death between the answer commit and the carry leaves the receipt visible on the interim home, so the answering machine's next session, and every ADOPTER (fetch-and-adopt adopts the answer AND its owed carry, never the answer alone), inherits the duty; a bare chosen base with no receipt in its history is un-carried, never self-consistent — the interim copy stays as history and later merges arbitrate by the answering D-line; the Decision log is an append-only EVENT LOG — a merge UNIONS both sides' lines by CONTENT, never drops either (both machines' events happened). Precedence over byte-identical pairs (three rules formally claim them — this order governs): the union treats a byte-identical pair as ONE event and dedups it, EXCEPT kinds whose honest repetition is semantically possible (park/un-park cycles, repeated gate events, anything a phase can legitimately do twice in a day) — those take the same-day twin repair below and survive as two timed records; the same-event collapse governs DIFFERENT-WORDING pairs only; D-ids are per-file ordinals, never cross-machine references — consumers match D-lines by content (kind + phase ids + date; every D-line carries its date), so after the union the log renumbers sequentially by date — same-date ties break by kind (THIS list, in THIS order, is the single normative home of the kind enum — SKILL.md's schema points here; position in this list IS the tie-break order: creation, scope, parking/un-parking, parallel sanction, voided acceptance, amendment, declined backfill, milestone closing, re-open, divergence answer, milestone-status answer, scope answer, base change, unattested-demotion restore, carry receipt, and LAST "decision" (plain decisions hold the final position — the order is total)), then referent ids, then text, lexicographically (a stated total order: the same final log from any merge order) — and the renumber CLOSES with a completeness check against the POST-COLLAPSE union (the same-event collapse is part of the union step: collapses first, then the renumber, then the check): the same line count and the same multiset of FULL LINE TEXTS minus the ids (a lost or mutated line — including a mutated why-text — is a broken repair: redo it; at volume the renumber rewrites ids only, never retypes lines); all dates in stamps, records, and D-lines are ISO — YYYY-MM-DD, plus HH:MM where a same-day arm applies — written in UTC (dates are arbitration keys compared ACROSS machines; local clocks in different zones invert "later-dated supersedes" — legacy local-time records arbitrate as-is, and the zone-doubt arm is BOUNDED: it applies only when the compared dates lie within one day of each other (the maximum zone skew — farther apart, mechanical comparison stands); within that band on a pre-0.5.3 campaign, doubt → the user); and the same-day arm is FAMILY-WIDE: ANY record whose identity or arbitration keys on a date appends the time whenever a same-day sibling of the same kind exists (stamps, milestone records, arbitrated appends, waiver tokens, withdrawal/re-acceptance records, re-open annotations and ledger lines, pending-entry headers, rename notes) — the writer of the SECOND same-day record owes the time on its own record — the scan scope is the file being written; records minted the same day on machines that had NOT fetched each other collide only at the MERGE, where neither writer broke the rule: the MERGE WRITER repairs identity mechanically — SCOPE: byte-identical same-day twins ONLY (an identity collision; two same-day records that DIFFER in content are not twins — they take their cell's own arbitration, and with no strict order they go → the user, never time-ordered by the writer) — it appends each twin's minting-commit COMMITTER UTC time to that record, an IDENTITY disambiguator and never an arbitration key (a record's arbitration date stays the date of its EVENT — commit clocks are writing order, skewed by offline machines, and must never pick between two recorded user decisions); twins STILL byte-identical after the repair (same committer minute, or both rode one rewrite/carry commit whose located time they share) → the user like same-date records (git knows both commits; a repair, never an ask), so byte-identical same-day twins never survive a union (normative here because dates are sorting and arbitration keys; in-file records use `HH:MM` — artifact NAMES use the hyphenated `_HH-MM`, a deliberately different shape); the hand-demotion suspect ask's RESOLUTION protocol (the router detects; this cell resolves): present BOTH facts — the row's demoted status and the row history's last evidenced status — with two named outcomes: the user attests the demotion → their answer WRITES the missing accounting D-line, a dated voiding or decision event; the user can't attest ('not mine / don't know') → the row RESTORES to its last evidenced status per the row-history test and a dated D-line records the unattested demotion and the restore (evidence-backed status beats an unaccounted rewrite nobody claims) — either way committed and pushed as the never-defer class, so the ask never re-fires; conflicting recorded answers to the SAME hand-demotion suspect ask — one side's attest (voiding/decision D-line + the standing `reopened` row) against the other's can't-attest restore (restore D-line + the restored stamp) — are two recorded user decisions: → the user, never mechanically arbitrated, and the equal-token voiding-signature cell NEVER fires over a row whose restore D-line answers THAT unaccounted demotion (a recorded restore answer is a user decision, not a stale forward flip) — but the bar is scoped to the demotion the restore answered: a LATER-dated SANCTIONED voiding record (a re-gate red's voiding event, an amendment demotion) fires the cell normally — records arbitrating records, the later-dated voiding beats the earlier restore; two lines recording the SAME event (same kind and referents, different wording — and, for answer-kind events, the SAME CHOSEN SIDE: two answering D-lines for the same divergence choosing DIFFERENT sides never collapse — they are two conflicting recorded user decisions: a strictly later-dated one supersedes (records arbitrating records, the milestone discipline); with no strict order the pair RE-DURABILIZES between the two chosen sides (SCOPE-answer conflicts — keep vs remove — re-durabilize as the row's `unresolved-scope` shape, their native form, never the between shape), noted as conflicting answers → the user) collapse to one, keeping the fuller text (the LONGER; equal lengths tie-break lexicographically — a total order, the same final text on every machine); the Milestones `status:` line arbitrates by the RECORDS' DATES, never by shape precedence — every transition writes a dated record (a closure: the audited/waived shape plus its D-log event; a RE-OPEN: the un-park rule's D-log line — a record too; a record's arbitration date is the date of its EVENT, never of its writing — a backfill carries the historical date): the later-dated record wins (a re-open dated after a closure re-opens it everywhere — a user's recorded re-open is never mechanically overridden by a stale closure; a later re-closure closes it again), and this covers closed-vs-closed of any shapes the same way (a post-re-open waiver beats a stale audit; a later audit beats an older waiver); a same-day second transition appends the time, like the extend-waiver token — the same-date escalation is for genuinely simultaneous records, not every fast day; recordless sides (a bare `closed` — a hand edit or the migration shape; NOT one standing under a declined-backfill D-line — that milestone is closed-with-record for ROUTING, but the decline is a NON-ANSWER record: it never competes with a genuine closing record by date — any closing record, dated or undatable-attested, supersedes it by KIND (the milestone-backfill cell below); the decline's date serves only the once-only ask and decline-vs-decline collapse), same-date records, or any doubt → the user; the top `status: active|complete` line is DERIVED — never arbitrated, recomputed from the merged Milestones section (the writer that flips it writes the date of ITS flip).

## Spec, plan & premortem-yaml merge cells

Spec files and premortem yamls travel in the same state commits and get their own merge cells — never ours/theirs: a MARKER-line conflict (plain vs pending) resolves from the merged open/deferred ENTRIES — any `pending re-approval` entry with no answer appended keeps the marker PENDING (a restore only ever follows recorded per-entry answers, crucible's shapes (its `references/marker-machine.md`), so a pending-clause-less marker — plain or waived — over an unanswered entry is the bad-merge shape every consumer already refuses); premortem-yaml conflicts recompute the verdict from the merged yaml's own records (`accepted_risks`/`mitigated` and rejection reverts are append-only unions); two sides appending DIFFERENT answers to the SAME pending entry (re-approved on one, rejected+reverted on the other) → the user — two recorded user decisions conflict, never auto-picked; the merged entry keeps BOTH appends (the double-answer shape IS the durable signature — crucible's cross-check treats it as pending on every machine, so a death between the merge commit and the ask loses nothing); until the answer, the spec BODY holds approved-plus-surviving-pending text — the stacked revert target, never a wholesale rollback that would silently destroy OTHER still-pending entries' edits in the section; a surviving stacked edit that does not COMPOSE with the removal takes the entangled shape instead (disposition owed, body unreverted — EXACTLY the entangled-revert carve: its body holds the pre-revert text until its disposition, crucible's pending shape) and the yaml recompute waits; a terminal `— rejected <date>: reverted` append meeting an entangled `— rejected <date>: revert entangled with …; disposition owed` append on the SAME entry is ONE user decision (reject) with two EXECUTION states, never the double-answer shape — the decision is never re-asked: the ENTANGLED side governs the merged state (the merge union contains the surviving entry's edit, so the composition problem is real in the merged spec — body pre-revert, sections pending, disposition still owed; the terminal side's executed revert was its machine's local state without the surviving edit in view, superseded by the union), both appends are kept, and the standing disposition ask proceeds — UNLESS the entangled side's closing `— reverted` append already landed: an answered disposition is terminal, both sides agree closed, the union holds the closed entry and nothing re-fires; spec-body conflicts OUTSIDE recorded answers have no mechanical cells — ordinary git conflict work, surfaced to the user AND recorded: the resolving merge lands a dated spec open/deferred note naming the resolved sections (`;`-separated names — crucible's list grammar) (the resolution record — crucible's hand-edit test reads it: a merge hunk matching neither parent with no record NAMING its section is the smuggle shape — the record binds by section name, an unrelated note exempts nothing). PLAN files are state too (they carry recorded user decisions): task annotations (`[re-opened by amendment <date>]` / `[re-opened by rejection <date>]`) are append-only records — a merge UNIONS them, never drops one side's annotation (a dropped annotation silently closes owed rework); conflicting hunks in a task's BODY text are ordinary content conflicts → the user, never ours/theirs and never auto-arbitrated (the plan's wording is approved ground under the amendment route).

## Durable unresolved shapes and their merge cells

Unanswered ARBITRATION cells (whose only native signature is the transient merge conflict) are made DURABLE before the merged file commits — the row is written as `unresolved (between: <full row X> ||| <full row Y>; merged <ISO date>)` with the sides in LEXICOGRAPHIC order, ` ||| `-separated (machine-independent — two machines durabilizing the same divergence write the same pair; a legacy single-pipe pair from a 0.4.24-era session is unparseable into sides — NORMALIZE it to the triple separator on first read: the split point is the ` | ` whose two halves BOTH parse as well-formed roadmap rows (well-formed = the schema's five cells `id | phase | milestone | status | links`, the status in the enum or a sanctioned stamped shape, the links cell legitimately EMPTY — the row shape is the oracle) — exactly one qualifying split → rewrite it, a repair, never a new divergence; zero or several qualifying splits → the user with every candidate parse shown, never a guess), a sanctioned shape the self-consistency check treats as a standing batch member (routing blocked: the stall binds THE ROW and its milestone's closure — the router routes the milestone's next not-done phase per SKILL.md's routing rule (dependency exception there), EXCEPT when ANY side of the pair — or the was-row of an `unresolved-scope` — carries `executing`: SKILL.md's campaign-wide executing scan counts such a row as an executing row in EVERY milestone (a live phase may be running on another machine) — it holds the whole campaign's new starts on the ask, never relieved and never resumable; an unresolved row also bars the all-done milestone-audit gate; every other row routes normally); the user's later answer rewrites the row to the chosen side, recorded at the moment given — the conflict is never consumed by commit mechanics into a silent ours/theirs. The unresolved shape is N-ARY — sides sorted lexicographically, separated by ` ||| ` (a roadmap row's own pipes never collide with the triple) — and has its own merge cells: unresolved rows over the same divergence merge by UNION OF SIDES (flattening — an unresolved row is never itself a side: same sides collapse, the earlier `merged` date kept; two VARIANTS of one side — the same side advanced in place on one machine — collapse by the per-row arbitration cells (the same rules as any row pair: an answering D-line first, then further-wins/stamped-beats-stampless and the token-direction cells — the mechanical winner's text becomes the side); a variant pair those cells route to the user is itself a divergence: BOTH variants join the ask as separate sides — never resolved by text length, which would discard a recorded stamp value — EXCEPT a pair escalated by a NON-choice surfacing cell (the malformed-write cell's route-neither-arm): that keeps the escalating cell's own framing, surfaced, never pickable); the user's ANSWER replaces the pair with the chosen side — and when a side's status is OUTSIDE the enum (the durabilization sanctioned it as a side; a bare row it may not be), the ask NAMES this and the answer maps it to a family status (or parks the row) before the rewrite lands: a non-enum status never lands as a bare row status, which would trip the self-consistency abort on every later run AND writes a dated D-log event naming the divergence, QUOTING each side verbatim, and marking the chosen one (the milestone cell's verbatim discipline — this record is what the coverage and match/extend tests below, and the non-chosen-side surfacing rule in the per-row cells above read; a loose paraphrase leaves them undecidable) — rewrite and D-line in ONE commit, never-defer transport (SKILL.md Mechanics class 5) — the record the next merge arbitrates by; an AMENDMENT ordered against a phase whose row stands unresolved consults the ORDER's text: an order that NAMES a side (quotes or otherwise discriminates one side's status/stamp) is that ask's answer — record it first (the same rewrite-plus-D-line commit), then arcane's demotion runs on the chosen row; an order referencing only the phase or its spec discriminates NEITHER side — the standing ask fires FIRST (present the pair, take the user's pick) and the amendment proceeds on the answered row: never mint an answer from a non-discriminating order, never demote an unanswered pair; a NORMAL row meeting the pair is the recorded answer ONLY on that D-line's evidence (it matches — or extends with later cumulative evidence: a `branch:` append, a later same-lineage token, or a further status carrying the same token — the CHOSEN side: it wins, never re-asked — EXCEPT against a LATER-dated sanctioned voiding record of the chosen side: a re-gate red or amendment demotion postdating the answer voids the chosen lineage — the demoted shape stands and its own route governs, the answer stays history; a voiding record and the answer with NO strict date order are unorderable → the user, never auto-picked either way; a divergence between the chosen side and its own demoted shape is NOT covered by the answer D-line); ABSENT the D-line, a normal row matching or extending one side is independent advancement by a machine that never saw the conflict, never an answer: it UPDATES that side in place (the side text is replaced by the advanced row — sides stay current, the divergence and the ask stand); a row REWRITE whose was-clause carries a side's own lineage token is that side DEMOTED, never a new side: it REPLACES that side (the voiding travels — the stale pre-demotion text is not pickable, the ask presents the demoted shape); a normal row matching and extending no side JOINS as a new side (the durable form of the new divergence — the user picks among all sides). The same durability governs NON-row →user cells: an unanswered Milestones `status:` cell persists as `status: unresolved (between: <side X> ||| <side Y>; merged <date>)` — the derived top status treats an unresolved milestone as NOT closed (a campaign never completes over it), the milestone-audit route is BARRED on it — an unresolved `status:` is blocked-on-user, never "not closed, all phases done → audit": a fresh audit's later-dated closing record would mechanically supersede the disputed sides and moot the pending arbitration — the same side-union merge cells apply, and the user's answer keeps the chosen shape's OWN historical date while writing the answering D-log event dated by the answer and NAMING the chosen shape verbatim (the event-plus-named-shape pair is the arbitration record — a historical date is never rewritten; one commit, SKILL.md Mechanics class 5) so the next merge arbitrates records, never re-asks; unanswered scope differences persist the disputed ROW as the WHOLE-LINE shape `unresolved-scope (was: <full row>; merged <date>)` — like the between shape, the line replaces the row entirely, never embedded inside a status cell (a full row inside one cell breaks the table's own grammar and truncates every columnar parse; the `was:` row keeps its own cells verbatim) — a sibling shape with the between-row's full consumer set (enum carve-out, standing batch member re-surfacing until the Decision-logged answer, its row's routing stalled on it (row+closure scope, same relief and same `executing`-side exception as the between shape), the milestone-audit/all-done gate barred while one exists) and its own merge cells: two `unresolved-scope` markings of the same row collapse (the earlier `merged` date); a machine advancing the row against the other's scope marking UPDATES the `was:` row text in place — never consumes the ask; the answer is the Decision-logged user decision (one commit with the row outcome, SKILL.md Mechanics class 5): keep → the row returns as its current `was:` text; remove → the row deletes and the D-line is the durable record — a later merge meeting a still-standing `unresolved-scope` for the same row consults the D-log FIRST: an answering D-line consumes it, never re-durabilized, and when the consumed marking's `was:` text had been UPDATED past the answered text (advanced work), the discarded advance is SURFACED to the user in the same message — every merge order shows the same visibility — arbitrated as records thereafter. Family assignment is exclusive: row-VALUE conflicts durabilize as the between shape in the row's AGREED section — parked on BOTH sides → the pair stays in ## Parked (arbitration never un-parks; a parked pair keeps Parked's exemptions — it stalls no routing and bars no MILESTONE gate, the ask re-surfaces via the standing batch, and the campaign's terminal route waits on it — the completion guard; an un-park request against a parked PAIR first ANSWERS the ask — the user naming a side to un-park IS the answer, class 5 — and the un-park protocol then runs on the chosen row); parking itself disputed (park-vs-advanced) → the ACTIVE section (parking is the claim under dispute; deterministic on both machines); row-MEMBERSHIP conflicts (present on one side only) as `unresolved-scope` — one conflict, one shape.

## Legacy & migration shapes

Rows stamped by pre-0.4.10 versions (a bare `done (accepted: <date>)`, no lineage token, no `branch:`) are a MIGRATION shape, not deaths: ask the user once per campaign to backfill the executed choices — never re-execute them blindly; a missing lineage token arbitrates as unknown (→ the user). The migration backfill ask's outcome is RECORDED: backfilled stamps, or `branch: unknown (user <date>)` on rows the user can't reconstruct — a recorded unknown is terminal (never re-asked; the audit treats it like `kept`, no ancestry claim to check — terminal against RE-ASKING, not against the user: a later user-volunteered reconstruction — a dated genuine `branch:` value with its D-line — supersedes the unknown by KIND, like a closing record over a declined backfill, and unions preserve both records), and a declined/unreconstructable backfill also writes its Decision-log EVENT line (the schema's declined-backfill record — the D-log trace of the once-per-campaign ask); a `branch:` part carrying a bare choice word without its artifact (`merged` with no sha — the 0.4.10 shape) joins the same backfill class: recover the artifact from git history with the user aware, never treat the bare word as fully evidenced.

A BARE milestone `closed` is the pre-0.4.16 migration shape — ask the user once to backfill, offering the two HONEST shapes: the audit ran and its rounds are reconstructable → the audited shape; otherwise the waived shape (the closure stands on the user's word — never fabricate an audit record); BOTH shapes and the D-log event are dated by the HISTORICAL closure the user attests (the backfill moment is only a D-line annotation) — the shape's date and the record's arbitration date are one date, and a backfill never outdates a later genuine record such as a re-open; the user attests the closure but cannot DATE it → the shape carries `date unknown (attested <ISO date>)` in the date's place, and its D-log event line SORTS by the attested date (its only date, the line noting "historical date unknown") — the renumber's total order holds (sorting is bookkeeping), while ARBITRATION treats the record as unorderable against any dated record → the user; the user declines or cannot attest at all → the status stays bare `closed` and the declined-backfill D-line (dated by the ask) is the TERMINAL record making the ask once-only: the router thereafter passes the milestone as closed on that record (the decline was the user's answer) while the shape still never reads as audited — and a declined record meeting a GENUINE closing record for the same milestone at a merge (two machines asked concurrently, one answer attested) is not a contradiction to collapse: the closing record answers what the decline left open and GOVERNS — by record KIND, never by date (a decline is a non-answer, and this holds for the undatable-attested closing shape too), the decline stays as D-log history; a datable attested backfill carries a dated closing record like any genuine closure, the undatable shape carries its attestation record — either way arbitration reads records, never bare shapes; never route past a bare `closed` that has neither been asked nor carries the declined record.

A legacy PROGRAM.md without the `base:` line is backfilled at first read — AFTER a FULL-refspec fetch when a remote exists (a single-branch clone's default fetch brings nothing new; stale refs derive a stale answer): "newest state commit" is by HISTORY, not timestamp — a state commit that is an ancestor of no other state commit (same-second timestamps and clock skew make date order wrong; several such tips = diverged histories → the user); candidates are LOCAL branches (refs/heads) containing it AND containing the campaign dir's FULL state history back to the creation commit (a branch holding only the newest commit — a misplaced write — is not a candidate; a remote-only container is surfaced as information, its local branch checked out first — a remote-prefixed name is never written: no ON-base compare can ever equal it) — exactly one → write it (a repair, named to the user); zero or several (live phase branches forked after that commit routinely qualify — expect the ask) → the user, never a guess; a merge can leave `base: unresolved (between: … ||| …; merged <date>)` (the PROGRAM.md section cell above) — a standing ask, never a branch name to pull or compare against.
