# Campaign router (normative)

Verbatim campaign doctrine (0.7.0 distillation; split from mechanics.md by read-trigger in 0.7.x). Where SKILL.md's sketch differs in detail, THIS file governs. Internal references like "Rules", "Mechanics above", "(below)" refer to the pre-split SKILL.md layout; their targets live across the `references/` files — `mechanics.md` is the router. Merge cells and legacy shapes: `arbitration.md`.

## 8. Router mechanics (step 0 and the scans)

**Step 0 — pull first** when a remote exists (Mechanics above; failure shapes routed there) — the branch pulled and read is PROGRAM.md's `base:` line, read from the local file BEFORE the pull (the schema; a legacy file without the line runs its backfill arm first; a `base: unresolved (…)` value is arbitration.md's standing ask, not a branch name — routing suspends on it), never a guessed default; STEP 0 LOOPS: if the pulled state rewrites `base:` (or delivers a base-change / divergence-answer D-line), re-run step 0 against the NEWLY named base before reading anything else — the first pull was consumed by the old branch, and the new base's local checkout may be stale (bounded: a second rewrite on the re-run durabilizes `base: unresolved` → the user, never a chase); and when the local Decision log carries a base:-ANSWER or base-change D-line (both kinds — the order path records `base change`, not `divergence answer`), verify its TRANSPORT RECEIPT per arbitration.md's receipt rule (ancestry, its carry-receipt D-line for squash bases, its shallow caveat) — not receipted → the owed carry runs FIRST, before anything routes on that base (a cheap check that only fires when such a D-line exists). Then read ROADMAP.md and PROGRAM.md's Milestones section and Decision log — the router's own checks below read D-lines — (nothing else yet) **from the BASE checkout** — a phase worktree's copy is stale as of its fork; resuming inside a worktree (e.g. via a handoff), locate the base checkout first, and anchor every `.catalyst/` path to the repo root, never the cwd; the self-consistency sweeps are INCREMENTAL where an anchor exists: a session re-sweeps only the D-lines and rows the pull/merge INTRODUCED (the diff against the pre-pull local head — git shows it); the FULL-log sweep runs on first contact with a campaign on this machine (no prior local head) — never on every run of a long campaign (the spec test's frontier principle; an agent-executed full scan of hundreds of lines gets silently skimmed under pressure, which defeats the detection); a roadmap failing self-consistency splits by severity: duplicate phase ids or a status outside the enum make the roadmap structurally unreadable — surfaced alone, routing aborts before any batch (the sanctioned merge shapes — `unresolved (between: … ||| …; merged <date>)` and `unresolved-scope (was: …; merged <date>)` — are NOT outside the enum (references/arbitration.md): standing batch members, never an abort); a phase id no Decision-log line accounts for (neither the creation line nor a later scope line — a hand-added row; scope edits are never silent), the MIRROR orphan — a D-line ASSERTING A LIVING REFERENT whose named phase id or milestone appears in no roadmap row (kinds whose semantics REMOVE their referent — a scope-remove answer — are expected-absent by design, never orphans (a PARKING event's referent lives in ## Parked and resolves as a ROW — parked rows are rows, never expected-absent), and a creation line whose row a later remove-answer D-line accounts for is paired WITHIN the log: the check is log-internal first, no history archaeology in the happy path; only an unpaired living-referent D-line escalates to row history, and history absence concludes only on FULL history — unshallow a shallow clone first), a record fabricated beside nothing, or a rewrite that never landed: the suspect shape — paired writes land together, so an unpaired half is surfaced, never trusted alone, a row returned from ## Parked with no un-park D-line, a `reopened` row with NO accounting D-line (every sanctioned demotion binds one into its commit — a voiding event or the amendment D-line; EXCEPT a reopened row whose spec carries the pending marker: the marker-only legacy amendment shape, sanctioned; without either, the stamp is the hand-demotion suspect, batched BEFORE any fix-wave resume, never routed silently — its resolution protocol (both facts presented, attest and can't-attest outcomes, the never-defer answer write) is arbitration.md's cell), a `closed` milestone whose status carries neither the audited nor the waived record (a hand edit, or the pre-0.4.16 migration shape — the backfill ask, dated, never skipped past as closed; EXCEPT one whose declined-backfill D-line exists — that record is terminal, not a batch member), and a `closed` milestone one of whose rows is NOT `done` (a `reopened` row, an un-parked phase — arbitration or hand edits can produce it: the closure no longer holds, re-open it with the user per the un-park rule) — these are BATCH members, surfaced with the run's one question, never routed past and never an interrogation drip (a roadmap predating the creation-line rule — or one whose creation line a torn genesis lost, indistinguishable from state — has its provenance baseline in git instead: ids present in ROADMAP.md's first committed version are accounted for — unshallow a shallow clone first, its truncated history's "first version" is a late snapshot).

Every "surface to the user" the rest of this router run produces is a MEMBER of the one batched question (Rules) — only that self-consistency failure aborts routing on its own, before any batch exists.

An `executing` phase ANYWHERE in the campaign takes precedence: resume it first — and this scan reads WRAPPED sides too: an `unresolved (between: …)` row any of whose sides carries `executing`, and an `unresolved-scope` whose was-row does, COUNT as executing rows for this precedence check and for the two-`executing` consistency check below, in EVERY milestone (the wrapper must not hide a live phase; a wrapped executing row is not resumable — it holds its milestone on the ask instead, and no new phase starts anywhere while it stands unanswered, the same bar as a plain executing row) — milestone ordering governs which phase STARTS next, it never preempts one mid-flight (so a milestone re-opened by an un-park queues behind a later milestone's executing phase; the one-executing-phase rule wins).

TWO OR MORE `executing` rows (the sanction defines N ≥ 2 — three sanctioned phases are three executing rows, same rules) are consistent only with the parallel-sanction D-line in the Decision log whose named phase ids COVER the currently-executing rows — the live executing set must be a SUBSET of one open sanction's ids (a phase finishing normally shrinks the set without spending the sanction; a sanction naming only other ids is spent history, not a standing license; when the LAST sanctioned phase closes, the sanction is spent): matching → each session resumes the row whose owner label on the D-line is its own; a session that can't tell which label it is, or whose label isn't there, ASKS the user rather than squatting on an owned row (two sessions in one phase break the single-writer workspace); no matching D-line → surface to the user, never auto-park either row.

Otherwise take the first milestone whose `status:` is not `closed`: if ALL its phases are `done`, the route is its milestone audit (an `unresolved (between: …)` milestone `status:` is blocked-on-user — arbitration.md's bar — never auditable) — a later milestone's pending phases never skip an unclosed earlier one past its audit. Otherwise route by that milestone's first phase whose status is not `done` — and when that phase's row stands in an unresolved shape (a standing ask), OR is a `verified` row under a standing acceptance decline (its `acceptance declined:` D-line recorded and no NEW verification evidence since — the decline freezes the row: the acceptance ask is barred until new evidence or a re-open, and the re-gate is deliberately deferred with it — running it would either change nothing or force a fix wave on a phase the user just declined to move, so the row waits for one of the decline's two exits), route the milestone's NEXT not-done phase instead — skipping any further phase that itself stands in an unresolved shape by the same rule; no candidate left → the milestone stalls — (the stall binds THE ROW and the milestone's closure, not every disjoint phase: phase order within a milestone yields to a standing ask UNLESS a later phase's spec/plan names a dependency on the stalled one, or ANY side of the unresolved pair — or an `unresolved-scope` was-row — carries `executing`: the campaign-wide executing scan above already counts such a row as executing (a phase may be live on another machine) — it holds the whole campaign's new starts on the ask, never relieved; the shape stays a batch member every run, and the milestone cannot close or audit over it — program-wide suspension is reserved for `base: unresolved`).

## 9. Reconciliation, both ways

Before dispatching into the route, reconcile BOTH ways: (a) forward — if the phase's NEXT evidence already exists on disk (an approved spec for `pending`, the ledger's `verification: converged` line — OR its sanctioned substitute `verification: waived by user <date> — <scope>` — as the ledger's final line for `executing` — trailing `session:`/`uat:`/`minor:`/`disposition:`/`Task N: re-opened` lines are sanctioned (verification.md's below-the-line rule; a re-opened trailer is also the amendment/rejection resume signal), but TASK lines below that final verification line mean re-opened verification that never refreshed — under a converged line it is STALE, route resume; under a waived line the waiver is state-specific, route the disposition ask (a `Task N: re-opened` trailer first dispatches the re-opened work — the ask follows the wave, verification.md's placement) (verification.md); don't flip either way; resolve the ROADMAP ledger link, which may point into a phase worktree; an unreachable ledger is NOT evidence: route resume, and arcane re-derives from `git log` only on the machine that has the phase branch — on a fresh clone without it, the phase's progress is unrecoverable — the UNRECOVERABLE-PROGRESS rule the routes below invoke by name: say so to the user, never silently restart the phase), the previous session died in the flip window — flip (with the stamp) first, then route — UNLESS the ROW-HISTORY TEST fires: the roadmap's own history shows the row previously carried the FURTHER status and a later commit rewrote it backward (row-level — `git log -L` on the row or searching the roadmap's history for the row's earlier value; never file-level, which false-positives on any later commit touching the file): that is a deliberate demotion, a user-side decision — surface it in the batch, never silently re-flip forward (a user decision is never silently discarded; an UNCOMMITTED backward edit in the working tree is the same decision — surfaced, not overwritten; the death shape leaves the row NEVER having carried the further status in any commit — the evidence exists but the flip never landed);

(b) backward — over EVERY row of the routed milestone, not just the routed phase (this is the only checkpoint the milestone-audit and terminal routes pass through — a hand-edited stampless `done` on the LAST phase must not slip a milestone closed): a status whose row lacks its evidence STAMP and whose evidence isn't reachable locally either (`planned` with neither stamp nor plan file, `done` with no acceptance stamp — a hand edit?) → do NOT route on it: surface the gap to the user with the evidence rule quoted; a `done` stamp missing its `branch:` part — or carrying a bare choice word without its artifact (`merged` with no sha, the 0.4.10 shape: recover the artifact from git history with the user aware, per the backfill class): a BARE pre-0.4.10 stamp (no lineage token either) is the migration shape — the backfill ask per references/arbitration.md, never a re-execution; a token-carrying one died in the choice window — verify against git FIRST: base already contains the phase head → the choice DID execute, recover the merge sha from `git log` and append the artifact instead of re-asking (ancestry lies in two known shapes: a squash-merge integration leaves the phase head a NON-ancestor — check the integration history for the squash commit before concluding "never executed"; a shallow clone unshallows first — a false "not integrated" here re-opens an integrated phase and can void its acceptance); otherwise route to arcane's branch-finish question for THAT phase first, never past it ("keep the branch as-is" is itself a recorded choice — `branch: kept`);

if the re-gate on that resume goes red, the fix re-opens convergence and VOIDS the acceptance (it no longer references the final head): demote the row IMMEDIATELY at the red — `verified (converged: reopened <date>; was rounds N..M)` (waived lineage: `verified (waived: reopened <date>; was waived <date>)`), the voiding D-log event riding the SAME commit (the amendment path's one-commit rule — no crash window between void and record), the one sanctioned backward flip, committed — and pushed when a remote exists (Mechanics' never-defer push rule) — before the fix wave starts so no other machine integrates on the voided acceptance meanwhile — the voiding is named to the user and Decision-logged (a recorded acceptance never vanishes silently); re-verification rewrites the stamp with its refreshed lineage token — replacing ONLY the `converged:`/`reopened` clause: a was-clause carrying executed parts (`done <date>, head:, branch:`) stays in the refreshed stamp verbatim until a NEW `done` stamp supersedes it (the schema's demoted-shape rule) — (a waived lineage routes the post-fix disposition ask first — the `verified` route below), and acceptance re-runs before any new `done` with the UAT pass re-offered in FULL (the fix changed the behavior the old `uat:` lines recorded — they are void with the acceptance); a stamp git REFUTES on a full history (a `merged <sha>` that is not an ancestor of the base branch) → surface to the user with both facts, never routed on and never auto-repaired — and on a shallow clone an absent commit proves nothing: unshallow first, refute only on full history — and the ask's answer is RECORDED so it never re-fires (the correction repair or the terminal `branch: unknown (user <date>)` — arbitration.md's migration class); a stampless row whose evidence IS locally reachable → write the missing stamp from that evidence now (a repair, committed), then route — left stampless, the row loses the next stamp-beats-stampless arbitration to any stale stamped counterpart;

a stamped row is evidenced on every machine ONCE its repo-file referent is committed: for `specced`/`planned` rows resolve the referent path in the BASE tree — and a spec whose open/deferred records a spec-gate verdict requires `<spec>.premortem.yaml` beside it (premortem's detectability rule); an absent referent is a half-done flip or a row predating the referent-commit rules, not a death: recover the file from the phase branch/worktree — or from base git history when it was committed and later deleted (unshallow a shallow clone first) — when reachable, and commit it (a repair), otherwise surface it in the batch — NEVER silently re-run crucible or plan-writing over an approved-but-uncommitted artifact; rows whose stamps carry no repo-file referent need no reach — no false alarm;

and a `verified` stamp's LINEAGE TOKEN is a cache of the ledger's verification line: on the machine where the phase ledger is reachable, cross-check it before anything consumes it — on mismatch the ledger wins (the deeper source; the stamp rewrites, a repair, committed with the state), with two guards: a reachable ledger with NO verification line at all is an INCOMPLETE source, never a mismatch (verification lines live only in the ledger and a rebuild from `git log` cannot recreate them — the stamp, possibly the ONLY record of a user's waiver, stands); and a `done` stamp's token is NOT a cache — it RECORDS which verification the acceptance referenced (the schema): a ledger whose last verification line carries a LATER token than a `done` stamp is the overtaken-acceptance signature → the user (arbitration.md's done-vs-verified overtaken-acceptance cell), never a rewrite — rewriting it would fabricate an acceptance of rounds the user never saw; an EARLIER last verification line under a `done` stamp is the incomplete-source shape (a partially recovered ledger — the stamp's referent line is missing, not contradicted): the stamp stands.

An unreachable ledger leaves any stamp standing (the evidenced-on-every-machine doctrine), never re-derives.

Then warm up: if `.catalyst/handoffs/` (or `thoughts/shared/handoffs/` — the ContinuousClaude tree handoff writes to when the project has one) holds a handoff carrying this campaign's pointer, read the latest one — the roadmap decides WHAT happens next, the handoff's mental model saves re-discovering HOW things work.

## 10. Route table — full cells

| State found | Route |
|---|---|
| Phase `executing` | resume it: its ledger + `git log` are the truth (arcane-mode rules) — and the resume's SPEC and PLAN referents resolve in the BASE tree (the state-transport home: never-defer answer commits — reverts, re-open annotations, pending markers — land there; the worktree's copies are stale as of its fork, so the worktree merges base forward before dispatching re-opened or amended ground — a CODE conflict in that merge is ordinary git conflict work surfaced to the user, never auto-arbitrated (the phase-divergence principle)) — never start another phase on top |
| Phase `planned` | dispatch into arcane-mode execution |
| Phase `specced` | check the ROADMAP plan/ledger links first: a ledger naming this phase's plan AND carrying a PASS/WARN premortem verdict line holds `planned`'s full evidence (a death before the flip) — cross-check `<plan>.premortem.yaml` before flipping: the line is a CACHE and the yaml wins in every direction (arcane's verification rule — a fresh BLOCK yaml under a stale WARN line means the gate failed; a stamp written off the stale line would make the false verdict durable on every machine); a MISSING yaml under a verdict line is first a LOST referent — recover it from git (committed with the plan; unshallow a shallow clone first — absence in truncated history proves nothing) before declaring the gate owed (verification.md's rule, including its stale-recovery arm: a recovered yaml whose mismatch the line's named acceptances EXPLAIN repairs from the line ONLY under that rule's CORROBORATION precondition — a COMMITTED record backs the acceptance; uncorroborated, the acceptance is UNATTESTED and is RE-CONFIRMED with the user in one line (the ledger line is a forgeable machine-local cache, never promoted flat) — an unexplained mismatch re-runs the gate; never blessed flat, and a genuinely corroborated acceptance is never re-asked); flip and route only on a yaml-confirmed verdict; a ledger with NO verdict line means the plan is approved but the gate is owed — resume at the premortem gate, the status stays `specced` until the verdict; a BLOCK verdict line means the phase is premortem-blocked — route to the pending mitigation rework / user acceptance decision (premortem's BLOCK rule), never to plan-writing and never to Task 1; otherwise arcane-mode step 1: write the plan; its premortem gate precedes Task 1 |
| Phase `pending` | catalyst:crucible for its spec — the spec is produced standalone, but control returns to THIS router (no self-routing onward); a markerless spec already linked on the row whose approval the campaign's history records (a pre-marker effort) → arcane's fast re-approval arm (named to the user), never a silent re-interview; a spec carrying `status: approved (edits pending re-approval: <date>)` — or the waived combo carrying that clause — → the same arm for the sections its open/deferred entries name as pending (the owed ask — a crash before it left this shape on purpose), never a re-interview; fog wider than a question → starchart first |
| Phase `verified` | a `reopened` stamp first: re-verification is in progress — EXCEPT a reopened row whose spec carries the pending-re-approval marker or whose demotion D-line names an amendment: that routes to arcane's AMENDMENT wave (its `references/amendment.md`) (on an integrated phase — the was-clause's `branch:` part shows the executed choice — no machine requirement applies, nothing is unrecoverable; a demoted row with NO `branch:` part is FIRST verified against git — the executed-and-unrecorded arm WITH its ancestry caveats (a squash-merge integration leaves the phase head a NON-ancestor: check the integration history for the squash commit before concluding nothing integrated; a shallow clone unshallows first): base containing the phase head OR its squash means the integration DID execute (recovered into the was-clause for a demoted `done`; surfaced as integration-WITHOUT-acceptance for a demoted `verified`) and the integrated arm applies on any machine; genuinely nothing integrated → its wave runs off the still-live phase branch, on the machine with that branch, elsewhere the unrecoverable-progress rule); otherwise resume the fix wave/rounds (a waived-lineage `reopened`: the fix wave, then the disposition ask below) on the machine with the phase branch (ledger + `git log`), elsewhere the unrecoverable-progress rule; never treat it as a normal verified row. Otherwise FIRST the moved-base re-gate: base advanced past the branch's last verified merge point (the fork, or a previous re-gate's merged head whose suite AND tail gates ran green — a red re-gate's head is not a verified point; "since fork" alone re-fires forever after any advance) with advances that can affect the BUILD — an advance consisting SOLELY of the family's bookkeeping commits (`.catalyst/campaign/` and `.catalyst/map/` paths, flip-carried specs/plans/premortem yamls, the sanctioned `.gitignore` repair; check `git diff --name-only <verified point>...base` — THREE dots, merge-base semantics: a two-dot diff from a merged-head verified point lists the branch's own files and never reads state-only; the `.gitignore` member alone needs the PATCH, not the name — `git diff <verified point>...base -- .gitignore` must show only the sanctioned repair: the `.catalyst/*`-plus-negations lines it ADDS and the swallowing rule lines it REMOVES; any other `.gitignore` change can affect the build and re-fires) does not re-fire the re-gate: the campaign's own bookkeeping must not tax every phase with a full re-gate — → merge it forward into the phase branch, re-run the suite + tail gates on the merged head (arcane's branch-finish rule) — a red is a fix wave, then re-verification: on a converged lineage convergence is re-entered until the stop criterion holds again, the converged line refreshed AND the row's `verified` stamp rewritten with the refreshed rounds (stale rounds in the stamp force the arbitration escalation later) — or the user EXPLICITLY waives the re-run (verification.md's review-floor rule applies to re-runs too): the waived line lands and the stamp rewrites converged→`verified (waived: <date>)`, the lineage converts; on a WAIVED lineage the recorded waiver covers the pre-fix code state — after the fix wave, ONE question to the user: run the rounds now (the lineage converts to converged) or extend the waiver (a fresh `waived by user <date>` ledger line + stamp date) — neither arm re-demands the original waiver, and neither stretches it silently over unreviewed code; THEN acceptance on the FINAL head (UAT offer per arcane-mode's `references/verification.md` — the phase's one UAT pass, deferred out of arcane's verification step; never twice — EXCEPT a voided pass, which re-offers every scenario (the old `uat:` lines recorded pre-fix behavior), and an INTERRUPTED pass, which resumes from the first unrecorded scenario (verification.md's resumable rule)); acceptance has THREE recorded dispositions, never a silent non-answer: ACCEPT → `done (accepted: <date>, <lineage token>, head: <sha>)` — `head:` = the final phase head the acceptance ran on (the schema's pin; branch-finish verifies the tip against it); DECLINE (user says not ready / wants rework, without a red re-gate) → the row STAYS `verified` and the decline is recorded — a D-line (`acceptance declined: P<N> <date>, <lineage token> — <reason>`, the token = the row's verification lineage the decline was given AGAINST) committed never-defer — and the acceptance ask does NOT re-fire until NEW verification evidence lands, decided MECHANICALLY: the row's CURRENT lineage token differs from the one the standing decline records (a fresh converged/waived token can only appear through re-verification), or the user explicitly re-opens; re-asking on the recorded token is the interrogation drip (a legacy decline D-line without a token compares by the coarser test — the row's CURRENT stamp lineage dated after the decline; the stamp travels with base commits, so the test is executable on every machine, unlike the self-ignored machine-local ledger — and the next decline records the token form) — but the standing decline is NOT silent: every router run's batch carries one INFORMATIONAL line naming the declined phase, the recorded reason, and the two exits (a user re-open/amendment for the rework the reason asks for, or accept on explicit user order), and for phase ordering the routing rule treats the declined-standing row like an unresolved shape — the milestone's next not-done phase routes past it (the stall rule's dependency and executing caveats apply), while milestone closure and audit still wait on the row (a campaign never silently completes over a decline, and never silently waits on one either); PARK → the Parked protocol (its own user decision). The accept arm continues: the token just verified (rounds, or the waived date) → arcane's branch-finish question executes the choice and APPENDS `, branch: <choice>` to the stamp. The flip and the append are two commits by design: a death between them leaves a `done` row without `branch:`, which backward reconciliation routes back to the question — never past it |
| All phases of a milestone `done` | milestone audit (below) — mandatory, before any celebration or next milestone |
| Every milestone `closed` | the campaign is COMPLETE: write `status: complete` (with the date) at the top of PROGRAM.md, commit, tell the user — nothing routes; bootup skips complete campaigns. The terminal route does NOT fire over a standing unanswered ask — a parked pair, any unresolved shape, OR a pending, double-answered, or vanished-section (lost-referent) spec entry, OR a premortem-yaml `withdrawn:` record whose re-acceptance never landed — a withdrawn risk acceptance on a shipped row is a standing BLOCK (spec-gate's reversal machinery), not history (crucible's shapes — its `references/marker-machine.md`) on ANY row including `done` ones (the completion sweep reads every row's spec marker across EVERY milestone — all of them are already closed here, and closure sweeps can predate later answers' merges, so the terminal sweep re-covers the whole campaign, never only the last-closed milestone: after `done`, nothing else consumes the spec, so this sweep is the ask's last reader; a spec unreachable on this machine is a LOST REFERENT, never a silent pass — recover it from git history first (unshallow a shallow clone first — absence in truncated history proves nothing), genuinely absent → the sweep bars and surfaces it; a REJECTION answered on a `done` row creates a spec-code divergence — the code implements the rejected edit: the answer surfaces it and offers the amendment route, completion never rides over it silently): completion is its last surfacing channel — the final batch carries the ask, and the campaign completes only on the answer or the user's explicit recorded disposition to close over it (a D-logged decision, never a default). Both evidenced shapes count — `closed (audited: …)` and `closed (waived: …)` — and so does a bare `closed` standing under its declined-backfill D-line (the schema's terminal record: the campaign completes on it, the shape still never reads as audited); a bare `closed` with NO such record is the self-consistency backfill case, never silently complete |
| No roadmap | campaign creation (below) — only when the user actually asked for a program; a bare "what's next" with no roadmap is not that ask → route back (handoff resume / bootup routing) |

One session advances ONE phase's state as far as it honestly goes; parallel phases only by explicit user decision and only when file-disjoint (the sanction rule's hard precondition — one rule, stated there).
