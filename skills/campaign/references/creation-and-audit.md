# Campaign creation and audit (normative)

## 11. Creating a campaign — full steps

1. Intent and milestones — through catalyst:crucible (the intent is a decision, not a transcription); starchart's finished map converts naturally AND completely: Destination → Intent, closed decisions → Decision log (each D-line carrying its Resolution's ORIGINAL date), ## Out of scope → Non-goals (the scope fence survives the conversion — nothing graduates back silently), ## Notes' standing agreements → Decision log (each with its own date; an UNDATED legacy Notes line converts dated by the CONVERSION and annotated as such — a historical date is never invented), remaining work clusters → phases; the creation D-line names the map path (the pointer starchart promises).
2. Slice phases tracer-first (each phase leaves the system demonstrably better/shippable), 3-9 phases per milestone (a legitimately smaller MILESTONE — 2 phases, a small program or a trailing milestone — is fine: the floor guards against skipped decomposition, not small scope; ONE phase is the single-feature red flag — at CREATION this REFUSES, not merely warns: route the effort to crucible → arcane-mode instead, and create the umbrella anyway only on the user's explicit order after the cost is named in one line; the red flag below is the same rule seen from inside a standing campaign); a phase should fit one arcane-mode plan (a few tasks) — a bigger chunk splits into more phases; a milestone is a set of phases, never one oversized phase.
3. Write both files — PROGRAM.md's `base:` line records the branch the approval commits on (`git branch --show-current` at creation; the schema's field every later ON-base check reads) — get the user's approval of the roadmap as ONE question, set every milestone `status: open` and every phase `pending` — and Decision-log the approval as the creation line naming the approved phase ids — ROADMAP.md, PROGRAM.md, and the creation line land in ONE genesis commit (paired writes land together: a torn genesis leaves every row a hand-added suspect on first resume) (the provenance every later scope check reads).

## 12. Milestone audit — full protocol

When a milestone's phases are all `done` — each row carrying its full `done` stamp (a stampless, `branch:`-less, or bare-choice-word row is a backward-reconciliation case, not an audit input), and the audit spot-checks the `branch:` claims against git first: a `merged <sha>` whose commit is not an ancestor of the base branch is a claim git refutes — a reconciliation case, never an audit input: goal-backward audit of the MILESTONE against `PROGRAM.md ## Intent` and the milestone's must_haves — fresh-eyes auditor (top tier, clean context — the milestone-intent tiering's normative home), the same discipline as arcane verification but one level up — and the audit FIRST sweeps the milestone's rows' spec markers AND premortem yamls with the terminal sweep's own arms: a pending, double-answered, or vanished-section (lost-referent) entry, or a `withdrawn:` acceptance with no landed re-acceptance (the shipped-risk divergence — surface it and offer the amendment/fix-phase route, the same arm as a rejection answered on a `done` row) — crucible's shapes — is a standing user ask that bars closure (batched), exactly like an unresolved shape; a spec unreachable on this machine is a LOST REFERENT, never a silent pass — recover it from git history first (unshallow a shallow clone first), genuinely absent → the sweep bars and surfaces it; a REJECTION answered on a `done` row is a spec-code divergence — the code implements the rejected edit: surfaced with the amendment route offered, closure never rides over it silently: phases all done ≠ intent achieved (integration seams between phases are exactly what per-phase verification never saw). Findings → fix phases into the roadmap (user approves), not silent patches. Convergence uses the same stop criterion — iterative rounds until 2 consecutive clean; then the milestone closes in PROGRAM.md as `closed (audited: <date>, rounds N..M)` PLUS its D-log event line — the closing record other machines arbitrate on; a `closed` written without its record is exactly the hand-edit shape the arbitration refuses to trust; next milestone or campaign end — the user's call. The user's EXPLICIT order to close WITHOUT the audit bends this gate like any named gate (arcane's Iron rules): cost stated in one line, then write `status: closed (waived: <date>)` — the waiver's recorded home — plus the D-log event; a recorded waiver is never re-demanded (a bare `closed` is never freshly written as a closure — it survives only as the migration/hand-edit shape the backfill ask exists for, or standing under a declined-backfill D-line, the schema's terminal record).
