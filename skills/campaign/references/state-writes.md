# Campaign state writes (normative)

Verbatim campaign doctrine (0.7.0 distillation; split from mechanics.md by read-trigger in 0.7.x). Where SKILL.md's sketch differs in detail, THIS file governs. Internal references like "Rules", "Mechanics above", "(below)" refer to the pre-split SKILL.md layout; their targets live across the `references/` files — `mechanics.md` is the router. Merge cells and legacy shapes: `arbitration.md`.

## 2. Committed state, gitignore repair, repo-less runs

Rules: PROGRAM.md and ROADMAP.md are **committed to the repo** — they are the program's memory for weeks and must survive `git clean`, clones, and machine changes. No git repo at all → the state cannot be committed: surface it to the user (init a repo, or an explicitly accepted ephemeral run — which covers CAMPAIGN state only: a phase routing into arcane execution re-raises arcane's repo gate there, git-init or the phase does not execute), never proceed silently — starchart's arm. If the project's `.gitignore` swallows the committed dir — a wholesale `.catalyst/` rule, OR a `.catalyst/*` rule missing this dir's negation (the shape an old half-remedy or a hand edit can leave) — the first state commit fixes the rule (named to the user, never silent): ensure `.catalyst/*` plus `!.catalyst/campaign/` and `!.catalyst/map/` — a bare negation UNDER an ignored directory doesn't work (git never descends into it), and `git add -f` is banned: it leaves the rule in place so every future clone re-hits the same swallow.

## 3. State commits: base branch, transport, pull/push failures, the write guard

State commits go to the campaign's BASE branch (the branch phases fork from), never onto a phase work branch — a discarded phase branch must not carry the program's memory away with it. Mechanics: when a remote exists, the router pulls the base checkout FIRST (step 0 below), and session end pushes the state commits AND the phase branch (an unpushed phase branch is unrecoverable from any other machine). Before that pull, a flip edit deferred by the single-checkout workflow (below) is committed FIRST — commit-then-pull, never stash. Pull failure shapes are routed, never improvised: the pull hitting divergence → fetch and merge, conflicts resolved by the cells in `references/arbitration.md`; remote unreachable → proceed on local state and tell the user the roadmap may be stale; a remote ATTACHED mid-campaign (no upstream at contact) → set the upstream, push the base state, and PUBLISH every live local referent from the machine that has it — kept and executing phase branches, the never-defer history — named to the user; the publish duty keys on ANY live local referent absent from the remote, checked at step 0 whenever a remote exists (a partial publish resumes — never first-contact-only); UNRELATED histories at contact (two machines attached the same fresh remote independently) → fetch and merge with the explicit unrelated-histories allowance — the one sanctioned use — file-level add/adds of STATE files (the campaign dir, flip-carried specs/plans/premortem yamls) arbitrated by `references/arbitration.md`'s cells (both trees are legitimate state), while SOURCE-file add/adds are ordinary git conflict work surfaced to the user — these rules never auto-arbitrate code, the phase-divergence principle — never a force-push (until published, other machines legitimately read local-only state through the recovery arms). Push failures are routed the same way: a rejected push of STATE commits (the remote advanced during the session) → fetch + merge under the same arbitration, then retry once; still failing, offline, or auth-refused → tell the user the state is local-only (the red flag below is about SILENT unpushed state, not about a named failure). A rejected push of the PHASE branch because it DIVERGED (two machines worked the same phase — nothing detects this earlier) is a CODE merge, not state arbitration: stop, surface to the user whose work continues where, resolve by ordinary git conflict work on one machine — these rules never auto-arbitrate code. EVERY state write — an answer recording, a flip, a demotion — RE-READS its referent from disk immediately before writing: a referent changed since the ask/derivation was built (another session's commit — two windows on ONE checkout produce the merge problem WITHOUT a merge, so no cell would ever fire) means the ask is STALE — adopt the landed record like a fetched answer, re-derive, never overwrite (a recorded decision is never last-writer-wins; adoption covers asks not yet answered locally — a landed record that CONFLICTS with an answer the user already gave THIS session is two user decisions → the user per arbitration's conflicting-answers rule, never a silent drop of either). The re-read NARROWS the race window, it does not close it (two writers can both read clean and then both write — TOCTOU): the serialization anchor is the COMMIT — every state write commits IMMEDIATELY (the never-defer discipline), and the commit step carries its OWN guard, because git's index lock serializes only simultaneous commit invocations — a second SEQUENTIAL committer writing from a stale full-file buffer commits cleanly and silently reverts the sibling's landed record, no conflict and no failure: immediately before staging, re-check that the file's blob at HEAD still equals the blob the pre-write re-read saw (`git rev-parse -q --verify HEAD:<path>` against the hash noted at re-read — the command legitimately FAILS when the path is not at HEAD yet: an unborn HEAD, the genesis commit, a first write of a new state file, a just-renamed path — that failure is the ABSENT state, not an error to improvise around: absent at re-read AND absent now = clean to proceed; absent at re-read but present now = a sibling landed the file since — re-read and re-derive; present→changed → same re-derive, never proceed; and ABSENT is concluded on the exact TREE spelling only — repository paths bind by their git-tree byte spelling: when the tree lookup fails but the file OPENS on disk (a case/Unicode-normalization alias resolved by an insensitive filesystem — macOS default) or `git ls-files` holds a casefold/normalization sibling, that is NOT the absent state, it is an identity mismatch → the user, never the absent arm (the absent arm on an alias would silently fork a second identity of an existing file); a write lost in the residual check-to-commit window is DETECTED by the WRITER ITSELF, immediately after its commit — the writer re-reads the new commit's PARENT blob of the file (`git rev-parse <new commit>^:<path>`) and compares it against the hash its pre-staging check saw: a mismatch means a sibling landed inside the window and this commit was built on a stale buffer — the writer re-derives from the parent's actual content and RESTORES the vanished record in an immediate follow-up commit, then the two records arbitrate normally, nothing is last-writer-wins (this post-commit check is the claim's EXECUTOR: the incremental sweep keys on pull-introduced lines and its checks are presence-based — a local same-checkout overwrite is invisible to it, so without this step the insurance had no executor), and state files are edited surgically in place, never regenerated from a session's in-memory model (a regeneration silently reverts every sibling's landed write). State commits verify the base checkout is actually ON the base branch first — `git branch --show-current` compared against PROGRAM.md's `base:` line (the schema; a legacy file without it backfills the line first), never a guessed default: a detached HEAD or a wrong branch is surfaced, never committed onto — a detached-HEAD commit lands on no branch and dies silently in the reflog. **Arbitration & legacy shapes — the normative home is `references/arbitration.md`.** Any merge conflict in state files (ROADMAP.md/PROGRAM.md/specs/plans/premortem yamls), any standing `unresolved (between: …)` / `unresolved-scope (was: …)` / `base: unresolved (…)` shape, a base:-answer D-line whose transport receipt is missing (the answer commit not an ancestor of the base's head), the user's ORDER to change the base, and any legacy shape WITHOUT its terminal record (a bare `done (accepted: <date>)`, a bare milestone `closed`, a single-pipe pair, a missing `base:` line — but a bare `closed` standing under its declined-backfill D-line is RESOLVED: no read owed, the router's inline terminal rule handles it) → STOP and read that file ENTIRELY before resolving, repairing, or routing past anything — its cells are normative, and resolving a state conflict without them is a defect. Its →user outcomes are members of the one batched question (below); its recorded answers are never-defer commits (the class below).
