# Family pressure harness — mechanics (historical + new-rule baselining)

Moved VERBATIM from SKILL.md in the 0.7.0 distillation. POLICY (user-ratified, journal 0.6.3 NOTE): the harness serves its ORIGINAL purpose only — RED-baselining a genuinely NEW rule before authoring it. The per-edit GREEN regression gate and the clean-rounds convergence criterion for skill texts are RETIRED (a code-defect convergence criterion misapplied to prose; the harness's live sessions are expensive). The two paragraphs below are preserved in full: §1 for the still-live journal record machinery it names, §2 as the operating manual whenever a new-rule baseline IS run.

## 1. The original Iron Law paragraph (pre-0.7.0; the GREEN-per-edit clause is retired, the baseline-credit and journal-record rules remain live)

**Iron Law: A SKILL EDIT SHIPS ONLY AFTER A GREEN PRESSURE RUN.** For a new skill the baseline (RED) comes first — the recorded rationalizations are the requirements the skill must answer; and a COMPLIANT baseline is credited to clean model defaults only after checking the transcript for reliance on machine-injected process rules (rules loaded outside the harness's CLAUDE.md rename name themselves in the reasoning) — reliance found means the rule still earns its place: another machine lacks the injection. The user's EXPLICIT order to ship ungated is taken and recorded in the tests journal as its own anchored line — `GATE OWED <ISO date>: <edited files> — shipped ungated by user order` (new lines backtick-quote each file — `` `path` ``, comma-separated — so a path containing `, ` cannot split the list; readers fall back to comma-splitting only on a legacy line with no backticks, the paths: convention) — the owed gate stays visible and runs at the next session, it never silently becomes a passed one.

## 2. Harness mechanics (`tests/pressure/`)

`tests/pressure/`: one scenario file in `scenarios/<name>.md`, one line in `map.tsv` (`name<TAB>model<TAB>file[<TAB>file...]` — the FIRST file is the primary surface the scenario exercises (a SKILL.md or a references/*.md); related files are ANY family files the scenario exercises: references, other skills' SKILL.md, agents/*.md; a family surface no row names is OUTSIDE the gate — a coverage hole to close, not a free pass). Shell scripts are family surfaces too, gated by NAMED script-level tests instead of map rows (prose scenarios can't execute them): editing `tests/pressure/run-baseline.sh` OR `run-green.sh` obliges a `tests/pressure/test-baseline-lock.sh` run — that test NAMES its own coverage (the liveness classifier, the symlink-lock acquire/ownership machinery — free acquire, live-holder refusal, stale takeover, legacy-dir conversion — `bash -n` on both runners, and the skip-class gate on a sandboxed run-green copy) and nothing more: run-baseline's full lifecycle (traps, CLAUDE.md rename/restore, scenario loop) has no safe smoke — a smoke would rename the real CLAUDE.md — and is NOT covered by it (run-green's lifecycle IS smokeable and has its named smoke below); editing `run-green.sh`, `task-brief`, `review-package`, or `sdd-workspace` obliges `bash -n` plus that script's NAMED smoke — the smoke bodies are normative, and a smoke that exercises fewer contracts than named here is the shipped-blind-spot class: **task-brief** — extract from a scratch plan with plan-standard's CANONICAL bulleted anatomy (an inline-only fixture passes a parser dead on the canonical shape — shipped once) confirming the backtick-quoted `paths:` header + Global-Constraints block, PLUS the refusal set: unquoted-whitespace path (exit 5), embedded/unterminated-backtick path (exit 5), case-alias of a tracked path in a scratch git repo (exit 5), duplicate **Files:** (exit 6), mixed inline+bullets (exit 6), a **Files:** line after checklist steps began (exit 6 — the unfenced-illustration harvest), approval-with-Files (exit 6), and the positive set: fenced `**Type:**/**Files:**` fixture ignored, pure-approval sentinel, inline backtick-quoted comma path surviving, CRLF plans parsing clean in BOTH forms (no CR byte in any emitted span), a checklist step directly after the Files block/inline field parsing as steps (never as a path or the mixed form), an untracked future-file path passing the alias gate, non-numeric task id (exit 2); **review-package** — forward=0 / reverse=3 / equal=0 / non-commit rev=2, terminal marker present, full-OID title; **sdd-workspace** — in a scratch dir: fresh run yields a `.gitignore` holding a line exactly `*`, a negation-only `.gitignore` gets `*` PREPENDED preserving the negations, a second run is idempotent, and a symlinked `.catalyst` (or `/sdd`) REFUSES with exit 2 writing nothing through the link; **run-green** — a one-scenario run producing a MANIFEST.txt per gate-records' run-artifact grammar. `test-baseline-lock.sh` is itself a shell surface: its named check is RUNNING it (it is its own gate). An edited script with no runnable check is the same coverage hole). `run-baseline.sh [name...]` = RED, `run-green.sh [name...]` = GREEN; both run each scenario in a neutral cwd (agents started inside the harness find the scenario files and analyze the test instead of playing the role). Score per `tests/README.md`. Editing an existing family skill → re-run the scenarios whose mapped files you touched, not just new ones — and HOT-TRIGGER rows (rows that deliberately preload only entry surfaces so the scenario proves reference DISCOVERY through a STOP-read) decouple the preload list from the edit-trigger list: editing a reference such a scenario reaches through its trigger obliges re-running that scenario TOO, even though the row does not name the reference (the row lists what preloads, not everything the scenario exercises; the named hot-trigger pairs live in tests/README.md's map notes). An edit SHIPS (merges to main / releases) only after its GREEN run passes — until then it stays uncommitted, or on an unmerged branch when the edits ride a pipeline with per-task commits; either state found at session start = a gate owed — and session start ALSO checks the tests journal (`tests/README.md` — the journal's one home) for open owed lines: grep line-anchored `^GATE OWED` lacking the answer append, then VALIDATE every append found — presence is not closure: an append whose scenario coverage or run validity fails gate-records.md's checks leaves the line OPEN (a partial `scenarios:` list looks closed to a bare grep — that is the exact smuggle the validation exists to catch) — never a literal tail, and never the bare phrase (journal prose says "gate owed" narratively; only the anchored record shape is the record; an owed line buried under later entries is still owed; a shipped-ungated edit leaves a CLEAN tree — the journal line is its only trace). An open owed line FOUND — or RETIRING, CONSOLIDATING, or DELETING a map.tsv row (gate-records.md owns the recorded-retirement discipline; a silent row deletion is banned) — is a STOP-read trigger: `references/gate-records.md` owns the record machinery ENTIRELY (the closure append's shape and its scenarios:-coverage validity, rename re-binds, lost referents, the machine-local out/ identity) — the gate runs now and closes ONLY by that file's rules. This duty binds every session that loads this skill (the honest scope: "the next session" means the next skill-editing session — nothing else reads the journal). The gate is satisfied ONLY by a GREEN run in `tests/pressure/out/` newer than the edit that COVERS the scenarios mapped to the edited files — out/ dirs hold just the scenarios that invocation ran and record no verdict (scoring is manual), so re-score that run's outputs per `tests/README.md`; a newer run of OTHER scenarios, or a baseline (RED) dir, proves nothing.

## 2a. Prose-invariant-as-code and the non-vacuous-check discipline

Two harness-adjacent techniques absorbed from field-proven kits (journal 0.8.6); both are deterministic `scripts/lint.py`-class checks, not live runs — the cheap-and-terminating side of the family's structure-over-prose rule.

**Prose invariants pinned as code.** The 0.7.x distillation moved load-bearing doctrine VERBATIM into references behind read-triggers — which creates a silent-loss risk: a later refactor or re-distillation can drop a guarantee-carrying sentence and nothing fails. A prose invariant that MUST survive every future edit (a gate's exact refusal wording, a record shape's grammar, a "never do X" prohibition that a paraphrase would soften) is pinned by a lint check asserting the exact string is PRESENT in its home file, paired with a forbidden-phrase blacklist asserting a known-wrong regression wording is ABSENT. The check names the invariant and its home; a refactor that would drop or weaken it goes red instead of shipping a quiet hole. This is the same move as the founding lint checks — recurring correction becomes structure — applied to the doctrine text itself, and it is what makes verbatim-in-references safe to keep distilling.

**A check that cannot fail is not coverage.** When adding any check (lint, script test, scenario assertion), prove it can go red: a test runner reporting "Ran 0 tests" is a FAIL, not a pass (a suite that discovered nothing certifies nothing); a presence-assertion is paired with the absence-assertion that would catch the regression; a gate meant to reject an old-shape input is proven by feeding it the old shape and confirming rejection (a negative/ compatibility check that never sees the thing it forbids is decorative). CI-class runs may never SKIP a prerequisite and report green — a skipped gate is an owed gate, the tests-journal discipline in §2 applied to the check layer.
