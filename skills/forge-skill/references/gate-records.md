# GATE OWED records (normative home)

This file is the NORMATIVE home for the tests-journal GATE OWED record machinery: the closure append's shape and coverage validity, rename re-binds, lost referents, the machine-local out/ identity, and row retirement. `SKILL.md` keeps the always-hot duty — the session-start `^GATE OWED` grep — and points here.

**Trigger — read this file ENTIRELY, never a fragment:** an open `GATE OWED` line found at session start, a closure append about to be written, or a rename/retirement touching a line's named files.

An owed gate found runs now, and ONLY a run SCORED green is recorded by APPENDING to the owed line — `— gate run <ISO date>, <green out/ dir>, scenarios: <names scored green>`; an append that FAILS validation (partial coverage, invalid run identity, legacy-time doubt) does not close the line — the line is open by VALIDITY, not by append-presence, and the superseding closure is a SECOND append of the same shape (`— gate re-run <ISO date>, <dir>, scenarios: …`; the latest VALID append governs, earlier invalid ones stay as history) — the scenario list makes the closure's COVERAGE claim readable on every machine (checkable against map.tsv without the machine-local dir); the same append-the-answer shape as the family's other records (a red run never answers the line: it stays open through the refactor); a `GATE OWED` line without that append is mechanically open — and an append whose `scenarios:` list does NOT cover the scenarios map.tsv maps to the line's named files is INVALID: the line stays open (the session-start check verifies the list against map.tsv AS OF the gate run's UTC TIMESTAMP (the out/ dir name carries it — `green-YYYYMMDD-HHMMSSZ`, minted with `date -u` from 0.5.5, with an optional `-<pid>` collision suffix from 0.5.6, and from 0.5.6 each dir carries a MANIFEST.txt binding the run to its exact revision (`head:`, `dirty:`, `map_sha256:`) — a closure whose cited dir's manifest names a DIFFERENT head than the edit's commit history reaches, or a dirty tree, does not close the line (timestamp order alone is not revision identity) — which fixes the required ordering for closure-grade runs: the RED-GREEN-REFACTOR loop legitimately runs pre-commit (its dirty-manifest runs are working evidence, not closures), but the run a closure append CITES must be a post-commit clean-tree run naming the edit's committed head — commit first, then the closure run; and a `head: no-git` manifest identifies no revision at all — it never closes a line; a legacy no-`Z` name is WALL CLOCK, zone unrecorded — when row-vs-run order within a ±1-day window would decide a closure's validity against a legacy dir, the closure is NOT excused: re-run the gate, a fresh run is cheaper than a wrong closure; the mapping the run could have known); rows added later — including later the same day — create coverage duties for future edits, never re-open a recorded closure (a recorded answer is never re-litigated) — the paired-effects principle: an append closes only when its run did what closure requires, checked by shape, never assumed). Renaming or moving a file an open owed line names re-binds the line IN THE SAME commit — append `— now <new path>` (the record follows its referent); an open owed line whose named files map to ZERO scenarios in map.tsv is a LOST REFERENT — resolved with the user, never read as vacuously satisfied. The cited out/ dir is machine-local (gitignored): it is the run's identity on the machine that ran it — a later reader elsewhere re-verifies by RE-RUNNING the gate, never by demanding the dir; the append itself stays the mechanical closure. Map rows are not append-forever: a scenario proven model-default across cohorts (the journal's insurance-rule class), or fully subsumed by a newer scenario, MAY be retired or consolidated — grounds: proven model-default across cohorts, fully subsumed by a newer scenario, the row's primary SURFACE left the family (skill/agent removed or renamed away), or the scenario proven defective from birth — by an ANCHORED dated journal record (`^ROW RETIRED <name> <ISO date>: <ground>` — the anchor is the record, journal prose saying "retired" narratively is not), never a silent row deletion; the gate budget is owned, not merely accumulated.

## Run-artifact grammar (normative — the runners emit EXACTLY these forms)

A validator or re-implementation reads THIS block, never reverse-engineers the shell. Both runners (`run-green.sh` / `run-baseline.sh`) emit, per run dir:

- `MANIFEST.txt`, in order: `head: <full-oid|no-git>` · `dirty: <n>` · `map_sha256: <hex64>` · `cli: <version line|unknown>` · `utc: <YYYY-MM-DDTHH:MM:SSZ>` · one `scenario <name> model <model> sha256 <hex64>` line per LAUNCHED scenario (appended as scenarios enqueue) · one `result <name> exit <ec>[ timeout]` line per scenario that produced a status (appended by the closing aggregation). `<ec>` is the process exit code, or a skip class below.
- `<name>.status` per scenario, durable: `exit: <ec|skip-notmap|skip-nofile|skip-lostlock>` · optional `timeout: <N>s` · `utc_end: <YYYY-MM-DDTHH:MM:SSZ>`. ANY `exit:` value other than `0`, and any `timeout:` presence, makes the run BROKEN (nonzero runner exit) — skip classes included: a typo'd selector or a drifted map row is a recorded failure, never a silent green.
- baseline only: `NOTE-baseline-validity.txt` (this exact filename) — the diagnostic-evidence stamp; its absence from a baseline dir means the run predates 0.5.7 or the artifact grammar was violated.

A dir whose artifacts deviate from this grammar is not a sanctioned run artifact — closure appends must not cite it.
