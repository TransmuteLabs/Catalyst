# GATE OWED records (normative home)

This file is the NORMATIVE home for the tests-journal GATE OWED record machinery: the closure append's shape and coverage validity, rename re-binds, lost referents, the machine-local out/ identity, and row retirement. `SKILL.md` keeps the always-hot duty — the session-start `^GATE OWED` grep — and points here. Journal WRITES are commit-serialized: a closure append or `— now <path>` re-bind is an IN-PLACE append to the journal RE-READ immediately before the write — never a whole-file snapshot built earlier (a snapshot saved over a sibling's fresher file is the loss mechanism, and it needs no merge: two sessions in one checkout commit linearly and the second's tree silently drops the first's append) — COMMITTED immediately; right after its commit the writer re-reads the COMMITTED copy and verifies its append is present, and after any later merge it re-verifies survival again — a vanished append is re-appended (two sessions saving whole-file snapshots silently drop each other's closures: the lost append fails safe — the line looks open and forces a re-run — but the completed run's record must not evaporate). The duty is TWO-SIDED: the session that MERGES the journal (a pull's merge, a branch integration) preserves the closure and re-bind appends of BOTH parents as a line-level UNION — never ours/theirs: each parent's appends are recorded answers, the writer-side re-verify above catches only the writer's OWN append, and only the merge writer sees both parents at once — a parent's append missing from the merged result is restored in the merge commit itself. The line anchor is the data/grammar boundary, and journal WRITERS keep it honest from their side: prose that CITES the record form (a finding, a historical example) must never place `GATE OWED` at the start of a line — indent it, fence it, or quote it mid-sentence (a line-start citation is grepped as an open owed record).

**Trigger — read this file ENTIRELY, never a fragment:** an open `GATE OWED` line found at session start, a closure append about to be written, or a rename/retirement touching a line's named files.

An owed gate found runs now, and ONLY a run SCORED green is recorded by APPENDING to the owed line — `— gate run <ISO date>, <green out/ dir>, scenarios: <names scored green>`; an append that FAILS validation (partial coverage, invalid run identity, legacy-time doubt) does not close the line — the line is open by VALIDITY, not by append-presence, and the superseding closure is a SECOND append of the same shape (`— gate re-run <ISO date>, <dir>, scenarios: …`; the latest VALID append governs, earlier invalid ones stay as history) — the scenario list makes the closure's COVERAGE claim readable on every machine (checkable against map.tsv without the machine-local dir); the same append-the-answer shape as the family's other records (a red run never answers the line: it stays open through the refactor); a `GATE OWED` line without that append is mechanically open — and an append whose `scenarios:` list does NOT cover the scenarios map.tsv maps to the line's named files is INVALID: the line stays open (the session-start check verifies the list against the map the run's OWN MANIFEST names — resolve `map_sha256:` to the exact map.tsv content (the blob is reachable through the manifest's `head:` commit — `git cat-file` on `<head>:tests/pressure/map.tsv`, or any historical blob matching the hash) and check coverage against THAT map, never against "map.tsv as of the run's timestamp": a wall-clock selection is skewable (a slow clock names an older map and a subset closes over a row the run's actual head required) and internally contradicts the manifest, which already identifies the exact revision; timestamp-based selection survives ONLY for legacy manifest-less dirs, under the legacy-time doubt rule below (the out/ dir name carries the timestamp — `green-YYYYMMDD-HHMMSSZ`, minted with `date -u` from 0.5.5, with an optional `-<pid>` collision suffix from 0.5.6, and from 0.5.6 each dir carries a MANIFEST.txt binding the run to its exact revision (`head:`, `dirty:`, `map_sha256:`) — a closure whose cited dir's manifest names a DIFFERENT head than the edit's commit history reaches, or a dirty tree, does not close the line (timestamp order alone is not revision identity) — which fixes the required ordering for closure-grade runs: the RED-GREEN-REFACTOR loop legitimately runs pre-commit (its dirty-manifest runs are working evidence, not closures), but the run a closure append CITES must be a post-commit clean-tree run naming the edit's committed head — commit first, then the closure run; and a `head: no-git` manifest identifies no revision at all — it never closes a line; a legacy no-`Z` name is WALL CLOCK, zone unrecorded — when row-vs-run order within a ±1-day window would decide a closure's validity against a legacy dir, the closure is NOT excused: re-run the gate, a fresh run is cheaper than a wrong closure; the mapping the run could have known); rows added later — including later the same day — create coverage duties for future edits, never re-open a recorded closure (a recorded answer is never re-litigated) — the paired-effects principle: an append closes only when its run did what closure requires, checked by shape, never assumed). Renaming or moving a file an open owed line names re-binds the line IN THE SAME commit — append `— now <new path>` with `<new path>` BACKTICK-QUOTED in the actual record, like the line's file list (the mint rule — new lines backtick-quote each file, `` `path` `` comma-separated, so a path containing `, ` cannot split the list — is stated at SKILL.md's mint site and is normative HERE for every append this file owns; readers fall back to unquoted comma-splitting only on a legacy line or append with no backticks) (the record follows its referent); an open owed line whose named files map to ZERO scenarios in map.tsv is a LOST REFERENT — resolved with the user, never read as vacuously satisfied. The cited out/ dir is machine-local (gitignored): it is the run's identity on the machine that ran it — a later reader elsewhere re-verifies by RE-RUNNING the gate, never by demanding the dir; the append itself stays the mechanical closure. Map rows are not append-forever: a scenario proven model-default across cohorts (the journal's insurance-rule class), or fully subsumed by a newer scenario, MAY be retired or consolidated — grounds: proven model-default across cohorts, fully subsumed by a newer scenario, the row's primary SURFACE left the family (skill/agent removed or renamed away), or the scenario proven defective from birth — by an ANCHORED dated journal record (`^ROW RETIRED <name> <ISO date>: <ground>` — the anchor is the record, journal prose saying "retired" narratively is not), never a silent row deletion; the gate budget is owned, not merely accumulated.

## Run-artifact grammar (normative — the runners emit EXACTLY these forms)

A validator or re-implementation reads THIS block, never reverse-engineers the shell. Both runners (`run-green.sh` / `run-baseline.sh`) emit, per run dir:

- `MANIFEST.txt`, in order: `head: <full-oid|no-git>` · `dirty: <n>` · `map_sha256: <hex64>` · `cli: <version line|unknown>` · `utc: <YYYY-MM-DDTHH:MM:SSZ>` · interleaved per scenario as they enqueue: `file <rel> sha256 <hex64>` lines for each mapped skill/reference input not yet hashed in this run (green only — the governing inputs are content-bound like the scenario text; a symlinked or missing mapped path never launches: its scenario records `skip-badref` instead) and one `scenario <name> model <model> sha256 <hex64>` line per LAUNCHED scenario · one `result <name> exit <ec>[ timeout]` line per SELECTED scenario (closing aggregation — a selected scenario whose wrapper died without writing a status appears as `result <name> exit missing`; a status lacking its `utc_end:` tail is TORN and appears as `result <name> exit torn`). `<ec>` is the process exit code, or a skip class below. SIGNAL CARVE: a run terminated by INT/TERM exits 130/143 through its signal handler WITHOUT reaching the closing aggregation — its MANIFEST legitimately carries NO result lines; such a dir is broken by its runner exit and is never closure evidence, not a grammar violation.
- `<name>.status` per scenario, durable: `exit: <ec|skip-notmap|skip-nofile|skip-badref|skip-lostlock>` · optional `signal: terminated` (the wrapper's INT/TERM trap wrote it — the session group was killed) · optional `timeout: <N>s` · `utc_end: <YYYY-MM-DDTHH:MM:SSZ>`. ANY `exit:` value other than `0` (`missing` and `torn` in a result line included), and any `timeout:` presence, makes the run BROKEN (nonzero runner exit) — skip classes included: a typo'd selector or a drifted map row is a recorded failure, never a silent green.
- baseline only: `NOTE-baseline-validity.txt` (this exact filename) — the diagnostic-evidence stamp; its absence from a baseline dir means the run predates 0.5.7 or the artifact grammar was violated.

A dir whose artifacts deviate from this grammar is not a sanctioned run artifact — closure appends must not cite it.
