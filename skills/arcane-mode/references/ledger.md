# The ledger — identity, claims, marks, resume

Verbatim arcane-mode doctrine (0.7.x per-trigger split of verification.md; that file is now the router). Where SKILL.md's sketch differs in detail, THIS file governs.

## Ledger

`<repo>/.catalyst/sdd/progress.md` (the workspace self-ignores in git; `git clean -fdx` destroys it — recover from `git log` on the machine that has the branch; after a squash-merge integration the per-task `commits <base>..<head>` refs are historical only — recovery there uses the squash commit plus the plan/spec, not per-task hashes; a SHALLOW clone's truncated log under-reports completed tasks the same silent way: unshallow first (`git fetch --unshallow`), or treat the log as unreachable evidence).

- **The workspace is single-writer:** one arcane/review session per checkout at a time — the ledger, briefs, and diff files have fixed paths; concurrent efforts need separate worktrees. Before ARCHIVING a mismatched ledger: workspace files modified within the last few hours (`ls -lt .catalyst/sdd/`), or the user known to run parallel sessions, is a live-sibling signal — on the signal or in doubt, ask the user (archiving is destructive; one question is cheap). Resuming a MATCHING ledger needs no pre-emptive ask: append a `session: <ISO timestamp> resumed (<machine label>, pid <N>)` line on resume — the identity suffix is what makes two same-second claims distinguishable — then RE-READ the ledger before the first dispatch: a foreign session line whose ISO TIMESTAMP is at or after yours (compared as timestamps, never by line position — two machines' appends can land in either file order) means two sessions claimed together — the claimant with the strictly LATER timestamp defers, and on the SAME timestamp the byte-lex LATER claim line's writer defers (C-locale, the family tie-break) — the deferring writer routes to the live-sibling ask, the other proceeds (append alone elects no owner; both sessions previously dispatched the same task from one checkout); a foreign claim STRICTLY EARLIER than yours is not automatically dead — the live-sibling signal above applies to a matching resume too: that claim plus workspace files modified within the last few hours → the live-sibling ask before the first dispatch, never a silent proceed (an old claim with no fresh activity proceeds normally: sessions die without unclaiming); after that, any later ledger line you did not write (a foreign session line, an unexpected completion mark) means a live sibling is co-writing — stop and ask. A confirmed live sibling means THIS session moves to its own worktree (or stops); it never archives and never co-writes. When the user EXPLICITLY licenses parallel executors (the family default is sequential), the proven contract is two-tier (absorbed from a production swarm workspace, journal 0.8.3): one shared coordination repo that is never built or edited in — branches and merges only — plus one worktree per executor on a fast local path; worktrees share the object store, so commits are visible to the coordinator without any push, and gitignored artifacts (deps, env files) are copied into each worktree by a setup script, never assumed to follow `git worktree add`. Two additions from the same source: each executor's brief carries an OWNERSHIP FENCE — the explicit file list it may touch, with the shared seam files (routers, root configs, lockfiles) named as merge seams owned by the coordinator alone; and any step whose inputs are gitignored (env files, credentials) runs in the coordinator's own checkout, never delegated — an executor's worktree cannot prove what it read.
- **The ledger belongs to ONE plan.** Its first line is the plan identity: `Plan: <plan file path> (<branch>)` — the branch is the LAST parenthesized group at the END of the line, the path is everything before it VERBATIM (a legal path may itself contain parentheses; any first-paren or greedy split mis-reads it and a false identity mismatch archives a live ledger) — and the ONLY identity line: a ledger with a second identity line mid-file is corrupt (a buggy append instead of an archive) — archive it wholesale and rebuild from `git log` + the plan; completion marks in a corrupt ledger are not trusted. A REBUILT ledger has no verification/waived/uat/disposition lines, no `minor:` lines, and no `Task N: re-opened (amendment|rejection <date>)` lines — ALL five ledger-only RECORD classes: verification/waived, `uat:`, `disposition:`, `minor:`, `Task N: re-opened (…)`. (A DIFFERENT quintet governs what may sit below the last verification line — `session:`, `uat:`, `minor:`, `disposition:`, `Task N: re-opened (…)`: it includes `session:` lines and excludes verification lines because a `session:` line is a CLAIM a resuming session mints fresh, not a record a rebuild owes, while the verification line is the ANCHOR the trailers sit under, not a trailer itself.) The five record classes live ONLY in the ledger and `git log` cannot recreate them: consult the ARCHIVED file for every one of them before treating any as absent (lost `minor:` lines silently starve the branch-finish triage; lost re-opened lines silently close owed rework), and re-derive the re-open lines the plan itself records — every task the committed plan annotates `[re-opened by amendment|rejection <date>]` without a matching ledger line owes its line at the rebuild (the per-task idempotent signal — machine-independent, so a rebuild recovers what the archive alone might not show), and a rebuild never strips a roadmap stamp that records one (a `verified (waived: …)` stamp may be the user's waiver's only surviving record — it stands). It is created when the plan is APPROVED — the user's go on the written plan; the batched pre-flight question is the natural moment — before the premortem gate; plan-writing sessions correctly have no ledger yet. Execution start: read the ledger and compare its identity line against the current plan — a ledger with NO identity line at all (a pre-0.4.2 shape, or a torn creation) is not an automatic mismatch: derive the identity from content (the plan its task marks match) and REPAIR the line in place when it is this effort's — archiving live completion marks re-dispatches finished tasks; genuinely foreign or undecidable → the mismatch rule. A mismatch (new plan, new campaign phase) means the ledger is a previous effort's: archive it (`mkdir -p archive && mv progress.md archive/<plan-slug>-progress.md` — nothing creates `archive/` beforehand) and start a fresh one with the new identity line. Exception — the mismatch is merely a renamed/moved plan FILE of the same effort (same branch, same must_haves): update the identity line in place; archiving live completion marks would re-dispatch finished tasks. Completion marks are only ever trusted for the plan named in the identity line — never carried across plans.
- Same plan: FIRST the owed-line sweep, before any dispatch — every task the committed plan annotates `[re-opened by amendment|rejection <date>]` whose completion mark has no `Task N: re-opened (…)` ledger line carrying that FULL token owes its line NOW (the token, not the date — a same-day second amendment's time-suffixed token must never be satisfied by the first's line; the annotation is the machine-independent signal; the ledger line is this machine's copy — a resume that skips the sweep silently closes re-opened work); and the sweep's SECOND arm runs the amendment GATE test: any task annotated `[re-opened by amendment <token>]` in the committed plan while the governing premortem yaml's `amendments:` list lacks a member equal to that FULL token = the scoped gate re-run is OWED NOW, before any dispatch — this arm OUTRANKS the verdict-line/yaml agreement below (on un-regated amended ground both legitimately agree on the PRE-amendment verdict: their agreement proves staleness handled, not the amendment gated — the exact silent-skip window the token record exists to close); then: tasks marked complete are DONE — no re-dispatch; resume at the first unmarked task (a MARK is a whole LINE-ANCHORED record — a line BEGINNING `Task N:` or `verification:`, written by the ledger owner; the same shapes QUOTED inside prose — a pasted report excerpt, a concerns dump, a session note citing `Task 2: complete (…)` mid-line or indented — are CONTENT, never marks: every resume/sweep rule here keys on line-start records only, or a quoted historical line would falsely close or falsely re-open a task — and the SAME line-anchor rule covers `uat:`, `minor:`, and `disposition:` records: a mid-line occurrence inside prose or a pasted excerpt is content, never a filed result, a Minor, or a standing disposition answer) — EXCEPT a task with a `Task N: re-opened (amendment|rejection <date>)` line appearing LATER IN THE LEDGER than that task's completion mark: the re-open supersedes the mark (which stays as history) and the task dispatches again on the amended ground — this keys on ledger order alone and needs no verification line (mid-execution ledgers have none; post-convergence the re-open line legitimately sits below the last verification line like the other sanctioned trailers). An ABSENT ledger at resume of an APPROVED, committed plan is a fresh machine's normal state, not a defect: CREATE it now — identity line from the committed plan's path and the current branch — then run the arms below against the new (empty) ledger; creation at approval time (the rule above) covers the machine that approved, this arm covers every other machine. A ledger with no premortem verdict line is FIRST a missing cache, not automatically an owed gate: check for `<plan>.premortem.yaml` — an existing PASS/WARN yaml (a fresh machine's new ledger simply never wrote its cache line; the committed yaml IS the verdict) REPAIRS the ledger: write the verdict line from the yaml and dispatch, no re-run (the line is a CACHE in every direction — an absent cache rebuilds from its source, the same move as the router's stampless-row repair; a re-run here could mint a fresh BLOCK against the family's own recorded verdict on unchanged ground); an existing BLOCK yaml: the mitigations may or may not have landed — check the plan for them FIRST, then re-run the gate on the current plan (don't re-add tasks that already landed); NO yaml at all → the gate is owed: run it, and only then dispatch. A ledger whose verdict line IS BLOCK means the gate failed, not that it ran: mitigation work or the user's acceptance decision is owed — Task 1 stays barred until a PASS/WARN verdict replaces it. The verdict line is a CACHE of `<plan>.premortem.yaml` in every direction: before dispatching on a WARN/PASS line, cross-check the yaml — on any mismatch the yaml wins (a fresh BLOCK yaml under a stale WARN line = the gate failed; the line just never caught up); a MISSING yaml under an existing verdict line is first a LOST REFERENT, not an owed gate: the yaml is committed with the plan (plan-standard's rule) — recover it from git history and CROSS-CHECK it against the line like any cache (a recovered yaml can predate the line — the commit rule has crash windows, so "recovered" is not "fresh"): a match confirms the line; a mismatch the line's own named acceptances EXPLAIN (a recovered BLOCK naming exactly the tiger the line records as user-accepted — the crash fell between the acceptance and the yaml commit) → the repair-from-line arm requires CORROBORATION first: the ledger is a self-ignored, machine-local CACHE (a buggy or defective writer can mint a `WARN — tiger X accepted by user` line no user ever authorized, and repairing from it alone would promote a forgeable cache into the authoritative acceptance record) — repair the yaml FROM the line only when a COMMITTED record corroborates the acceptance (a campaign D-log decision naming it, the spec's open/deferred record, or acceptance evidence in the plan/spec commit history); uncorroborated → the acceptance is UNATTESTED: re-confirm with the user in one line naming the found ledger line (never a silent re-block, never a silent promotion); corroborated → apply the named acceptance (tiger → `accepted_risks`, premortem's recorded shapes), recompute the verdict, commit — a repair, never a re-ask and never a flat blessing; a mismatch the acceptances do NOT explain (different tigers, an unexplained verdict) → neither side is trusted: the gate re-runs, reading the recovered yaml first and honoring every line-named acceptance. "Genuinely never committed" is concluded only on FULL history (a pre-referent-era effort; unshallow a shallow clone first — absence in a truncated log proves nothing) → the gate is owed and re-runs, and the re-run honors the acceptances the ledger line names under the SAME corroboration rule as the repair arm above (a committed record must back them; an uncorroborated cache-only acceptance is re-confirmed with the user in one line, never silently honored and never silently re-blocked): a genuinely recorded acceptance is never re-asked. No verdict line anywhere AND no yaml = the gate never ran — owed, plainly.
- After a task's clean review — one line: `Task N: complete (commits <base>..<head>, review clean)` — `<base>`/`<head>` are git's full-uniqueness abbreviations (`git rev-parse --short=12`, the campaign's `<sha>` rule: a fixed 7-char prefix is unique only against the object database at write time, and a later fetch can make a stored range unresolvable; a legacy 7-char ref that HAS become ambiguous is a recovery case — resolve among candidates by the task's files/dates, still ambiguous → the user); Minor findings — same file, prefixed `minor:`. Fresh-eyes convergence writes its own line: `verification: converged (rounds N..M clean)` — the durable evidence the campaign router's reconciliation reads. The VERIFICATION line — that one, or its sanctioned waived substitute — is the ledger's final line MODULO five sanctioned residents: `session: <ISO> resumed (…)` lines (the claim-identity suffix is part of the shape), `uat:` result lines, `minor:` lines (a post-convergence fix wave's critic review files its Minors here too), the `disposition:` answer trailer, and `Task N: re-opened (amendment|rejection <date>)` lines may legitimately follow it — the first four don't contradict it, and a re-opened line is a sanctioned STATE SIGNAL: it is a TASK line, so the CONVERGED/WAIVED rules below read it as exactly the re-opened-work trigger it is, never as corruption. Every below-the-line rule keys on the LAST verification line, never an earlier one (a superseded line with TASK lines under it is history, not a trigger). TASK lines below the last verification line when it is CONVERGED mean convergence was RE-OPENED (a post-convergence fix wave) and the refresh never landed: the converged line is STALE, not corrupt — resume the fix wave/rounds and refresh it; never treat a stale one as current. A refresh, like every verification-line update, is APPENDED at the end — the superseded line stays above as history, never deleted or edited in place. TASK lines below the last verification line when it is WAIVED mean work landed after the waiver — the waiver is state-specific: route to the re-open rule's disposition ask (a later `disposition:` trailer means it is already answered — resume, don't re-ask), never read the waiver as covering those tasks — EXCEPT `Task N: re-opened` trailers: there the work is OWED, not landed — dispatch the re-opened task(s) FIRST; the disposition ask follows the fix/amendment wave (its own placement rule), never precedes it. Sanctioned successions, each appended AT THE END: a converged line following a converged line (the refresh after a re-open), a converged line following a waived line (the supersede), a fresh waived line following an old one (the extend arm), a waived line following a converged line (a re-opened convergence whose re-run the user waived — the review-floor rule applies to re-runs too); the superseded line stays as history, not a contradiction. Anything else after the last verification line (a foreign session's marks, a second converged line with no re-open between) is a contradiction — surface it, never trust it.
- The ledger stays current after every task — it is the recovery map after compaction; after compaction trust the ledger and `git log`, not conversation memory.
- **The ledger records status and coordinates only — never task content.** Steps, specs, and acceptance criteria live in the plan and briefs (the single authoritative source); a ledger that starts restating them forks the truth, and the copies drift silently (absorbed evidence, journal 0.8.5: a production QA tracker that duplicated spec content measured 3% id-consistency against its own spec docs). A mark references the plan's task by number; it never paraphrases the task.
