# Entry (spec markers) and branch finish — the normative deep layer

Moved VERBATIM from SKILL.md (v0.6.3) in the 0.7.0 distillation. SKILL.md keeps the executable core; where its sketch and this file differ in detail, THIS file governs.

**Read §1 ENTIRELY** when the spec's marker is anything but a plain top-of-file `status: approved` (waived/pending shapes, markerless files, two candidate specs, the repo gate), and **§2 ENTIRELY** at every branch finish (convergence reached, integrate question due).

## 1. Inputs — the full spec-marker grammar and repo gate

**Inputs:** no approved spec → catalyst:crucible first (decision hardening → spec; approved = the spec file carries crucible's `status: approved` marker AS ITS TOP-OF-FILE LINE (before the first heading — a status:-shaped line in the body is content, never the marker), including its skip-order shape `status: approved (waived: <date>)`; the pending shape `status: approved (edits pending re-approval: <date>)` — keyed on the clause: the waived combo `status: approved (waived: <date>; edits pending re-approval: <date>)` is pending too — is approved for routing, but its owed fast re-approval runs NOW for every still-pending entry — per entry: an entry already dated re-approved is approved ground; approve → the entry dates re-approved and the lineage's own marker (plain, or waived — crucible's RESTORE rule) returns when none remain — and an AMENDMENT-minted entry (the committed plan carries its dated `[re-opened by amendment <date>]` annotations; with NO plan yet the entry ITSELF is the signal — amendment.md's plan-less arm: the entry standing while `<spec>.premortem.yaml`'s `amendments:` lacks the full token = the scoped SPEC-gate re-run owed now) on approve also completes amendment.md's owed parts: the scoped gate re-run (owed while the governing premortem yaml's `amendments:` list lacks THIS amendment's FULL token — the exact-match test, amendment.md's rule; a date comparison is not executable) and the ledger's re-open lines — marker restore alone silently closes the reconcile; reject → premortem's revert arm (crucible's `references/marker-machine.md` defines all of this); a markerless file is a draft; a markerless spec the effort's history says WAS approved (a pre-marker effort resumed after an upgrade) is NAMED to the user with a fast re-approval offered — never silently re-interviewed; TWO approved specs for the same effort → the user picks, never the newest silently — under a campaign the ROADMAP row's spec link names THE spec). Effort larger than one session with a foggy route → catalyst:starchart (decision map), then crucible → spec → here. And a git REPOSITORY WITH A WORK TREE — the executable test: `git rev-parse --show-toplevel` from the approved SPEC's directory succeeds (anchored to the spec like task-brief anchors to the plan — a nested repo or submodule gates against the tree that actually holds the effort; a bare repo has no work tree and fails the test rightly: per-task commits, BASE recording, the review package script, ledger recovery all need the work tree); this pipeline has no repo-less mode (bootup's rule). Failing the test → STOP and name the gap to the user — bootup's git-init arm closes it (init + guarded initial commit, user consent), or the user's explicit decline means the pipeline does not run — never a silent repo-less degradation. A campaign's standing EPHEMERAL sanction does not waive this gate: it covers campaign/starchart state, not execution — the first phase routing into execution re-raises the gate here (git-init now, or the phase does not execute; named to the user, never silently absorbed by the earlier sanction).

## 2. Branch finish — full protocol

4. **Branch finish** — after convergence, never stop silently and never integrate silently: if the base branch advanced past the branch's last verified merge point (the fork, or a previous re-gate's merged head whose suite AND tail gates — the project's own non-test checks: lint, typecheck/build, formatter, as the repo's config/readiness docs name them; the term as used family-wide — ran green — a head whose re-gate ran red is not a verified point; "since fork" alone re-fires forever after any advance) with advances that can affect the BUILD — advances consisting solely of the family's bookkeeping commits do not re-fire the re-gate (campaign's carve-out; check `git diff --name-only <verified point>...base` — three dots, merge-base semantics) — merge it forward into the branch (or rebase per repo convention) and re-run the suite + tail gates on the MERGED head — green-on-branch proves nothing about the merge result, and a red here is a finding for a fix wave, not a reason to integrate anyway — and any post-convergence fix RE-OPENS convergence: re-enter fresh-eyes rounds until 2 consecutive clean and refresh the converged line — on a WAIVED lineage, the disposition ask instead: rounds now, or a fresh recorded waiver (`references/verification.md`) — a green suite after the fix is not that criterion; then verify the suite AND the tail gates are green on the final HEAD, triage any ledger `minor:` lines filed after the final review (post-convergence critics file here and nothing else consumes them), and present ONE question — merge into the base branch locally / push and open a PR / keep the branch as-is / discard — and execute the choice. Cleanup (worktree, temp branches) only after the choice lands — and under a campaign never before the row's repo-file referents (spec, plan, premortem yamls) are committed on base (campaign's transport rule): a `discarded` branch may hold their only copies. **Under a campaign, control returns to the campaign router at convergence:** the moved-base re-gate and the phase acceptance (campaign's `verified` row, in that order) happen BEFORE this question — acceptance decides whether the branch integrates at all, and branch finish then executes the choice without repeating the re-gate (unless base advanced again since).
