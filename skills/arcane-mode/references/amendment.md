# The user-ordered amendment route (normative home)

This file is the NORMATIVE home for arcane-mode's spec-amendment route: what happens when the user orders a change to an ALREADY-APPROVED spec — any time, including mid-execution and after a phase is `verified`/`done`. Entry and marker shapes are normative in crucible's `references/marker-machine.md`; the revert arms in premortem's `references/spec-gate.md`; the roadmap demotion shapes and transport in campaign's rules. `SKILL.md` points here from its Inputs rule.

**Trigger — read this file ENTIRELY, never a fragment:** the user orders an amendment to an approved spec. A partial read improvises exactly the steps this route exists to record.

## The route

A user-ordered spec AMENDMENT after approval — any time, including mid-execution — is routed, never improvised: crucible's fast path scoped to the change hardens it (the decision is the user's, recorded section-by-section or by explicit order), the amendment session writes the pending marker shape with the same commit-first mechanics (crucible's `references/marker-machine.md`; under a campaign the edits + marker are never-defer transport — committed to base and pushed NOW, campaign's class, or every other machine consumes the stale approved spec) — and that FIRST commit CARRIES the plan annotations and (under a campaign) the amendment's D-log event IN THE SAME COMMIT, exactly like the post-done arm binds its D-line to the demotion and the rejection arm binds its annotations to the answer: written as separate later commits, a death right after the marker push leaves a pending shape byte-indistinguishable from an ordinary ground edit — the resumed fast re-approval restores the marker on approve while the reconcile, the scoped re-gate, and the D-line silently never happen (the ledger sweep keys on the annotations: absent, it provably "finds nothing to owe") — the crash-binding is what makes the amendment's owed work machine-independent; and the scoped gate re-run is owed DETECTABLY: annotated tasks standing while `<plan>.premortem.yaml`'s verdict predates the amendment's date-token = the re-run is owed at resume, before any dispatch — the gate the effort actually has re-runs scoped to the amendment (a spec-gate lineage re-runs the SPEC gate; a plan-gate-only effort re-runs the PLAN gate on the reconciled plan — no first spec-gate yaml is invented; premortem's season rule bars retroactive analysis of shipped work, not re-gating amended ground; the plan-gate bridge then refreshes `<plan>.premortem.yaml` and the ledger verdict line, and — under a campaign — the `planned` stamp refreshes with the re-run's verdict date, the `specced` stamp's referent already being the amended file), the plan reconciles — the reconcile ANNOTATES each invalidated task in the COMMITTED plan file (`[re-opened by amendment <date>]` on the task heading — the durable, machine-independent record of the invalidated set, carried by the amendment's transport) and each such task re-opens as a `Task N: re-opened (amendment <date>)` line appended at the ledger's END (the old completion mark stays history; the resume rule reads the LATER re-open line, not the mark; the LEDGER parts belong to the machine that has the ledger — an amendment hardened elsewhere completes every base-visible part (spec edits + marker, gate re-run, D-log, AND the plan annotations — the committed plan's `[re-opened by amendment <date>]` marks, the very signal the ledger machine's sweep keys on: skip them and the sweep finds nothing to owe), and the ledger machine writes the owed re-open lines at RESUME, before any dispatch — the signal is PER TASK and idempotent: every task the committed plan annotates `[re-opened by amendment <date>]` whose completion mark has no `Task N: re-opened (amendment <date>)` line carrying THAT amendment's date-token owes its line (the family same-day arm applies to these records: a SECOND same-day amendment appends the time to BOTH its plan annotation and its ledger lines — `[re-opened by amendment <date> <HH:MM>]` (two amendments within one MINUTE — rare but real in scripted sessions — extend to `<HH:MM:SS>`; still colliding, append the answer commit's short OID: the token must be unique, a shared token makes the first `Task N: re-opened` line falsely satisfy the second amendment's sweep) — and the sweep matches the FULL token, so one day's second amendment never reads as already-swept) (evaluable on any ledger — no tail dates needed — and a death mid-append leaves the remaining owed lines still detectable)) — and under a campaign the change is Decision-logged (scope edits are never silent). 

On a phase whose roadmap row stands in campaign's `unresolved (between: …)` shape, consult the amendment ORDER's text: an order that NAMES a side (quotes or otherwise discriminates one side's status/stamp) is the pair's ANSWER — record it first (campaign's class-5 rewrite-plus-D-line commit), then proceed on the chosen row; an order referencing only the phase or its spec discriminates NEITHER side — fire the standing ask first and proceed on the answered row: never mint an answer from a non-discriminating order, never demote or amend an unanswered pair. (Stated here for the amendment writer; the normative home is campaign arbitration.md's unresolved-row cell — on any divergence that cell governs.) On a phase PARKED in the roadmap (preserved status inside `## Parked`): parked rows are never routed and un-parking is the user's own protocol — so an amendment order on a parked phase's spec FIRST surfaces the parked state: amend-and-leave-parked (the wave lands, the pending marker stands recorded, the row stays parked — the ask will fire when un-parked) or un-park first (campaign's protocol), the USER'S choice; never demote a parked row in place and never improvise an un-park. 

On a phase already `verified`/`done`, the amendment FIRST demotes the row — it voids the acceptance exactly like a re-gate red (the `reopened` demotion shape, never-defer commit+push; the demotion EXTENDS the was-clause with the row's executed `done`/`branch:` parts — campaign's schema — so the integration record survives the rewrite; the demotion commit ITSELF carries the amendment's D-log event — ONE commit, so the router's amendment trigger is never separated from the void: a death right after this push still routes every machine to the amendment wave, and a committed demotion carrying the amendment D-line is never mistaken for a hand demotion; a closed milestone re-opens through the existing non-done-row rule) — so a died amendment is visible to every router run, and the demoted row WITH the spec's pending marker (and the amendment's D-line) routes to the amendment wave, never to the fix-wave resume: on an integrated phase the machine-with-branch requirement does not apply — nothing is unrecoverable. 

The post-`done` amendment wave is a NEW plan scoped to the amendment (the old plan's tasks are history in git and the merged code — nothing re-dispatches them; goal-backward against the amended must_haves decides the work, including reworking shipped behavior; the fresh ledger carries the new plan's identity), run on a fresh branch/worktree off the row's INTEGRATED head — `branch: merged` → off base; `kept` or an un-merged `pr` → off the kept branch's head (base lacks that phase's code); the `branch:` part is read from the demoted stamp's was-clause, and a was-clause with NO `branch:` part (a demoted `verified` row, or a `done` row amended inside the acceptance window before the choice executed) is FIRST verified against git (campaign's executed-and-unrecorded arm, incl. its ancestry caveats — a squash-merge leaves the phase head a non-ancestor: check the integration history before concluding nothing integrated; unshallow a shallow clone first): for a demoted `done` row an integration git shows is the executed choice recovered — recorded into the was-clause, the integrated arm applies; for a demoted `verified` row (never accepted) an integration found is integration-WITHOUT-acceptance — surfaced to the user, never silently consumed; genuinely nothing integrated → the wave runs off the still-live phase branch's head and the machine-with-branch requirement APPLIES — the no-machine-requirement carve is only for integrated phases — and re-verification, the full UAT re-offer, re-acceptance, and a fresh branch-finish question follow the demotion's normal path.
