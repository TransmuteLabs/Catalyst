# Verification phase — goal-backward, UAT, final review, convergence

Verbatim arcane-mode doctrine (0.7.x per-trigger split of verification.md; that file is now the router). Where SKILL.md's sketch differs in detail, THIS file governs.

## Goal-backward: verify the goal, not the tasks

All tasks done ≠ goal achieved. Against the plan's `must_haves`:

- **truths** — every observable behavior demonstrated (command + output), not inferred from "the code was written".
- **artifacts** — every file exists and is non-empty.
- **key_links** — every link actually laid: X calls Y (not "both exist"), data from Z reaches its consumer.

**Stub scan** over all created/modified files: hardcoded empty values (`=[]`, `={}`, `=null`) flowing into render/output; placeholder texts ("not available", "coming soon", TODO/FIXME); components with no data source wired. Every found stub is either wired up or explicitly accepted by the user as intentional. A silent stub = verification failure.

**Grounding for doc/synthesis claims:** every claim is checked against the deepest source (code, not a retelling); separately check the connectives and quantifiers added during compression ("when", "always", "after", "therefore", "most") — distortions are born in the connective tissue absent from the source. Derived-vs-derived checks don't count.

**Performance claims (before/after measurement discipline):** a before/after perf comparison is confounded by drift when the two arms ran in different environments (warm vs cold cache, background load, thermal state, data growth) — INTERLEAVE the runs (A B A B, never all-A then all-B) so drift hits both arms, pin what can be pinned (seed, dataset, machine), and report the spread, not one number. Directional confounding is the one legitimate shortcut: when every identified bias works AGAINST the claimed effect (the new build measured on the loaded machine, the old on the idle one), the true effect is at least the observed one — state that direction explicitly instead of pretending no bias exists; a bias whose direction is unknown grants no shortcut.

**Machine re-verification of a reported number (never trust the implementer's figure):** when a task or its report states a quantitative outcome — line count dropped 412→354, bundle 2.41 MB→2.39 MB, coverage 78%→85%, N tests now pass — the number is re-derived from the committed artifact, not accepted from the report. Check out the effort's actual HEAD (or the task's commit), RE-RUN the exact command that produces the metric, and compare: numeric within a small tolerance (a ~10% band absorbs measurement noise; a claim outside it is a finding) AND the **unit must match** — 2.39 KB is not 2.39 MB, and a comparison that ignores the suffix passes a 1000× regression. A metric with no runnable command behind it is not a verified metric; either wire a command or drop the claim. This is the goal-backward `truths` rule applied to numbers: "the report says 354" is inferred, "I re-ran `wc -l` on HEAD and got 354" is demonstrated. It generalizes the perf discipline above to every self-reported figure, and it is the concrete guard against a vacuous-green report whose numbers were typed, not measured.

## Conversational UAT (user-facing behavior)

Code-side gates prove the code matches the plan; they do not prove the feature works from the user's seat. After goal-backward passes, for user-visible behavior offer a UAT pass: one scenario at a time ("do X — what do you see?"), the user answers in plain text, each result recorded (`uat:` lines in the ledger, exact shape `uat: <scenario> — pass|fail[ — <note>]`, line-anchored like every ledger mark; when the pass runs on a machine WITHOUT the phase ledger — campaign's cross-machine acceptance, the ask runs wherever the user is — the results travel as a `uat receipt` D-line (campaign arbitration's grammar — its receipt norm also holds the segment inner grammar and the backtick-free write-gate): the session FETCHES first and looks for a STANDING `uat receipt` for this P<N> whose lineage token is current — found → APPEND to it, never a second mint (two mints from machines that could not see each other are healed at merge by arbitration's segment-union cell); none → the FIRST answer MINTS `uat receipt: P<N> <lineage token> — `<scenario>: pass|fail[ — <note>]` (<machine label>, <date>)`, every later answer APPENDS its own backtick-quoted segment to that line as a sanctioned post-mint append (DECISION CORE excluded — arbitration's LOCATE rule), each append committed never-defer (a user answer must not die uncommitted) — an interrupted pass needs no special form, the line simply stops growing; the ledger machine at resume COPIES each segment as a `uat:` ledger line, presence-based per segment so re-adoption is idempotent, and ONLY while the receipt's lineage token equals the row's CURRENT token — a voided acceptance changed the token, its old receipt is inert, never re-imported under the re-offered pass; the copy is what keeps resume-from-first-unrecorded true across machines). Under a campaign this offer is DEFERRED to the phase acceptance (the campaign's `verified` route) — it runs once, there, not inside this step. A reported gap is a finding like any other: diagnose, fix through the pipeline (fix wave + critic), re-test that scenario. The recorded `uat:` lines make an interrupted pass RESUMABLE: on re-entry, resume from the first unrecorded scenario — "it runs once" bars repeating completed scenarios, never finishing an interrupted pass; a VOIDED acceptance (post-demotion) voids its recorded pass wholesale — the re-run acceptance re-offers every scenario, because the fix changed the behavior the old lines recorded. Never interrogate with a batch of ten questions, and never mark UAT passed on the user's silence.

**Any fix landed after convergence (a UAT gap, a moved-base re-gate red) re-opens convergence:** re-run the tail gates on the new HEAD and re-enter fresh-eyes rounds scoped to the fix diff until the stop criterion holds again (2 consecutive clean — one clean round re-declares nothing), then refresh the `verification: converged` line. On a WAIVED lineage (the effort's verification evidence is the waived line, not a converged one): the recorded waiver covers the code state it was written at — after the fix wave, put the disposition to the user ONCE: run the rounds now (the converged line supersedes the waiver) or extend the waiver (a fresh `verification: waived by user <date> — <scope>` line); neither arm re-demands the original waiver, and neither stretches it silently over unreviewed code. The ANSWER is recorded at the moment it is given, so no crash re-asks it: "extend" → the fresh waived line lands NOW, appended at the end; "rounds" → a `disposition: rounds by user <date>` trailer lands NOW and the refreshed converged line absorbs it when the rounds finish — a ledger whose last verification line is waived but which carries a later `disposition:` trailer means the ask is ANSWERED: resume the rounds, never re-ask. Acceptance evidence must reference the FINAL head, never a pre-fix one — under a campaign, a re-open AFTER the `done` flip voids the acceptance: the row demotes IMMEDIATELY at the red (campaign's `reopened` stamp shape, committed — and pushed when a remote exists, campaign's never-defer push rule — so every machine sees the void, the voiding named to the user), re-convergence rewrites the stamp to the refreshed rounds, and acceptance re-runs on the final head (campaign's rule).

## Final whole-branch review

The orchestrator reads the branch review package PERSONALLY (`<skill-dir>/scripts/review-package MERGE_BASE HEAD` — resolved against the arcane-mode skill's directory, never the project cwd; REQUIRE a clean tree first — `git status --porcelain -uall` empty (workspace/ledger paths self-ignored aside): a dirty tree means the tests and reads ran on state the package does NOT contain, and the verdict would certify a tree nobody can reproduce; MERGE_BASE = `git merge-base <base-ref> HEAD` where `<base-ref>` is the branch this effort forked from — never assume `main`). Exception — a package >~150KB: a fresh-eyes subagent on the same top tier with clean context; the orchestrator adjudicates its verdict. The generated package's SHAPE (norm parity with the script): a `# Review package: <full-OID>..<full-OID>` title line, then `## Commits`, `## Files changed`, `## Diff` sections, closed by the terminal marker — a package whose title carries short/absent OIDs or whose sections differ was not produced by the sanctioned generator: regenerate rather than review it (marker-presence alone does not certify shape). COMPLETE-READ DUTY (both arms, orchestrator and subagent): the package ends with the generator's terminal marker (`## End of package (complete)`) — the reader must OBSERVE that marker; a file-read that returns without it was TRUNCATED by the read tool, not finished (the file on disk is complete — the read wasn't): continue reading from the cut in bounded chunks until the marker is seen, and a verdict issued without having observed the marker certifies code that was never read — fail closed, never assume the tail was quiet. The final review receives the ledger's Minor list and triages what must be fixed before merge. Final findings → one fix wave with the complete list → re-review.

## Fresh-eyes convergence (after the final review, iterative)

Rounds of 1-2 auditors with lenses the previous reviewers did NOT have. Lens tiering:

- probing pass over the new mechanisms themselves (what breaks them from inside) → top tier, clean context;
- artifacts / claim-truth / conformance to the spec text → "standard" tier (directed verification).

Rules:
- Every finding of any severity is fixed immediately (completely, never legitimized), one wave per round's list.
- Every new round — fresh context + a NEW lens set; do not repeat exhausted ones.
- Plus the orchestrator's own sweeps: greps by defect class, SHA/doc claim checks, tail gates on the final HEAD. Sweep findings count as the current round's findings — a clean round means zero from auditors AND sweeps, and any finding resets the consecutive counter.
- **Stop criterion: minimum 2 consecutive rounds with zero findings of any severity.** One clean round ≠ convergence. When the criterion is met, immediately write the ledger line `verification: converged (rounds N..M clean)` — the durable convergence evidence — before proceeding to branch finish. The user may EXPLICITLY waive remaining rounds (the economics review-floor rule: offered with the cost stated, never self-initiated, casual deadline pressure is not a waiver): then write `verification: waived by user <date> — <what was dropped>` INSTEAD — the converged line's sanctioned substitute; every consumer treats it as the effort's verification evidence without pretending the rounds ran, and no resumed session re-demands what the user waived. The waiver is STATE-SPECIFIC: it covers the code state at its date — a fix landed after it routes through the re-open rule's disposition ask (above), never rides the old waiver. A later genuine convergence SUPERSEDES the waiver: the converged line becomes the effort's verification evidence (the waived line stays as history), and a campaign stamp upgrades to `verified (converged: rounds N..M)`. TOKEN INVARIANT — lineage tokens are unique and monotonic within an effort, because token equality is the campaign arbitration's voiding signature: round numbers CONTINUE across refreshes (a resumed session reads the last round number from the ledger/stamp, never restarts at 1), and an extend waiver's token must be strictly later than every earlier token — a same-day extend appends the time (`waived 2026-07-17 14:30`), and the writer CHECKS actual uniqueness before recording: a token equal at minute grain escalates to seconds (`14:30:07`); a collision surviving second grain is surfaced to the user as a repair, never self-ranked (token equality is the voiding signature; two extends must never mint one token — and no commit-OID rung exists here: the ledger self-ignores in git, so a waiver record never has a recording commit, and a writer's own commit does not exist at write time; amendment tokens alone carry an OID rung because their ANSWER commit pre-exists the writes citing it).
