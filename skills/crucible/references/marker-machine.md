# The post-approval marker machine (normative home)

This file is the NORMATIVE home for everything that happens to an approved spec's `status:` marker and its `pending re-approval` entries after approval: edit waves and their answers, stacked and entangled reverts, the RESTORE rule, the hand-edit/git-log accounting test with its incremental anchor, double answers and arbitration, retitles and rename chains, lost referents. `SKILL.md` holds the happy path (interview → section-by-section approval → `status: approved` → commit) and points here.

**Trigger — read this file ENTIRELY, never a fragment:** any post-approval edit to an approved spec, any answer to a pending entry, and any marker or entry anomaly (an unanswered entry under a non-pending marker, a post-approval commit no record accounts for, conflicting answers, an entry naming a section the spec no longer contains). The shapes below are load-bearing; a partial read resolves anomalies wrongly. Cross-references to premortem's arms (ground-edit, stacked-revert, entangled, rejection — its `references/spec-gate.md`) and campaign's cells (arbitration asks, never-defer answers, recovery arms — its `references/arbitration.md`) are normative in THOSE skills — this file names where they plug in.

## The pending marker and edit waves

A third sanctioned shape, `status: approved (edits pending re-approval: <date>)`, marks post-approval edits awaiting the user — on a WAIVED-approval lineage the rewrite PRESERVES the waived token, `status: approved (waived: <date>; edits pending re-approval: <ISO date>)`: consumers key on the `edits pending re-approval` clause for pending-ness and on `waived:` for lineage — (premortem's spec-gate rule writes it; so does arcane's user-ordered amendment route): each edit wave adds an open/deferred entry of the EXACT shape `pending re-approval <ISO date>: <sections>` (multiple section names `;`-separated — titles legitimately contain commas, so comma is never the list delimiter; a listed name matching no section heading is FIRST re-checked as a mis-split against the spec's actual headings before any lost-referent conclusion — the grammar for every section list in entries and resolution records), and the user's answer APPENDS to it — `— re-approved <ISO date>`, or `— rejected <ISO date>: reverted` (the edit reverts — on STACKED entries only its OWN edit: sections carrying other still-pending entries' edits keep them, each surviving edit restored from its own committed history, premortem's stacked-revert arm — a surviving edit that does not COMPOSE with the removal (it edited text the rejected entry introduced) is surfaced to the user, premortem's entangled arm, never an improvised merge — the given rejection still records AT THE MOMENT as `— rejected <ISO date>: revert entangled with <surviving entry's date>; disposition owed` (a durable pending shape: body unreverted, sections pending) until the disposition's closing append lands the actual revert; premortem's rejection arm recomputes the spec-gate yaml).

## Per-entry consumption and the RESTORE rule

Consumption is PER ENTRY: sections of an entry dated re-approved are approved ground immediately; only still-pending entries' sections stay barred; the marker that returns is the LINEAGE'S OWN shape — plain `status: approved`, or `status: approved (waived: <date>)` on a waived lineage: a restore never upgrades a waived approval to a plain one — in the commit that dates the LAST pending entry (the RESTORE rule — either answer counts).

## Marker cross-check: the hand-edit test and double answers

Every consumer of a marker WITHOUT the pending clause — plain OR waived — cross-checks the entries: a `pending re-approval` entry with no answer appended under such a marker is the hand-edit/bad-merge shape — treat the spec as pending (run the owed ask), never as approved; a spec-body change committed AFTER approval that no entry ACCOUNTS FOR is the hand-edit shape too — entries account by SHAPE, answered or not: the sanctioned post-approval spec commits are an entry's edit wave, its rejection's revert, and an arbitrated restore (each landing WITH its marker/entry write in the same commit), the in-spec spec-gate VERDICT record (premortem's write — it lands in open/deferred, never in approved-ground sections), a RECOVERY repair recommitting the file (campaign's lost-referent arms) — exempt ONLY when its content byte-matches the file's LAST committed state ON THE REF BEING CONSUMED or on the recovery arm's NAMED source (campaign's arms name where they recover from — the phase branch, base history; following renames — `git log --follow`; recovery restores what was lost, never edits: content matching NO committed state, matching only an OLDER superseded state, or matching only an unrelated stale ref no arm names — a rollback that would silently revert later answered ground and delete its answer records — is a hand edit regardless of what its diff or markers look like), and a MERGE commit resolving spec-body conflicts (campaign's surfaced ordinary-git-conflict arm — which WRITES a dated resolution note into open/deferred naming the resolved sections; a merge is exempt only for hunks matching one of its parents, and an approved-ground hunk matching NEITHER parent is accounted for only by a resolution record NAMING that hunk's section — recordless, or covered only by a note naming other sections, it is the smuggle shape, treated as a hand edit) — so the test is the spec's `git log`, and accounting is PER CHANGE, not per commit — a write beside a change accounts only for what it names (an edit-wave entry for the sections it lists, a rename note ONLY for the heading rename it records — a body-text change riding beside a mere rename note is unaccounted, the hand-edit shape): a post-approval NON-MERGE commit touching APPROVED-GROUND section text with no marker/entry/verdict write ACCOUNTING FOR THAT CHANGE beside it, or a merge hunk matching neither parent with no resolution record naming its section — treat the changed sections as pending (the owed ask), never as approved ground — and the test is INCREMENTAL: a consumer that already classified this spec BLOB (same SHA) re-classifies nothing, otherwise it classifies only commits since the newest dated in-file record (every sanctioned post-approval class lands a dated record in the file — the file anchors its own audited frontier), never the whole post-approval history on every read; an entry carrying TWO CONFLICTING answers (`— re-approved` AND `— rejected` — a merge of two machines' answers) is the double-answer shape: treat as pending too — the re-fired ask is the arbitration question (campaign's cell: two recorded user decisions conflict → the user; the sections meanwhile read as the reverted approved text), never consume either side; the arbitration answer APPENDS a third, dated record to the same entry — `— arbitrated <ISO date>: re-approved` or `— arbitrated <ISO date>: rejected (reverted)`, a same-day arbitration appending the time (the family's same-day arm) — recorded and committed at the moment given like every entry answer (premortem's one-commit-per-answer; under a campaign the never-defer answer class).

An entry whose arbitrated append is STRICTLY later than every other arbitrated append READS as that answer everywhere: an arbitrated `re-approved` also RESTORES the edited text to the spec body (the merge held the reverted text — recover the edit from its own committed history; the restore commit dates the answer), a rejected arbitration runs the revert arm, it counts for the RESTORE rule, and append-only merge unions preserve it so the conflict never re-fires; a strictly-later arbitrated append (only the user's answers mint them) supersedes an earlier one — records arbitrating records.

TWO conflicting arbitrated appends with NO strict order (equal dates — unless EVERY append carries a time and the times differ; a timed append never auto-beats an untimed same-date one) are themselves the double-answer shape: pending again, → the user, and the RESOLVING arbitration appends with the marker `(resolves both)` — `— arbitrated <ISO date> <time>: <answer> (resolves both)` — which supersedes ALL earlier arbitrated appends by its kind, not by order (an independent machine arbitration never carries the marker, so the auto-beat ban is untouched; two conflicting `(resolves both)` appends escalate the same way, recursively) — never auto-picked.

## Retitles, rename chains, lost referents

A section RETITLE is itself an approved-ground edit — the heading is text the user approved: it rides an edit wave (the marker rewrites to pending, an entry names the retitle, the user re-approves — premortem's ground-edit rule; a meaning-shifting rename never slips through as bookkeeping) — and ADDITIONALLY APPENDS a dated rename note to every OPEN entry naming it IN THE SAME commit — `— section renamed: <old> → <new>, <ISO date>` — append-only, so merge unions preserve it like any answer append and the entry's identity (its dated header) never mutates; consumers read an entry's sections THROUGH its rename chain; ANSWERED entries are historical records — left untouched, and old names inside them are never lost referents; an OPEN entry naming a section the spec no longer contains, with no rename note accounting for it, is a LOST REFERENT: treat the whole spec as pending until the user resolves that entry — never dropped silently, never auto-matched to a lookalike section; a chain link whose old/new split is ambiguous (titles may contain ` → ` or commas) re-checks candidates against the spec's actual headings at that commit — the heading list is the oracle, the `;` grammar's own re-check.
