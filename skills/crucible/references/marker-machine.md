# The post-approval marker machine (normative home)

THE marker is a POSITION, not a grep hit: it is the `status:` line at the TOP of the file — before the first markdown heading (crucible writes it there; nothing else legitimately precedes the title region). A `status:`-shaped line ANYWHERE ELSE in the file — a body example, a decisions note, a quoted shape in open/deferred — is CONTENT, never the marker: every consumer keyed on "the file carries the marker" reads the top-of-file line ONLY (a presence-grep over the whole file previously let a body example approve a draft, or a body `pending` shape shadow a plain top marker).

This file is the NORMATIVE home for everything that happens to an approved spec's `status:` marker and its `pending re-approval` entries after approval: edit waves and their answers, stacked and entangled reverts, the RESTORE rule, the hand-edit/git-log accounting test with its incremental anchor, double answers and arbitration, retitles and rename chains, lost referents. `SKILL.md` holds the happy path (interview → section-by-section approval → `status: approved` → commit) and points here.

**Outside a git repo** (crucible's sanctioned cwd fallback) the COMMIT-shaped tests of this file — the hand-edit/git-log test, paired-effects policing, the ancestry frontier, every one-commit rule — have no substrate: they are HONESTLY ABSENT, never vacuously passed and never a reason to distrust answers (a no-git answer append is consumed on its FILE shape; the defective-write policing is a git-repo protection and never re-fires asks outside git); the file-shape rules — markers, entries, appends, answers recorded at the moment — remain fully normative, and the absence of the git-backed protections is NAMED to the user once at the spec's approval.

**Trigger — read this file ENTIRELY, never a fragment:** any post-approval edit to an approved spec, any answer to a pending entry, and any marker or entry anomaly (an unanswered entry under a non-pending marker, a post-approval commit no record accounts for, conflicting answers, an entry naming a section the spec no longer contains). The shapes below are load-bearing; a partial read resolves anomalies wrongly. Cross-references to premortem's arms (ground-edit, stacked-revert, entangled, rejection — its `references/spec-gate.md`) and campaign's cells (arbitration asks, never-defer answers, recovery arms — its `references/arbitration.md`) are normative in THOSE skills — this file names where they plug in.

## The pending marker and edit waves

A third sanctioned shape, `status: approved (edits pending re-approval: <date>)`, marks post-approval edits awaiting the user — on a WAIVED-approval lineage the rewrite PRESERVES the waived token, `status: approved (waived: <date>; edits pending re-approval: <ISO date>)`: consumers key on the `edits pending re-approval` clause for pending-ness and on `waived:` for lineage — (premortem's spec-gate rule writes it; so does arcane's user-ordered amendment route): each edit wave adds an open/deferred entry of the EXACT shape `pending re-approval <ISO date>: <sections>` (a SECOND same-day wave appends the time to its header — entry identity keys on the dated header, and two byte-identical headers are indistinguishable to every consumer; the same arm covers an entangled append's `with <surviving entry's date>` cross-reference and a rename note's date-match to its minting entry: any same-day sibling forces the time, the family same-day arm) (multiple section names `;`-separated — titles legitimately contain commas, so comma is never the list delimiter; a `;`-list whose split admits MORE THAN ONE parse against the actual heading set (headings may themselves contain `;` — `A`, `B` and `A; B` all existing makes `A; B` two valid readings) → the user with every candidate parse shown, never a guess; a listed name matching no section heading is FIRST re-checked as a mis-split against the spec's actual headings before any lost-referent conclusion — the grammar for every section list in entries and resolution records), and the user's answer APPENDS to it — `— re-approved <ISO date>`, or `— rejected <ISO date>: reverted` (the edit reverts — on STACKED entries only its OWN edit: sections carrying other still-pending entries' edits keep them, each surviving edit restored from its own committed history, premortem's stacked-revert arm — a surviving edit that does not COMPOSE with the removal (it edited text the rejected entry introduced) is surfaced to the user, premortem's entangled arm, never an improvised merge — the given rejection still records AT THE MOMENT as `— rejected <ISO date>: revert entangled with <surviving entry's date>; disposition owed` (a durable pending shape: body unreverted, sections pending) until the disposition's closing append lands the actual revert; premortem's rejection arm recomputes the spec-gate yaml).

## Per-entry consumption and the RESTORE rule

Consumption is PER ENTRY: sections of an entry dated re-approved are approved ground immediately; only still-pending entries' sections stay barred; the marker that returns is the LINEAGE'S OWN shape — plain `status: approved`, or `status: approved (waived: <date>)` on a waived lineage: a restore never upgrades a waived approval to a plain one — in the commit that dates the LAST pending entry (the RESTORE rule — either answer counts, and an entangled disposition's closing `— reverted` append is the rejection's EXECUTION: dating the last pending entry, it restores the marker like any answer).

## Marker cross-check: the hand-edit test and double answers

Every consumer of a marker WITHOUT the pending clause — plain OR waived — cross-checks the entries: a `pending re-approval` entry with no answer appended under such a marker is the hand-edit/bad-merge shape — treat the spec as pending (run the owed ask), never as approved; a spec-body change committed AFTER approval that no entry ACCOUNTS FOR is the hand-edit shape too — entries account by SHAPE, answered or not: the sanctioned post-approval spec commits are an entry's edit wave, its rejection's revert, and an arbitrated restore (each landing WITH its marker/entry write in the same commit), the in-spec spec-gate VERDICT record (premortem's write — it lands in open/deferred, never in approved-ground sections), a RECOVERY repair recommitting the file (campaign's lost-referent arms) — exempt ONLY when its content byte-matches the file's LAST committed state ON THE REF BEING CONSUMED or on the recovery arm's NAMED source (campaign's arms name where they recover from — the phase branch, base history — and the recovery COMMIT MESSAGE records that named source durably (`recovered from <ref>`): later consumers on machines without the source branch read the message, and a recovery-shaped commit whose message names no source byte-matches only against the consumed ref's own committed states; following renames — `git log --follow`; recovery restores what was lost, never edits: content matching NO committed state, matching only an OLDER superseded state, or matching only an unrelated stale ref no arm names — a rollback that would silently revert later answered ground and delete its answer records — is a hand edit regardless of what its diff or markers look like), and a MERGE commit resolving spec-body conflicts (campaign's surfaced ordinary-git-conflict arm — which WRITES a dated resolution note into open/deferred naming the resolved sections; a merge is exempt only for hunks matching one of its parents, and an approved-ground hunk matching NEITHER parent is accounted for only by a resolution record NAMING that hunk's section — recordless, or covered only by a note naming other sections, it is the smuggle shape, treated as a hand edit) — so the test is the spec's `git log`, and accounting is PER CHANGE, not per commit — a write beside a change accounts only for what it names (an edit-wave entry for the sections it lists, a rename note ONLY for the heading rename it records AND only as its edit wave's companion write (the retitle rule below: a note is an ADDITION to its wave, never a substitute — a heading rename carrying a bare note with NO wave entry and no marker rewrite is itself unaccounted, the smuggled-retitle shape, and a note with no minting entry is a DEAD link in every chain); a body-text change riding beside a mere rename note is unaccounted, the hand-edit shape): a post-approval NON-MERGE commit touching APPROVED-GROUND section text with no marker/entry/verdict write ACCOUNTING FOR THAT CHANGE beside it, or a merge hunk matching neither parent with no resolution record naming its section — treat the changed sections as pending (the owed ask), never as approved ground — and the test is INCREMENTAL, with an ANCESTRY frontier, never a date frontier: a consumer that already classified this spec BLOB (same SHA) re-classifies nothing; otherwise it classifies every post-approval commit NOT an ancestor of the commit that landed the newest dated in-file record — a spec with NO post-approval records yet anchors at the APPROVAL itself (the commit that landed the `status:` marker: first consumption classifies the whole post-approval history from there), and a shallow clone whose truncation hides the anchor or any post-approval commit unshallows FIRST (an unenumerable history classifies nothing honestly) (the record's writer could only have audited what its commit descends from — commits merged in from another machine's history enter as non-ancestors and are classified on the first read after the merge, whatever their dates), never the whole post-approval history on every read; EVERY sanctioned record writer closes the gap the same way — before landing ANY dated record (an entry append, an edit-wave entry, a verdict record, a resolution note, the restore), it classifies the commits since the previous anchor (they are about to fall inside its new record's ancestry — incremental and usually empty, a no-op on clean histories), so a hand edit smuggled before ANY intermediate record is caught at that record, never laundered into an anchor's ancestry (the restore is the common case, not the definition — a duty riding only the restore would let a stacked entry or a verdict record swallow the window); the sanctioned classes anchor this because each lands a dated record in the file — EXCEPT the recovery class, which by construction byte-matches a prior state and adds nothing: recovery commits are re-classified by byte-match on each read (cheap, idempotent — the exemption is the classification); an entry carrying TWO CONFLICTING answers (`— re-approved` AND `— rejected` — a merge of two machines' answers) is the double-answer shape: treat as pending too — the re-fired ask is the arbitration question (campaign's cell: two recorded user decisions conflict → the user; the sections meanwhile read as the reverted approved text), never consume either side; and every answer append is policed by its PAIRED EFFECTS, the same commit-shape test the sanctioned classes define: a `— rejected <date>: reverted` append whose commit contains NO revert, a closing `— reverted <date>` append (the entangled disposition's answer, premortem's arm — its entire meaning IS the revert) whose commit contains NO revert, a `— re-approved`/arbitrated re-approval dating the LAST pending entry whose commit restores NO marker (or restores the wrong lineage's shape), or an answer append landing in a commit that also touches UNRELATED spec content — content outside the entry's own append, the answer's mandated revert/restore targets (including BOTH entries' reverts on a dual rejection, premortem's arm), the marker line, and any rename notes the same wave mandates — is the DEFECTIVE-WRITE shape — treat the entry as pending and surface it (the owed ask), never consume the append — with two scope carves: the policing covers only commits the FRONTIER classifies (an initial git-init import carrying pre-git history sits at or before the anchor — imported answers are never policed, the no-git degradation already covered them), and a linear-history interleave (two sessions, one checkout) is checked before declaring: an append whose mandated effects landed in the SAME or immediately adjacent commits of the interleave is a benign interleave — a commit-sweep artifact of one shared checkout, not a defective write; campaign's stale-write guard prevents the content loss, nothing needs re-unifying (appends are obeyed for their recorded decision only when their commit did what the decision requires — 'only the user's answers mint them' is checked by shape, not assumed); the arbitration answer APPENDS a third, dated record to the same entry — `— arbitrated <ISO date>: re-approved` or `— arbitrated <ISO date>: rejected (reverted)`, a same-day arbitration appending the time (the family's same-day arm) — recorded and committed at the moment given like every entry answer (premortem's one-commit-per-answer; under a campaign the never-defer answer class).

An entry whose arbitrated append is STRICTLY later than every other arbitrated append READS as that answer everywhere: an arbitrated `re-approved` also RESTORES the edited text to the spec body (the merge held the reverted text — recover the edit from its own committed history; the restore commit dates the answer), a rejected arbitration runs the revert arm, it counts for the RESTORE rule, and append-only merge unions preserve it so the conflict never re-fires; a strictly-later arbitrated append (only the user's answers mint them) supersedes an earlier one — records arbitrating records.

TWO conflicting arbitrated appends with NO strict order (equal dates — unless EVERY append carries a time and the times differ; a timed append never auto-beats an untimed same-date one) are themselves the double-answer shape: pending again, → the user, and the RESOLVING arbitration appends with the marker `(resolves both)` — `— arbitrated <ISO date> <time>: <answer> (resolves both)` — which supersedes ALL earlier arbitrated appends by its kind, not by order (an independent machine arbitration never carries the marker, so the auto-beat ban is untouched; two conflicting `(resolves both)` appends escalate the same way, recursively) — never auto-picked.

## Retitles, rename chains, lost referents

A section RETITLE is itself an approved-ground edit — the heading is text the user approved: it rides an edit wave (the marker rewrites to pending, an entry names the retitle, the user re-approves — premortem's ground-edit rule; a meaning-shifting rename never slips through as bookkeeping) — and ADDITIONALLY APPENDS a dated rename note to every OPEN entry naming it IN THE SAME commit — `— section renamed: <old> → <new>, <ISO date>` — append-only, so merge unions preserve it like any answer append and the entry's identity (its dated header) never mutates; consumers read an entry's sections THROUGH its rename chain — where a chain LINK is live only while its MINTING retitle entry (the wave entry naming that old → new, matched by the note's date) stands unanswered or re-approved: a link whose minting entry is dated REJECTED is a VOID link — the heading reverted with the rejection, the chain reads through it unchanged, and a void link never legitimizes a match to a later section that happens to carry the rejected new name (the lookalike ban holds); ANSWERED entries are historical records — left untouched, and old names inside them are never lost referents; an OPEN entry naming a section the spec no longer contains, with no rename note accounting for it, is a LOST REFERENT: treat the whole spec as pending until the user resolves that entry — never dropped silently, never auto-matched to a lookalike section; a chain link whose old/new split is ambiguous (titles may contain ` → ` or commas) re-checks candidates against the spec's actual headings at that commit — the heading list is the oracle, the `;` grammar's own re-check.
