# Spec-gate ground edits and their answers (normative home)

This file is the NORMATIVE home for premortem's post-approval machinery on a spec-gate lineage: the ground-edit rule (marker rewrite + entry in one commit), per-answer recording, the revert arms (stacked, entangled), the spec-gate yaml recompute, and the amendment-class rule for rejections landing mid-execution. Entry and marker SHAPES are normative in crucible's `references/marker-machine.md`; campaign's cells own the cross-machine transport and arbitration. `SKILL.md` holds the gate itself (method, verdict, BLOCK/WARN/PASS, the bridge) and points here.

**Trigger — read this file ENTIRELY, never a fragment:** any edit landing on an approved spec's ground after its approval, any user answer to a pending entry, any rejection's revert, and any rejection or recompute landing while the phase is already executing. A partial read loses recorded user decisions.

## The ground-edit rule and the answers

A spec edit landing AFTER the spec's approval — an edit to approved GROUND (sections, must_haves, decisions; the family's own bookkeeping writes — the in-spec verdict record, entry appends, a rejection's REVERT (restoring approved text is not a ground edit), and an arbitrated re-approval's RESTORE of the edited text (crucible's arbitration arm — the answer's own record, landing with its entry append) — never trigger this) — never hides under the standing `status: approved` marker: the SAME commit that lands the edits rewrites the marker to `status: approved (edits pending re-approval: <ISO date>)` (a waived-approval lineage keeps its token — crucible's lineage rule) and adds an open/deferred entry of crucible's exact shape (normative home: crucible `references/marker-machine.md`) — `pending re-approval <ISO date>: <sections>` (commit-first — nothing hides even if the session dies before the ask); the changed must_haves/decisions are then named to the user for arcane's fast re-approval arm (its input rule), and each answer is RECORDED at the moment it is given, ONE commit per answer (an `arbitrated` answer to a double-answered entry — crucible's shape — travels the same way):

approve → the entry appends `— re-approved <ISO date>` (its sections are approved ground immediately, per entry), and the commit dating the LAST pending entry also restores the lineage's own marker (crucible's RESTORE rule — plain, or waived);

reject → the same commit REVERTS the edit (the sections return to their approved text — on STACKED entries the revert removes only THIS entry's edit: sections carrying other still-pending entries' edits KEEP them, each surviving edit restored from its own committed history — the revert target is approved-plus-surviving-pending text, never a wholesale rollback to the pre-stack state, which would silently destroy the other entries' asked-about edits; a surviving edit that does not COMPOSE with the rejected one's removal — it edited text the rejected entry introduced, the diffs overlap — has no mechanical restore: surface it to the user — the rejection entangles the surviving entry (rework it against the approved text, or reject it too) — never an improvised merge of a recorded pending edit; and the given rejection is still RECORDED at the moment: the entry appends `— rejected <ISO date>: revert entangled with <surviving entry's date>; disposition owed` — a durable pending shape (the body keeps its current text, the sections stay pending, the marker keeps its pending clause; every machine reads it as pending, like a double answer) — and the user's disposition lands as the closing append, `— reverted <ISO date>` with the actual revert in that commit (or both entries' reverts when the user rejects both); an entangled append with no closing append is a standing ask, never terminal), the entry appends `— rejected <ISO date>: reverted` (terminal — the ask never re-fires), and the spec-gate yaml recomputes: a mitigation living in the reverted edit returns to `tigers:` and the BLOCK rule governs (mitigate differently, or the user accepts the risk explicitly) — and EITHER answer dating the last pending entry restores the lineage's own marker (crucible's restore rule — plain, or waived).

## Amendment-class: a rejection landing mid-execution

A rejection (or any recompute) landing while the phase is already EXECUTING is amendment-class: a BLOCK verdict bars ALL further dispatch until the verdict recomputes to PASS/WARN — the gate gates the PLAN, not a task subset (Task 1 is the common case, not the definition), tasks implementing the rejected ground re-open EXACTLY like an amendment's: the rejection's ANSWER COMMIT ITSELF carries the plan annotations — `[re-opened by rejection <date>]` on each invalidated task heading in the COMMITTED plan file, landing in the SAME commit as the entry append and the revert (the one-commit rule: the terminal `— rejected` append consumes the standing pending signal, so a crash window between the answer and the annotation would close owed rework with no machine-independent trace — binding them is what makes the record durable; the annotation is the same shape and transport as the amendment's, so a rebuilt or foreign ledger re-derives the owed re-opens from the plan; a rejection at a pure spec-gate stage with NO plan yet has nothing to annotate and nothing owed) and appends `Task N: re-opened (rejection <date>)` lines at the ledger's END (arcane's resume shape — the re-open line later in the ledger than the completion mark supersedes it; the resume sweep owes a ledger line for every annotated task missing one, arcane's per-task idempotent signal), and the bridge refreshes `<plan>.premortem.yaml` and the ledger verdict line from the recomputed spec yaml — the records execution actually consults must not stay stale-green; a second edit wave over a still-pending marker stacks its own entry and refreshes the marker's date — the ask covers every still-pending entry, not only the newest — and the verdict-record and plan-gate rules in `SKILL.md` still apply (the verdict is recorded in the spec's open/deferred section; the plan-gate run still happens after the plan is written).
