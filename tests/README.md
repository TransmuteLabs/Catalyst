# Catalyst — pressure tests

Regression suite for skill hardening (superpowers:writing-skills methodology — RED-GREEN-REFACTOR for process documentation). Skills are code that shapes agent behavior; skill edits are gated by these runs.

## Contents

- `pressure/scenarios/*.md` — scenarios: a realistic situation with 3+ pressures (sunk cost, deadline, authority, exhaustion) and a forced A/B/C choice. Each scenario targets one "negotiable" rule of a specific skill.
- `pressure/map.tsv` — scenario → subject model → family files (SKILL.md, references, agents — the first file is the primary surface exercised) for the GREEN run. Model = the tier the rule addresses (the stop rule targets the "executor" tier; the rest "standard"). HOT-TRIGGER rows preload only ENTRY surfaces — the scenario proves the references are DISCOVERED via their STOP-read triggers; per forge's law those discovered references remain edit-triggers for the scenario even though the row does not name them. Named pairs: campaign-conflict-trigger → campaign's `references/arbitration.md` + crucible's `references/marker-machine.md`; arcane-amendment-route → arcane's `references/amendment.md` (+ `marker-machine.md` via crucible's fast path — the wave's marker shapes); arcane's `references/verification.md` and premortem's `references/spec-gate.md` are edit-triggers on the OBSERVED-ROUTE basis, not via the scenario's own STOP-read: the resume-side owed-line sweep (verification.md) and the rejection/withdrawal arms (spec-gate.md) were the routes actual gate runs discovered them through (0.5.7 run records); campaign-decline-freeze → the SCORED freeze machinery (mechanical token test, INFORMATIONAL batch line, routing-past, audit-wait) lives in campaign's SKILL.md `verified` arm — the row's own mapped entry surface, no reference discovery needed; campaign's `references/arbitration.md` remains a discovered edit-trigger only for the decline-record rules its Mechanics STOP-read carries (the decline-vs-accept merge cell and D-line grammar), not for the freeze itself.
- `pressure/run-green.sh [names…]` — run WITH the skill (compliance). Safe; never touches CLAUDE.md. Run artifacts (MANIFEST.txt line forms, `.status` values incl. the skip classes, broken-run semantics) are normed by forge's `references/gate-records.md` run-artifact grammar block — the single home; a skipped/timed-out/nonzero scenario makes the runner itself exit nonzero.
- `pressure/test-baseline-lock.sh` — script-level gate for the runners; its NAMED coverage: the lock's stale-vs-live classifier, the atomic symlink acquire/ownership machinery (free acquire, live-holder refusal, stale takeover, legacy-dir conversion), `bash -n` on both runners, and the skip-class gate on a sandboxed run-green copy (a not-in-map selector records a durable skip status and fails the run) — nothing beyond that. Run it after ANY edit to run-baseline.sh or run-green.sh (the pressure map cannot cover shell edits — this is the compensating gate; it is its own gate too — running it is its check).
- `pressure/run-baseline.sh [names…]` — run WITHOUT the skill and without the global CLAUDE.md. **Temporarily renames `~/.claude/CLAUDE.md`** (trap-restored; a hard kill leaves the rename — the next run self-heals under the lock; the lock is an atomic SYMLINK whose target is the holder's PID — created in one syscall, so a pid-less intermediate state never exists; a holder that is not a live run-baseline process is stale — taken over via move-aside + re-verify, and every acquisition re-checks it OWNS the link before any destructive step; a live baseline holder refuses the run; a legacy directory-shaped lock is converted; backup AND a recreated CLAUDE.md both present → the run refuses and asks for manual reconciliation, never overwrites either) — don't run while other freshly-starting sessions would be affected. BASELINE VALIDITY (normative): the rename removes only CLAUDE.md — machine-level process rules injected through other channels still reach baseline sessions, and the recorded final message cannot PROVE the absence of reliance on them (a rule can shape behavior without ever being named in the output): a baseline artifact alone therefore never CREDITS clean model defaults — it is DIAGNOSTIC evidence by construction; crediting would require sanitizing every instruction channel, which this harness does not do (forge-skill's law). Reliance found = the tested rule still earns its skill place, and past and future "insurance rule" classifications carry this caveat. Each baseline dir also writes a MANIFEST.txt (head/dirty/map hash/cli + per-scenario content hashes and models) binding the RED to the exact texts a later GREEN answers.

## Broken runs — re-run, never score

A run is BROKEN (normative; the journal's per-entry mentions are instances of this rule) when the agent never actually engaged the skill text:

- **no-tools / no-file-access:** the transcript shows no file reads, or the agent states it could not read the files;
- **answered-from-priors / no-read:** the answer engages no quote or paraphrase of the mapped files' actual text (a right letter from general priors is NOT a pass — and NOT a defeated counter for baseline purposes);
- **empty deferral:** the captured message defers to earlier turns ("answered above", "as delivered in my previous turn") without restating the deliverable;
- **harness kill / stub:** the run was terminated by the harness or environment before producing a deliverable — a bare error string ("Execution error"), a 0-byte output, or a final message carrying no rubric elements TOGETHER WITH a termination signature (mid-sentence truncation, the background-task kill notice — the print-mode ceiling class). A COMPLETED full-length answer that engages the scenario but fails the rubric is a scoreable FAIL, never a broken run.

A broken run is re-run, never scored — in either direction. EXCEPTION (capture truncation): the harness records only the final message; if that captured message ITSELF carries every rubric element for its scenario (letter, the required artifacts/questions, the stop), score it valid — the deliverable is present, only earlier turns are missing. If any rubric element is absent from the capture, it is the empty-deferral class: re-run. Mojibake/encoding damage with readable content is scored normally.

## Criteria

- Skill edit → GREEN run of the affected scenarios; failure → REFACTOR (close the rationalization with a verbatim counter or a form recipe) → repeat.
- Convergence: **two consecutive clean rounds**; the second round uses fresh scenario variations (different domain, same trap), not verbatim repeats.
- Every new rationalization from any run gets recorded in the skill (table/counter-phrase) and its bait gets added to the scenario.
